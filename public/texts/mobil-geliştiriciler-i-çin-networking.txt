Uzak ˙Iki Ofisi Nasıl Bağlarız?
Uzak iki ofisin birbirine güvenli ve stabil biçimde bağlanması için genellikle iki
teknoloji kullanılır:
a) VPN (Virtual Private Network)
VPN, iki uzak lokasyon arasında şifreli bir tünel oluşturur.
• IPsec veya SSL tabanlı olabilir.
• Kurumsal ortamlarda site-to-site VPN kullanılır.
• ˙Internet hattı üzerinden kurulduğu için maliyet düşüktür.
b) Cloud Interconnect / Private Link
Bulut sağlayıcıların sunduğu özel hatlardır. Genellikle şu isimlerle bilinir:
• AWS Direct Connect
• Azure ExpressRoute
• Google Cloud Interconnect
Bu bağlantılar:
• ˙Internete çıkmadan, özel bir WAN hattı sağlar,
• Daha düşük latency sunar,
• SLA garantisi olduğu için kurumsal yapılarda tercih edilir.

Internetworking Nedir?
Internetworking, farklı ve bağımsız ağların (router veya gateway gibi ara ciha-
zlar kullanılarak) birbirine bağlanması ve tek bir büyük ağ gibi çalışmasıdır.
Bu kavramın temel noktaları şunlardır:
a) Tanım
Internetworking, “inter” (arasında) ve “networking” (ağ kurma) kelimelerinden
oluşur. Amaç, farklı teknolojilere, farklı IP aralıklarına veya farklı topolojilere
sahip ağları birleştirip iletişim kurabilir hale getirmektir.
b) Nasıl sağlanır?
Bu birleşme genellikle şu cihazlarla gerçekleştirilir:
• Router (IP seviyesinde yönlendirme yapar)
• Gateway (farklı protokol veya mimarileri bağlar)
• Layer-3 Switch (gelişmiş yönlendirme yapabilir)
c) Internetwork’ün Özelliği
Bağlanan ağlar şu türlerde olabilir:
• Özel kurum ağı (private network)
• Kamu ağı
• Endüstriyel ağ
• Ticari ağ
Tüm bu ağlar intermediate devices üzerinden bağlanınca, tek bir büyük
“internetwork” oluştururlar.
d) ˙Internet = En büyük internetwork
Bugünkü küresel internet; milyonlarca bağımsız LAN, WAN ve ISP ağının
router’lar üzerinden birbirine bağlanmasıyla oluşmuş dev bir internetwork yapısıdır.

OSI Modelindeki User Support (Software) Layers
Hangileridir?
OSI (Open Systems Interconnection) modelinde uygulamaya en yakın olan,
son kullanıcıya hizmet veren katmanlara User Support Layers veya Soft-
ware Layers denir. Bu katmanlar, uygulamaların ağ üzerinde veri alışverişi
yapmasını sağlar ve genellikle iş mantığı, veri formatlama ve oturum
yönetimi ile ilgilenir.
User Support Layers (Layer 5–7)
1. Application Layer (Katman 7)
2. Presentation Layer (Katman 6)
3. Session Layer (Katman 5)
a) Application Layer (Layer 7)
Son kullanıcıya en yakın katmandır. Uygulama protokollerinin çalıştığı kat-
mandır.
Görevleri:
• HTTP, HTTPS, FTP, SMTP, DNS gibi protokolleri çalıştırır
• Kullanıcı ile ağ arasındaki etkileşimi sağlar
Gerçek Hayat Örnekleri:
• Tarayıcıda bir web sitesini açmak (HTTPS)
• Gmail üzerinden e-posta göndermek (SMTP)
• Spotify API üzerinden şarkı listesi çekmek (REST/HTTPS)
• DNS sorgusu yapmak (DNS protokolü)
b) Presentation Layer (Layer 6)
Verinin formatlanması, sıkıştırılması ve şifrelenmesinden sorumludur.
Görevleri:
• Veri şifreleme (Encryption)
• Veri sıkıştırma (Compression)
• Veri format çevirimi (JSON, XML, JPEG, MP4 vb.)
Gerçek Hayat Örnekleri:

OSI Modelindeki Hardware Layers (Network Sup-
port Layers) Hangileridir?
OSI modelinin alt üç katmanı, ağın fiziksel olarak çalışmasını sağladıkları için
Hardware Layers veya Network Support Layers olarak adlandırılır. Bu
katmanlar, bit düzeyinden çerçevelere, yönlendirmeden fiziksel sinyale kadar
tüm düşük seviyeli ağ faaliyetlerini yönetir.
Network Support Layers (Layer 1–3)
1. Physical Layer (Katman 1)
2. Data Link Layer (Katman 2)
3. Network Layer (Katman 3)
1. Physical Layer (Layer 1)
Bu katman veriyi bit düzeyinde taşır ve fiziksel sinyal iletimini sağlar.
Görevleri:
• Voltaj seviyeleri, ışık sinyalleri, radyo sinyalleri
• Kablolama (Fiber, UTP), frekanslar, bant genişliği
• Bitlerin iletilmesi (0 ve 1)
Gerçek Hayat Örnekleri:
• Ethernet kabloları (Cat5e, Cat6)
• Fiber optik kablolar
• Wi-Fi radyo sinyalleri
• Donanım: Hub, Repeater
2. Layer 2 (Data Link Layer)
Layer 2, aynı yerel ağ (LAN) içindeki cihazlar arasındaki iletişimi sağlayan kat-
mandır. Bu katman, MAC adreslerini kullanarak çerçevelerin (frames) doğru
cihaza yönlendirilmesini sağlar.
Ana Özellikler:
• Kimlik: MAC adresi
• Veri birimi: Frame
• Temel cihaz: Switch

• Kapsam: Aynı ağ (LAN)
Gerçek Hayat Örnekleri:
• Bir switch’in MAC adres tablosu ile trafiği doğru porta iletmesi
• Bilgisayarın kendi ağı içindeki başka bir cihaza veri göndermesi
• Ethernet çerçevelerinin taşınması
Benzetme:
Aynı apartmandaki komşuya kapı numarasını kullanarak mektup

3. Layer 3 (Network Layer)
Layer 3, farklı ağlar (LAN → WAN → Internet) arasında paketlerin yönlendirilmesini
sağlar. Bu katman IP adresleri ile çalışır ve en kısa/uygun rotayı belirler.
Ana Özellikler:
• Kimlik: IP adresi
• Veri birimi: Packet
• Temel cihaz: Router
• Kapsam: Ağlar arası iletişim (internetworking)
Gerçek Hayat Örnekleri:
• Bir router’ın farklı ağlara paket yönlendirmesi
• Bir cihazın internete (örneğin 8.8.8.8’e) erişmesi
• OSPF veya BGP ile rota hesaplaması yapılması
Benzetme:
Başka bir şehre giden otobüsün rotasını seçmek.
Özet
Network Support (Hardware) katmanları:
• Physical Layer (1) → bitler, sinyal, kablo, radyo
• Data Link Layer (2) → çerçeve, MAC adresi, switch
• Network Layer (3) → IP adresi, router, yönlendirme
Bu üç katman, ağın fiziksel olarak çalışmasını ve paketlerin iki cihazdan tüm
internete kadar taşınmasını sağlar.

HTTPS Protokolü Nedir?
HTTPS (HyperText Transfer Protocol Secure), HTTP protokolünün
şifrelenmiş ve güvenli versiyonudur. ˙Istemci (client) ile sunucu (server) arasındaki
veri alışverişini, TLS/SSL kullanarak şifreler ve böylece üçüncü tarafların veriyi
okumasını veya değiştirmesini engeller.
Temel Özellikler
• Varsayılan port numarası: 443
• HTTP + TLS/SSL birleşiminden oluşur.
• Veri gizliliği, bütünlüğü ve kimlik doğrulaması sağlar.
HTTPS’in Sağladığı Üç Temel Güvenlik Özelliği
1. Gizlilik (Confidentiality) Tüm trafik TLS ile şifrelenir. Araya giren
biri (MITM) veriyi okuyamaz.
2. Bütünlük (Integrity) Veri gönderildiği gibi ulaşır. TLS, verinin değiştirilmesini
engeller.
3. Kimlik Doğrulama (Authentication) Sunucunun gerçekten iddia ettiği
sunucu olduğunu kanıtlar. ( Örnek: google.com gerçekten Google’ın ser-
tifikasına sahiptir.)
HTTPS Nasıl C alışır? ( Özet Akış)
1. Client, sunucuya bağlanır: https://example.com
2. Sunucu, TLS sertifikasını gönderir.
3. Client sertifikanın doğruluğunu CA üzerinden kontrol eder.
4. Client ve sunucu bir ortak oturum anahtarı üretir.
5. Sonraki tüm iletişim bu anahtar ile şifrelenir.
Gerçek Hayat Örnekleri
• Bir bankanın sitesine giriş yaparken (internet bankacılığı)
• E-ticaret ödemelerinde (kredi kartı işlemleri)
• WhatsApp Web gibi uygulamalarda web trafiğinin şifrelenmesi
• API isteklerinin güvenli taşınması (REST/GraphQL)

Header ve Trailer Hangi OSI Katmanlarında Ek-
lenir?
OSI modelinde veri üst katmanlardan alt katmanlara doğru ilerlerken, her kat-
man kendi kontrol bilgilerini verinin başına (header) veya sonuna (trailer)
ekler. Bu işleme encapsulation (kapsülleme) denir.
Header Ekleme
Aşağıdaki katmanlar, veriye sadece header ekler:
• Application Layer (7)
• Presentation Layer (6)
• Session Layer (5)
• Transport Layer (4)
(TCP/UDP header ekler)
• Network Layer (3)
(IP header ekler)
Her header katmana özgü kontrol bilgilerinden oluşur:
• TCP Port bilgileri (Katman 4)
• IP adresleri (Katman 3)
• Protokol tip bilgisi (Katman 3–4)
Trailer Ekleme
Sadece Data Link Layer (Katman 2) trailer ekler.
• Ethernet Frame Trailer (FCS/CRC)
• Hata tespit bilgisi içerir
Özet
• Header → Katman 7, 6, 5, 4, 3 tarafından eklenir.
• Trailer → Sadece Data Link Layer (Katman 2) ekler.

Veri Alt Katmanlardan Üst Katmanlara Doğru
˙Ilerlerken Ne Olur?
OSI modelinde bir veri paketi alt katmanlardan üst katmanlara doğru
çıktığında, her katman kendi eklediği kontrol bilgisini (başlık/header ve varsa
trailer) çıkartır. Bu sürece decapsulation (kapsülü açma) denir.
Temel Mantık
Alt katmanlardan gelen paket → her katmanda o katmanın header’ı
çıkarılır → üst katmana sadece ihtiyaç duyulan veri aktarılır.
Katman Bazında Decapsulation Süreci
1. Layer 1 (Physical): Bitleri alır ve Layer 2’ye iletir.
2. Layer 2 (Data Link): Ethernet Header + Trailer (FCS/CRC) çıkarılır.
3. Layer 3 (Network): IP Header çıkarılır.
4. Layer 4 (Transport): TCP/UDP Header çıkarılır (port bilgisi bu kat-
manda okunur).
5. Layer 5–7 (Session/Presentation/Application): Uygulama verisi
kademeli olarak soyutlanır.
Örnek Akış
Bir HTTP isteği alındığında şu işlem gerçekleşir:
[Ethernet Header] [IP Header] [TCP Header] [HTTP Data] [Ethernet Trailer]
↓ (L2 decapsulation)
[IP Header] [TCP Header] [HTTP Data]
↓ (L3 decapsulation)
[TCP Header] [HTTP Data]
↓ (L4 decapsulation)
[HTTP Data]
↓ (L5--L7)
Application Layer’a aktarılır
Özet
• Veri yukarı çıktıkça header’lar çıkarılır.
• Sadece uygulama seviyesindeki veri (payload) kalır.
• Bu sürece decapsulation denir.

Veri Üst Katmanlardan Alt Katmanlara Doğru
˙Ilerlerken Ne Olur?
OSI modelinde bir veri paketi üst katmanlardan alt katmanlara doğru il-
erlediğinde, her katman kendi kontrol bilgisini verinin başına ekler. Bu işleme
encapsulation (kapsülleme) denir.
Üst katmandaki uygulama verisi, alt katmanlara indikçe yeni header’lar ek-
lenir ve sonunda fiziksel sinyale dönüştürülüp iletilir.
Kapsülleme (Encapsulation) Adımları
1. Layer 7 (Application): Uygulama verisi üretilir (ör. HTTP Request).
2. Layer 6 (Presentation): Veri şifrelenir, sıkıştırılır veya encode edilir.
3. Layer 5 (Session): Oturum bilgileri hazırlanır.
4. Layer 4 (Transport): TCP/UDP header eklenir (port numaraları bu-
rada).
5. Layer 3 (Network): IP header eklenir (kaynak hedef IP).
6. Layer 2 (Data Link): Ethernet header + trailer eklenir (MAC adresleri,
FCS).
7. Layer 1 (Physical): Veri elektriksel/optik/radyo sinyallerine dönüştürülür.
Encapsulation Örneği
Bir HTTP isteği gönderilirken oluşan yapı şöyledir:
Application Data (HTTP)
↓ L6
[Presentation Header] HTTP Data
↓ L5
[Session Header] ...
↓ L4
[TCP Header] [HTTP Data]
↓ L3
[IP Header] [TCP Header] [HTTP Data]
↓ L2
[Ethernet Header] [IP Header] [TCP Header] [HTTP Data] [Ethernet Trailer]
↓ L1
Sinyal olarak iletilir

Mobil Uygulamalarda DNS (Domain Name Sys-
tem)
DNS, bir alan adını (api.example.com) ilgili IP adresine çözen sistemdir. Mobil
uygulamalarda DNS çözümlemeleri performans ve güvenilirlik açısından kritik
öneme sahiptir.
Neden Mobilde DNS Daha Kritik?
• Mobil ağlarda DNS cache süresi çok kısa olabilir.
• Ağ geçişlerinde (Wi-Fi ↔ LTE) DNS cache boşa düşer.
• Captive portallar (otel, havaalanı Wi-Fi) DNS’i redirect edebilir.
• ˙Ilk API çağrısının 300–800 ms gecikmesinin ana sebebi DNS’tir.
• DNS çözümlemesi başarısız olursa tüm uygulama offline görünür.
DNS Lookup Süreci
Bir mobil cihaz bir alan adına istek yaptığında sırasıyla şu adımlar gerçekleşir:
1. Cihaz DNS cache’ine bakar.
2. Yoksa işletim sistemi DNS resolver’a sorar.
3. Resolver, ISP veya işletim sistemi DNS sunucusuna istek gönderir.
4. IP adresi çözülür ve uygulamaya döndürülür.
Mobil Cihazlarda DNS Davranışları
Android:
• Uygulamaya özel DNS cache yoktur.
• OS genelde kısa süreli (1 dakika) DNS cache tutar.
iOS:
• DNS cache süresi daha agresif olarak temizlenir.
• Ağ değişimlerinde tüm cache invalidate olur.
DNS Kaynaklı Mobil Problemler
• Cold-start latency artar.
• Retrofit/Alamofire gibi HTTP client’lar ilk istekte timeout atabilir.
• Captive ağlarda DNS sizi login sayfasına yönlendirir.
• Yanlış DNS konfigürasyonunda API çöker gibi görünür.

Mobil DNS Performansı: Basit Açıklama
Bu bölümde DNS gecikmesi, pre-connect, hızlı DNS sağlayıcıları, TTL ve CDN
gibi kavramlar, mobil uygulama geliştirme açısından en basit haliyle açıklanmaktadır.
1. Pre-Connect Nedir?
Pre-connect, uygulama açılır açılmaz API sunucusuna erken bağlantı kurul-
ması anlamına gelir.
Uygulama açılır açılmaz, kullanıcı henüz hiçbir işlem yapmadan:
DNS çözülür, TCP bağlantısı kurulur ve TLS el sıkışması tamam-
lanır.
Böylece ilk API isteği yapıldığında:
• DNS beklemez
• TCP bağlantısı kurulmasını beklemez
• TLS işlemlerini beklemez
Sonuç: ˙Ilk API isteği anında gider.
2. Hızlı DNS (Cloudflare / Google DNS) Neden Önemlidir?
Bazı mobil operatörlerin DNS sunucuları çok yavaştır. Örneğin:
• Yavaş bir DNS: 400–600 ms
• Hızlı DNS (Cloudflare/Google): 10–30 ms
Bu fark, uygulamanın açılış hızını doğrudan etkiler.
Hızlı DNS kullanmak = Uygulamanın daha hızlı açılması.
3. TTL Nedir ve Neden Stabil IP Kullanmak Önemlidir?
TTL, “Bu DNS bilgisi ne kadar süre saklansın?” sorusunun cevabıdır.
• TTL kısa olursa: DNS çok sık yeniden çözülür.
• TTL uzun olursa: DNS daha az çözülür, performans artar.
Mobil cihaz Wi-Fi ile LTE arasında geçiş yaptığında DNS cache silinebilir.
Bu durumda yeniden DNS çözümü gerekir ve gecikme oluşur.
API sunucuları stabil IP kullanırsa:
• TTL daha uzun tutulabilir
• DNS daha seyrek çözülür
• Daha az gecikme ve hata oluşur

4. CDN DNS’i Nasıl Hızlandırır?
CDN (Content Delivery Network), DNS çözümünü kullanıcıya en yakın nok-
tadan yapar.
• Normal DNS: Avrupa’daki sunucudan cevap (80–120 ms)
• CDN DNS: ˙Istanbul’daki edge noktadan cevap (5–10 ms)
CDN = DNS cevabını sana en yakın yerden veren sistem.
Bu nedenle CDN kullanımı DNS gecikmesini önemli ölçüde azaltır.
Özet
• Pre-connect: API’a giden yolu önceden hazırlamak.
• Hızlı DNS: Uygulamanın daha hızlı açılmasını sağlar.
• TTL: DNS bilgisinin ne kadar saklanacağını belirler.
• CDN: DNS sorgularını en yakın noktadan cevaplayarak hız kazandırır.
Mobil uygulamalardaki ilk yavaşlık çoğu zaman DNS çözümleme
süresinden kaynaklanır. Bu teknikler, açılış süresini belirgin şekilde
hızlandırır.

Mobil Uygulamalarda HTTPS, TLS ve Certifi-
cate Pinning
Mobil uygulamalar ile sunucu arasındaki tüm iletişim, güvenlik nedeniyle HTTPS
üzerinden yapılır. HTTPS, HTTP protokolünün TLS (Transport Layer Secu-
rity) ile şifrelenmiş halidir.
1. HTTPS Nedir?
• Varsayılan portu: 443
• HTTP + TLS birleşiminden oluşur.
• Amaç: veriyi şifrelemek ve güvenli iletişim sağlamak.
HTTPS = HTTP’nin şifrelenmiş hali.
2. TLS (Transport Layer Security) Nedir?
TLS, istemci ile sunucu arasındaki iletişimi şifreleyen protokoldür. Mobil uygu-
lamalarda TLS üç önemli güvenlik özelliği sağlar:
• Gizlilik (veri şifrelenir)
• Bütünlük (veri değiştirilmez)
• Kimlik doğrulama (sunucu gerçekten o sunucu mu?)
3. TLS Neden Mobilde Yavaşlık Yaratabilir?
Bir HTTPS isteği yapılmadan önce şu üç adım gerçekleşir:
1. DNS çözümü
2. TCP bağlantısı kurulması
3. TLS handshake (sertifika doğrulama)
Bu işlem özellikle mobilde:
• düşük sinyal gücünde,
• baz istasyonu değişimlerinde (handover),
• Wi-Fi ↔ LTE geçişlerinde
100–500 ms ek gecikme yaratabilir.

4. Certificate Pinning Nedir?
Certificate Pinning, istemcinin sadece belirli bir sertifikaya güvenmesini sağlayan
bir güvenlik tekniğidir.
Pinning: “Bu sunucu sadece şu sertifikaya sahip olursa bağlan.” de-
mektir.
Amaç: Man-in-the-Middle (MITM) saldırılarını önlemek.
Pinning Nerelerde Kullanılır?
• Bankacılık uygulamaları
• Ödeme sistemleri
• Hassas veriye sahip sağlık uygulamaları

Latency, RTT, Packet Loss ve Jitter
Mobil cihazların ağ performansı; baz istasyonu geçişleri, zayıf sinyal, Wi-Fi
ve LTE arasındaki geçişler gibi sebeplerle sürekli değişir. Bu bölümde mobil
uygulamaların performansını doğrudan etkileyen dört kritik kavram sade şekilde
açıklanmıştır.
1. Latency (Gecikme)
Latency, bir isteğin sunucuya gidip cevap almasının ne kadar sürdüğünü gösterir.
Mobil ağlarda latency genellikle 30 ms – 300 ms arasıdır.
• Wi-Fi: 20–50 ms
• LTE/4G: 50–150 ms
• 3G: 150–300 ms
• Zayıf sinyal: 300+ ms
Latency ne kadar yüksekse API çağrısı o kadar yavaş hissedilir.
2. RTT (Round-Trip Time)
RTT, bir paketin sunucuya gidip geri dönme süresidir. Latency ile karıştırılmaması
gerekir:
Latency = tek yön gecikme
RTT = gidiş + dönüş gecikmesi
TLS handshake gibi işlemler multiple RTT gerektirdiği için mobilde açılış
süreleri uzayabilir.
3. Packet Loss (Paket Kaybı)
Packet loss, gönderilen veri paketlerinin bir kısmının hedefe ulaşmamasıdır.
Mobil ağlarda yaygın sebepler:
• Zayıf LTE sinyali
• Baz istasyonu değişimi (cell handover)
• Wi-Fi parazitleri
• Aşırı yüklenmiş ağ
Etkisi:
• TCP bağlantılarında yeniden gönderim (retransmission) olur.
• Bu durum API çağrılarını birkaç saniye geciktirebilir.
• Video call’da kalite düşer, ses kesilir.
20

-- 20 of 58 --

4. Jitter (Gecikme Oynaklığı)
Jitter, paketlerin farklı zamanlarda varmasıdır. Yani gecikme sabit değildir;
dalgalanır.
Bir paket 40 ms, diğeri 120 ms, sonra 80 ms gelir → işte bu jitter’dır.
Jitter özellikle şuralarda kritiktir:
• WebRTC voice/video
• Online gaming
• Live streaming
Etkileri:
• Sesin bozulması
• Görüntünün donması
• Video/ses senkron kaybı
Özet
• Latency: Tek yön gecikme.
• RTT: Gidiş-dönüş süresi.
• Packet loss: Paketlerin kaybolması.
• Jitter: Paketlerin gecikmesinin dalgalanması.
Mobil ağlar sabit değildir; kullanıcı hareket ettikçe sinyal güçleri,
baz istasyonları ve ağ kalitesi değişir. Bu yüzden API performansı
ve gerçek zamanlı iletişim bu dört kavramdan güçlü şekilde etkilenir.

Mobil Ağlarda Timeout, Retry ve Backoff Strate-
jileri
Mobil ağlar sabit değildir; kullanıcı hareket ederken sinyal gücü, baz istasyonu,
Wi-Fi erişim noktası veya IP adresi değişebilir. Bu nedenle API isteklerinde za-
man zaman gecikme, kopma veya geçici hatalar oluşur. Bu durumları yönetmek
için timeout, retry ve backoff stratejileri zorunludur.
1. Timeout Nedir?
Timeout, bir isteğin belirlenen süre içinde yanıt alamazsa hata ile sonlandırılmasıdır.
Mobilde timeout’lar genellikle şu durumlarda tetiklenir:
• Zayıf LTE/Wi-Fi sinyali
• Ağ geçişi (Wi-Fi ↔ LTE)
• Paket kaybı (packet loss)
• DNS çözümlemesinin uzun sürmesi
Uygulama, sonsuza kadar beklememek için kısa ve makul timeout değerlerine
sahip olmalıdır.
2. Retry (Tekrar Deneme)
Retry, başarısız olan isteğin belirli durumlarda yeniden gönderilmesidir. An-
cak retry yalnızca şu tür hatalar için yapılmalıdır:
• Geçici ağ hataları (timeout, connection lost)
• DNS geçici hataları
• 502 / 503 / 504 sunucu hataları
Aşağıdaki durumlarda retry yapılmamalıdır:
• 400 (bad request)
• 401 (unauthorized)
• 403 (forbidden)
• ˙Işlem idempotent değilse (örneğin bir ödeme yapmak)

3. Backoff Stratejisi
Retry işlemleri arasında gecikme eklenmesine backoff denir. Bu, sunucuyu ve
ağı aşırı yüklemeden korur.
En sık kullanılan yöntem:
Exponential Backoff : Her retry denemesinde bekleme süresi kat-
lanarak artar.
Örnek:
1. deneme → hemen
2. deneme → 1 saniye sonra
3. deneme → 2 saniye sonra
4. deneme → 4 saniye sonra
Bu strateji özellikle mobilde önemlidir çünkü:
• Kullanıcı tünelden geçiyor olabilir (sinyal düşer)
• IP değişmiş olabilir
• Baz istasyonu değişmiş olabilir
• Wi-Fi internet bağlantısı kesilmiş olabilir
4. Idempotent ˙Isteklerde Retry
Idempotent istekler tekrarlandığında sonuç değişmez.
Örneğin:
• GET /profile
• PUT /user/settings
Bu isteklerde retry güvenlidir.
Ancak idempotent olmayan isteklerde retry sakıncalıdır:
• POST /payment
• POST /order
Bu tür istekler aynı işlemi iki kez gerçekleştirebilir.

Mobil Ağlarda TCP Flow Control, Congestion
Control ve Slow Start
Mobil uygulamalarda API çağrılarının bazen yavaşlamasının temel sebebi yalnızca
kötü sinyal veya kötü cihaz performansı değildir. Arka planda TCP’nin kendi
güvenlik ve denge mekanizmaları çalışır.
Bu bölümde mobil geliştiricilerin bilmesi gereken üç önemli TCP mekaniz-
ması sade şekilde açıklanmaktadır:
1. Flow Control (alıcıyı korur)
2. Congestion Control (ağı korur)
3. Slow Start (bağlantıyı yavaşça hızlandırır)
1. Flow Control (Akış Kontrolü)
Flow control, gönderici ile alıcı arasındaki hız farkını dengeleyen mekanizmadır.
Amaç: Gönderici çok hızlı gönderip alıcıyı boğmasın.
Alıcı taraf (örneğin mobil cihaz) kendi buffer kapasitesine göre şu işareti
gönderir:
“Benim şu kadar boş yerim var, şimdilik en fazla bu kadar veri
gönder.”
Mobilde neden önemli?
• Zayıf sinyalde cihaz paketleri gecikmeli işler.
• Cihaz arka planda yoğun işlem yaparken (kamera, video, oyun) buffer
küçülür.
• Flow control devreye girer → veri akışı yavaşlar.
2. Congestion Control (Ağ Tıkanıklığı Kontrolü)
Bu mekanizma ağ tıkanıklığını önlemek için kullanılır.
Amaç: Aşırı veri gönderildiğinde ağ çökmemesi için gönderme hızını
otomatik azaltmak.
Mobilde bu neden olur?
• Aynı baz istasyonuna çok kullanıcı bağlandığında
• Paket kaybı olduğunda
• Baz istasyonu değişiminde (handover)
Paket kaybı tespit edilirse TCP şöyle der:
“Ağ tıkalı olabilir. Hızı azaltıyorum.”
Bu, mobilde hissedilen “ani yavaşlamaların” ana sebebidir.
26

-- 26 of 58 --

3. Slow Start (Yavaş Başlangıç)
Slow Start, bir TCP bağlantısının başlangıçta küçük bir hızla başlaması ve ağ
ortamını test ederek yavaş yavaş hızlanmasıdır.
Amaç: Yeni bağlantının ağı tıkamadan kontrollü şekilde hızlanması.
Nasıl çalışır?
1. Başta çok az veri gönderilir.
2. Sunucudan “ACK” geldikçe gönderim miktarı iki katına çıkar.
3. Paket kaybı olursa hız azaltılır.
Mobil etkisi:
• ˙Ilk API çağrısı her zaman daha yavaştır.
• Büyük dosya indirmeleri ilk birkaç saniyede yavaş başlar.
• Zayıf sinyalde Slow Start sürekli tetiklenebilir.
Gerçek Örnek: Neden API Arada Bir 2 Saniye Gecikiyor?
Bir kullanıcı metroya biniyor:
• Sinyal zayıflıyor → paket kayıpları başlıyor
• TCP bunu tespit ediyor → hızı düşürüyor (Congestion Control)
• Birkaç paket gidip gelince tekrar hızlanıyor (Slow Start)
Kullanıcı şöyle hisseder:
“Bazen API yavaşlıyor, sonra düzeliyor.”
Bu tamamen TCP’nin doğal davranışıdır.
Özet
• Flow Control: Alıcı çok doluysa hız düşer.
• Congestion Control: Ağ tıkalı görünüyorsa hız düşer.
• Slow Start: Bağlantı yavaş başlar, sonra hızlanır.
Mobil ağ kararsız olduğu için TCP mekanizmaları sık sık devreye
girer. Bu, mobil API çağrılarının zaman zaman dalgalı hissettirmesinin
temel sebebidir.

Mobil Uygulamalarda Cache: Memory Cache, Disk

Mobil uygulamalarda ağ gecikmesi, paket kaybı ve enerji tüketimi yüksek olduğundan,
doğru cache stratejileri hem performansı hem de kullanıcı deneyimini önemli
ölçüde iyileştirir. Bu bölümde mobil geliştiricilerin bilmesi gereken üç temel
cache yaklaşımı açıklanır:
1. Memory Cache
2. Disk Cache
3. Stale-While-Revalidate (SWR)
1. Memory Cache (Bellek Üstü Cache)
Memory cache, verilerin doğrudan RAM üzerinde tutulmasıdır.
• En hızlı cache türüdür.
• Uygulama kapatılınca kaybolur.
• Tekrar eden ekran geçişlerinde veriyi anında gösterir.
Mobilde kullanım örnekleri:
• Profil bilgisi
• Ayarlar
• UI listeleri (kısa ömürlü veriler)
• Görsel placeholder cache
Memory cache = “Bu veriyi çok hızlı lazım olacak, RAM’de tu-
tayım.”
2. Disk Cache (Kalıcılık ˙Için Cache)
Disk cache, verilerin cihazın depolama alanına kaydedilmesidir.
• Memory cache’ten yavaştır.
• Uygulama kapansa da veri kalır.
• Offline-first uygulamalar için zorunludur.
Mobilde kullanım örnekleri:
• Haber listeleri

-- 34 of 58 --

• E-ticaret ürün listesi
• Kullanıcının geçmiş mesajları
• Büyük JSON yanıtları
Disk cache genellikle şu stratejiyle çalışır:
• Eğer network varsa: veriyi yenile
• Network yoksa: disk cache’ten oku
3. Stale-While-Revalidate (SWR) Stratejisi
SWR, modern mobil uygulamalarda en çok kullanılan cache modelidir.
“ Önce eski veriyi hemen göster, sonra arka planda yenisini al.”
Adımlar:
1. Disk veya memory cache’te veri varsa hemen UI’a verilir.
2. Arka planda sunucuya yeni istek gönderilir.
3. Yeni veri gelince UI güncellenir ve disk/memory cache yenilenir.

• Latency yüksek olsa bile UI anında açılır.
• Network kötü ise bile uygulama offline gibi çalışır.
• Battery cost düşer (daha az gereksiz API çağrısı).
• Kullanıcı anlık veri görür; “loading” ekranları azalır.
SWR = “Hızlı görünen ama her zaman güncel kalan UI.”

• Kritik veriler → 30 saniye

• Kullanıcının coğrafi konumu (IP bazlı)

HTTP Keep-Alive ve Short-Lived Connections:
Mobilde Hangisi Daha Verimli?
Mobil cihazların ağ bağlantısı sabit olmadığı için HTTP isteklerinin bağlantı
yönetimi performansı doğrudan etkiler. Bu bölümde iki temel bağlantı stratejisi
açıklanmaktadır:
1. Keep-Alive (Persistent Connection)
2. Short-Lived (Her istekte yeni bağlantı)
1. Short-Lived Connection Nedir?
Short-lived bağlantı, her API isteği için:
1. DNS çözümü
2. TCP bağlantısı
3. TLS handshake
işlemlerinin tekrar yapılmasıdır.
Dezavantajları:
• Her istek 200–800 ms ekstra gecikme alır.
• Battery cost çok yükselir.
• Mobil ağ değişimlerinde daha fazla hata oluşur.
Short-lived = “Her istekte sıfırdan telefon edip konuşmaya başlamak.”
2. Keep-Alive (Persistent Connection) Nedir?
Keep-alive, bir TCP + TLS bağlantısını bir kez açıp birden fazla istekte
tekrar kullanmak anlamına gelir.
Bu sayede:
• DNS çözümü yalnızca ilk istekte yapılır.
• TCP bağlantısı korunur.
• TLS handshake tekrar yapılmaz.
Avantajları:
• ˙Ikinci ve sonraki istekler 5–20 ms gibi çok hızlıdır.
• Battery tüketimi belirgin şekilde düşer.
• Sunucuya daha az yük gider.
Keep-alive = “Bağlantıyı açık tut, tekrar tekrar kullan.”
40

-- 40 of 58 --

3. Mobil Cihazlarda Keep-Alive Neden Daha Önemlidir?
Mobil ağlar sabittir:
• LTE sinyali sürekli değişir
• Baz istasyonları arasına geçilir
• Wi-Fi ↔ LTE geçişleri sık yaşanır
Bu nedenle short-lived bağlantılar:
• sık sık zaman aşımına uğrar
• TCP yeniden kurulur
• uygulama donuyor gibi hissedilir
Keep-alive sayesinde:
• sabit bir bağlantı sürekli yeniden kullanılabilir
• mobil ağ dalgalanmaları daha az hissedilir
• API performansı çok daha stabil olur
4. Zayıf Sinyalde Neden Short-Lived C ok Kötüdür?
Her yeni bağlantı açmak:
• 3 kez paket alışverişi (TCP handshake)
• 2 kez paket alışverişi (TLS handshake)
gerektirir.
Zayıf LTE sinyalinde bu işlemler saniyelere yayılabilir.
Bu durumda keep-alive her zaman daha hızlıdır.
5. Modern Mobil Kütüphaneler Keep-Alive Nasıl Uygular?
React Native:
• Fetch API → otomatik keep-alive
• Axios → keep-alive adapter ile gelişmiş kullanım
• React Query → bağlantı reuse edilir
Android (OkHttp):
• Keep-alive varsayılan olarak etkindir.
• Connection pool ile birden çok bağlantı reuse edilir.
iOS (URLSession):
• HTTP/2 keep-alive otomatik çalışır.
• Aynı host için bağlantı paylaşılır.

6. Keep-Alive’ın Riskleri
• Bağlantı uzun süre açık kalınca “stale” hale gelebilir.
• Bazı load balancing mimarileri kısa timeout kullanır.
• Ağ değişimi olursa bağlantı kopar.
Bu nedenle:
• Heartbeat (ping) önerilir.
• Bağlantı koparsa otomatik retry yapılmalıdır.
Özet
• Short-Lived: Her istekte bağlantı sıfırdan kurulur → yavaş + pahalı.
• Keep-Alive: Bağlantı tekrar kullanılır → hızlı + enerji tasarruflu.
Mobil uygulamalarda performans ve enerji verimliliği için Keep-
Alive neredeyse her zaman en doğru bağlantı yönetimi stratejisidir.

Mobil Ağlarda TCP ve UDP: Hangisi Ne Zaman
Kullanılır?
Mobil cihazlarda ağ koşulları sürekli değiştiği için (sinyal, hız, paket kaybı),
doğru protokol seçimi uygulamanın kararlılığı ve performansı açısından çok
önemlidir.
Bu bölümde TCP ve UDP’nin mobil ortamdaki davranışları ve hangi du-
rumda hangisinin tercih edilmesi gerektiği sade şekilde açıklanır.
1. TCP Nedir?
TCP, güvenilir ve sıralı veri iletimi sağlar.
Temel özellikleri:
• Paketler eksiksiz ulaşır.
• Paketler sırayla gelir.
• Kaybolan paketler yeniden gönderilir (retransmission).
• Bağlantı kurulması gerekir (handshake).
Bu nedenle:
TCP = “Doğruluk önemliyse, hız ikinci planda kalabilir.”
Mobilde kullanım alanları:
• API çağrıları
• HTTPS bağlantıları
• Dosya indirme / yükleme
• Kullanıcı verisi (mesaj, profil)
2. UDP Nedir?
UDP, hızlı fakat güvenilir olmayan bir protokoldür.
Özellikleri:
• Bağlantı kurulmaz (connectionless).
• Kaybolan paket tekrar gönderilmez.
• Paket sırası garanti edilmez.
Bu nedenle:
UDP = “Hız çok önemliyse, bazen paket kaybına tolerans vardır.”
43

-- 43 of 58 --

Mobilde kullanım alanları:
• WebRTC toplantıları
• Canlı video yayını
• Sesli görüşme
• Online oyunlar
• Real-time çizim / konum senkronizasyonu
3. Mobilde TCP’nin Dezavantajları
Mobil ağlar kararsızdır ve TCP şu durumlarda ciddi yavaşlamalar yaratır:
• Paket kaybı olunca hız düşer (congestion control).
• Handover (baz istasyonu geçişi) olunca bağlantı kopar.
• LTE ↔ Wi-Fi değişiminde bağlantı tamamen resetlenir.
Bu nedenle:
TCP, sık sinyal değişen mobil ortamlarda “dalgalı” performans gösterir.
4. Mobilde UDP’nin Avantajları
UDP paket kaybı olduğunda durmaz, hızını düşürmez. Bu nedenle gerçek za-
manlı uygulamalarda idealdir.
Özellikle WebRTC şu nedenle UDP kullanır:
• Ses/video için “biraz paket kaybı” kaliteyi bozmaz.
• Ama gecikme artarsa konuşma tamamen bozulur.
Gerçek zamanlı iletişimde gecikme, doğruluktan daha kritiktir.
5. WebRTC Neden UDP Kullanır?
WebRTC, mobilde kararsız ağları tolere edebilmek için UDP üzerine kendi hata
düzeltme mekanizmalarını ekler:
• Jitter buffer
• FEC (Forward Error Correction)
• Packet prioritization
Böylece:
• Gecikme çok düşüktür
• Paket kaybı ses/video kalitesini tamamen bozmaz

6. Hangi Durumda TCP, Hangi Durumda UDP?
TCP kullan:
• Doğruluk önemliyse
• Kaybolan paket kabul edilemezse
• Veri mutlaka eksiksiz gelmeli ise
• API çağrıları, dosya transferleri, ödeme işlemleri
UDP kullan:
• Hız kritikse
• Birkaç paketin kaybolması sorun değilse
• Ses/video/chat real-time akıyorsa
• Oyun ve canlı yayın gibi durumlarda
Özet
• TCP: Güvenilir ama yavaş — API ve kritik veri için idealdir.
• UDP: C ok hızlı ama güvenilir değil — WebRTC ve real-time iletişim için
idealdir.
Mobil ağda en önemli karar: “Doğruluk mı önemli, hız mı?” Cevap
hıssa UDP, doğruluksa TCP seçilir.

Authorization: JWT, OAuth2 ve Token Rotation
(Mobil Özet)
Mobil uygulamalarda oturum yönetimi genellikle iki ana kavram üzerine kuru-
lur: access token ve refresh token. Bu yapı, güvenliği artırmak ve mobil
cihazlardaki ağ değişikliklerine rağmen oturumu korumak için kullanılır.
1. Access Token
• Kısa ömürlüdür (5–15 dakika).
• API isteklerinde Authorization: Bearer <token> olarak gönderilir.
• C alınırsa uzun süre kullanılamaz (çünkü süresi kısadır).
Access token = “Hızlı tükenen giriş bileti.”

• C ok daha uzun ömürlüdür (günler/haftalar).
• Sunucuya gönderildiğinde yeni bir access token üretir.
• Güvenlik nedeniyle genellikle sadece login sunucusuna gönderilir.
Refresh token = “Yeni erişim bileti üretme hakkı.”
3. Token Rotation (Yenileme Döngüsü)
Token rotation, her refresh işlemi sonrasında:
1. yeni bir access token,
2. ve yeni bir refresh token
üretme prensibidir.
Amaç:
• çalınmış refresh token’ların kullanılmasını zorlaştırmak,
• oturumu sürekli güncel ve güvenli tutmak.
4. Mobilde Neden Gereklidir?
• Mobil cihazlar sık sık IP değiştirir.
• Ağ kopmaları sonucu access token süreleri dolabilir.
• Kullanıcıları sürekli yeniden login yaptırmamak gerekir.

3. Yeni token’larla API isteği tekrar edilir.

API Gateway ve Rate Limiting (Mobilde Etkisi)
API Gateway, mobil uygulamanın tüm API çağrılarının geçtiği merkezi kontrol
katmanıdır. Mobilde yaşanan birçok gizli performans probleminin arkasında
gateway politikaları bulunur.
1. API Gateway Nedir?
• Tüm API trafiğinin giriş kapısıdır.
• Auth, rate limit, routing ve güvenlik kontrolleri burada yapılır.
• Mobil cihaz backend’e doğrudan konuşmaz; gateway’e konuşur.
Gateway = “Backend dünyasının gümrük kapısı.”
2. Mobilde Neden Kritik?
Mobil cihazlarda bağlantı sık sık kopup bağlandığı için:
• aynı istek birkaç kez tekrar edilebilir,
• hızlı ardışık isteklerde rate limit’e takılabilir,
• zayıf ağda threshold’lar hızlıca dolabilir.
Bu yüzden gateway politikaları mobil uygulamada beklenmedik hatalara yol
açabilir.
3. Rate Limiting Türleri
• IP-based: Mobilde problemli, çünkü mobil operatör NAT kullanır.
• User-based: Kimlik doğrulaması olan uygulamalar için en yaygını.
• Endpoint-based: Belirli API çağrılarının saniye başına limiti.
Mobil kullanıcılar aynı baz istasyonu IP’sini paylaştıkları için IP-
based rate limit en sık hata yaratan tiptir.
4. Mobilde Rate Limit Nasıl Hissedilir?
• C ok hızlı scroll yapılınca API’nın “bir anda durması”
• Infinite list’in aniden kesilmesi
• “Unexpected 429 too many requests” hatası
• Birkaç refresh sonrası API’nın cevap vermemesi
Kısacası: “Uygulama bozuk gibi davranır.”

5. Gateway’in Mobilde Yaptığı Diğer ˙Işler
• Token doğrulama (401)
• CORS kontrolü
• Request body size limit
• Timeout ve retry politikaları
• Routing (canary release, blue-green)
Bu politikaların herhangi biri mobilde yavaşlık veya hata yaratabilir.

Download / Upload Optimizasyonu: Chunking,

Mobil ağlar stabil olmadığı için büyük dosya işlemleri sık sık yarıda kesilir. Bu

ked) aktarım kullanır.
1. Chunked Upload (Parçalı Yükleme)
• Büyük dosya, küçük parçalara (ör. 1 MB) bölünür.
• Her parça ayrı bir API isteği ile gönderilir.
• Bağlantı koparsa sadece eksik parça tekrar yüklenir.
Avantajları:
• Zayıf ağda yükleme daha stabil olur.

Yükleme kesildiğinde kaldığı yerden devam edebilmek için:
1. Sunucu hangi chunk’ların alındığını tutar.
2. Mobil cihaz sadece eksik parçaları tekrar gönderir.
Resume = “Baştan değil, kaldığın yerden devam et.”
3. Range Requests (Bölmeli ˙Indirme)
Büyük dosyalar için indirme şu şekilde yapılır:
• ˙Istemci Range: bytes=0-999999 gibi bir aralık ister.
• Sunucu sadece istenen bölümü döner.
• ˙Indirme yarıda kesilirse yeni range ile devam edilir.
Bu mekanizma özellikle video indirmede kullanılır.

Reachability, cihazın gerçek ağ durumunu izleyen mekanizmadır:
• Online → API istekleri aktif
• Offline → tüm istekler kuyruklanır veya önbellekten okunur
• Değişim → retry tetiklenir
React Native:
• @react-native-community/netinfo
iOS:
• NWPathMonitor
Android:
• ConnectivityManager
3. Offline-First ˙Için Temel Teknikler
• Local cache: MMKV, SQLite, WatermelonDB, Room
• Write queue: istekler sıraya alınır, ağ gelince gönderilir
• Mutation deduplication: aynı veri birden fazla kez gönderilmez
• Conflict resolution: hem lokal hem sunucu değişmişse birleştirme yapılır

4. Reachability’nin Mobil ˙Için Zor Yanları
• LTE sinyali varken internet olmayabilir (captive network)
• Wi-Fi aktif olsa da router interneti olmayabilir
• Sinyal anlık dalgalanabilir (flaky network)
Bu yüzden:
• API başarısız olunca hemen offline moduna geçilmez
• birkaç başarısız istekten sonra offline kabul edilir
5. En ˙Iyi Retry Stratejisi
• Exponential backoff (1s → 2s → 4s → 8s)
• Jitter (rastgele küçük gecikme ekleme)
• Sabit tekrar sayısı (ör. max 5 retry)
Amaç:
• Gateway’i boğmamak
• Battery drain yaratmamak
• Ağı gereksiz tüketmemek
Özet
• Mobil uygulama her zaman offline çalışabilmelidir.
• Network reachability → ağ dalgalanmalarını doğru algılar.
• Retry mekanizması akıllı olmalı (backoff + jitter).
• Veri önce lokal depoya yazılır, sonra senkronize edilir.
Gerçek mobil dayanıklılığı, “internet yokken bile çalışan uygulama”
tasarlamakla başlar.

Offline-First Mutations: TanStack Query ile “ Önce
Lokal DB, Sonra Sunucu” Modeli
Mobil cihazlarda ağ sürekli kesildiği için mutasyonların (veri gönderme işlemleri)
her zaman sunucuya ulaşması garanti değildir. TanStack Query, “offline-first”
çalışma modelini destekleyerek önce verinin lokal depoya kaydedilmesine, ardından
bağlantı geldiğinde sunucuya otomatik gönderilmesine izin verir.
1. Temel Prensip
1. Kullanıcı bir mutasyon başlatır (ör. form gönderimi).
2. Veri önce lokal depoya (MMKV, SQLite, WatermelonDB) yazılır.
3. TanStack Query mutasyonu kuyruğa alır.
4. ˙Internet yoksa sunucuya gönderilmez.
5. Bağlantı geri geldiğinde mutasyon otomatik olarak yeniden çalışır.
Offline-first mutation = “Veriyi önce sakla, sonra fırsat bulunca
gönder.”
2. TanStack Query Ayarı
Aşağıdaki ayar, bir mutasyonun offline modda kuyruğa alınmasını sağlar:
typescript useMutation( mutationFn: sendToServer, retry: true, network-
Mode: ’offlineFirst’ );
networkMode=’offlineFirst’ TanStack Query’e şu talimatı verir:
• ˙Internet yoksa mutasyonu çalıştırma.
• Kuyruğa ekle ve beklet.
• Bağlantı gelince otomatik gönder.
3. Lokal Veri Yazımı
Mutasyon tetiklendiğinde veri önce lokal DB’ye kaydedilir:
typescript saveLocal(data); // MMKV veya SQLite mutate(data); // Kuyruğa
alınır
Böylece kullanıcı çevrimdışı olsa bile uygulama akışı kesilmez.

Mobil Ağ Hataları: Server Unreachable, No In-
ternet, Flaky Network
Mobil cihazlar masaüstüne göre çok daha fazla ve tahmin edilemez ağ hatası
üretir. Bu hatalar çoğu zaman cihaz çevrimdışıyken değil, çevrimiçi görünmesine
rağmen yaşanır. Bu bölüm mobilde en yaygın ağ hata türlerini özetler.
1. No Internet (Gerçekte internet yok)
Cihaz bağlantılı görünse bile aşağıdaki durumlarda internet yoktur:
• Wi-Fi bağlı fakat modem internete çıkamıyor.
• LTE sinyali var ama operatör veri aktarımını kesmiş.
• Captive portal (otel / havaalanı Wi-Fi) login gerektiriyor.
Bu durumda API istekleri hemen timeout olur.
2. Server Unreachable (Sunucuya hiç ulaşılamıyor)
Aşağıdaki durumlarda sunucuya bağlantı kurmak mümkün olmaz:
• DNS çözülemiyor (DNS lookup failure).
• Baz istasyonu değişiminde IP resetleniyor.
• Gateway, yeni bağlantıyı kabul etmiyor.
Belirti:
Cihaz çevrimiçi, fakat uygulama sürekli “network error” veriyor.
3. Flaky Network (Dalgalı ağ)
Flaky network, “var gibi ama stabil değil” anlamına gelir.
Belirtiler:
• Paket kaybı yüksek.
• RTT (latency) sürekli değişiyor.
• Baz istasyonu zayıf sinyal veriyor.
Sonuç:
• WebSocket sık sık kopar.
• Video call kalitesi bozulur.
• API istekleri bazen çok hızlı, bazen çok yavaş olur.

4. Wi-Fi ↔ LTE Geçişi
Mobil cihaz ağ geçişi sırasında:
• IP adresini değiştirir.
• DNS cache temizlenir.
• Tüm TCP bağlantıları kopar.
Bu nedenle:
• Upload / download yarıda kalır.
• WebSocket kapanır.
• ˙Ilk API çağrısı çok yavaş olur.
5. Timeout Hatalarının Nedenleri
Timeout genellikle üç nedenle oluşur:
1. DNS çözümü başarısız.
2. TCP bağlantısı kurulamadı.
3. Paket kaybı veya gecikme çok yüksek.
6. Mobil ˙Için En ˙Iyi Korunma Yöntemleri
• Retry + exponential backoff kullan.
• Timeoutları kısa tut, ilk paket gelmezse hemen bırak.
• Reachability ile gerçek ağ durumunu izle.
• Offline-first stratejilerle veri kaybını önle.
• Chunked upload/download ile kopmalara dayanıklı ol.
Özet
• Mobil ağlar çoğu zaman “bağlı gibi görünüp bağlı olmayan” durumlar
yaratır.
• DNS, paket kaybı ve ağ geçişleri mobil hataların ana sebepleridir.
• Dayanıklı bir uygulama için timeout, retry, reachability ve offline-first
stratejileri zorunludur.
Gerçek mobil dayanıklılığı, ağ mükemmelmiş gibi değil, ağ kötüymüş
gibi tasarım yaparak elde edilir.
