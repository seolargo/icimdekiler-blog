GraphQL Nedir?
GraphQL, API’ler için kullanılan bir sorgu dili ve bu sorguları çalıştıran sunucu taraflı bir
çalışma zamanı (runtime)dır. Temel yaklaşım şu şekilde özetlenebilir:
• İstemci (mobil uygulama) tam olarak hangi alanlara ihtiyaç duyduğunu bildirir.
• Sunucu, ızca talep edilen alanları geri döndürür.
GraphQL’in başlıca özellikleri şu şekildedir:
• Belirli bir veritabanına veya depolama motoruna bağımlı değildir; mevcut arka uç (backend)
kodunun ve verinin üzerine konumlandırılır.
• 2015 yılında açık kaynak olarak yayımlanan bir spesifikasyona dayanır.
• Pek çok programlama dili ve ekosistem için hem sunucu (server ) hem de istemci (client)
kütüphaneleriyle desteklenmektedir.
Klasik REST yaklaşımında “şu endpoint’e git, ne verirse al” anlayışı ağırlıktayken, GraphQL’de
istemci “yalnızca şu alanları gönder” diyerek çok daha kontrollü veri çekimi sağlayabilir.

REST ve GraphQL Arasındaki Temel Farklar
REST Yaklaşımı
REST tabanlı API’lerde çoğunlukla her kaynak türü için ayrı endpoint’ler tanımlanır:
• /users
• /users/5
• /users/5/posts
Bu modelin mobil tarafta sık karşılaşılan sorunları aşağıdaki gibidir:
Over-fetching İlgili ekran için gerekli olandan çok daha fazla alanın istemciye gönderilmesi-
dir.
Under-fetching Tek bir istekle ihtiyaç duyulan tüm verinin elde edilememesi ve ek endpoint
çağrılarına ihtiyaç duyulmasıdır.
Çoklu istek gecikmesi Her REST çağrısı:
• Ayrı bir HTTP isteği,
• Ayrı bir ağ (network) gidiş-dönüş süresi (round-trip),
• Ayrı bir JSON çözümleme (parse) maliyeti anlamına gelmektedir.
Özellikle hücresel ağların kullanıldığı mobil ortamlarda bu ek istekler, belirgin düzeyde gecik-
meye (latency) ve pil tüketiminin artmasına neden olur.
GraphQL Yaklaşımı
GraphQL servisleri genellikle tek bir endpoint üzerinden hizmet verir:
/graphql
İstemci, bu endpoint’e GraphQL sorgusunu (query) HTTP isteğinin gövdesinde iletir. Örne-
ğin:
POST /graphql
{
"query": "query { users { id name email } }"
}
Sunucu, öncelikle sorguyu tanımlı şemaya göre doğrular, ardından arka planda çalışan resolver
fonksiyonlarını çağırarak sonucu üretir ve yanıtı JSON formatında döndürür.
Bu model ile:

• İhtiyaç duyulan alanlar tek tek seçilebilir.
• Birden fazla REST isteği ile elde edilebilecek ilişkili veriler tek bir GraphQL sorgusunda
bir araya getirilebilir.
• Gönderilen ve paketlenen veri miktarı (örneğin 1 MB yerine 100 KB) çok daha sıkı
biçimde kontrol edilebilir.
Tip Sistemi ve Şema (Schema)
GraphQL, API’nin tipler ve alanlar üzerinden açık biçimde tanımlanmasını gerektirir. Bu ta-
nımlamaya schema adı verilir.
Örneğin, oturum açmış kullanıcının adını döndüren basit bir şema şu şekilde olabilir:
type Query {
me: User
}
type User {
name: String
}
Bu tiplerin arkasında, her alan için veriyi sağlayan resolver fonksiyonları bulunmaktadır.
Sadece İhtiyaç Duyulan Verinin İstenmesi
Sunucu başlatıldıktan sonra istemciler GraphQL sorguları gönderebilir. Örneğin:
{
me {
name
}
}
Bu sorgunun olası JSON yanıtı aşağıdaki gibi olacaktır:
{
"data": {
"me": {
"name": "Luke Skywalker"
}
}
}

Burada dikkat edilmesi gereken hususlar şunlardır:
• Sorgunun yapısı ile dönen verinin yapısı bire bir örtüşmektedir.
• İstemci yalnızca name alanını talep ettiği için, yanıtta başka herhangi bir alan yer alma-
maktadır.
Mobil bakış açısıyla bu durum şu sonuçları doğurur:
• İlgili ekran için gereksiz olan alanlar (address, settings, logs vb.) sunucu tarafından
hiç gönderilmez.
• Toplam veri boyutunu, örneğin 1 MB düzeyinden 100–200 KB seviyesine indirmek mümkün
hale gelir.

Mobil Perspektiften Performans: 1 MB Yerine 100 KB
REST Senaryosu (Örnek)
Bir mobil uygulamanın ana ekranı (Home) göz önünde bulundurulsun. Bu ekranda:
• Kullanıcının temel bilgileri,
• Öne çıkan birkaç içerik (örneğin romantik mekânlar),
• Küçük bir promosyon alanı
gösterildiği varsayılsın.
REST ile tipik istekler şu şekilde olabilir:
• /me → kullanıcı bilgileri (∼ 150 KB)
• /spots → tüm spot’ların tam içerikleri (∼ 800 KB)
• /promotions → promosyon bilgileri (∼ 100 KB)
Bu durumda toplamda:
• Yaklaşık ∼ 1 MB veri,
• 3 ayrı HTTP isteği,
• 3 ayrı ağ gecikmesi ve JSON çözümleme maliyeti söz konusu olur.
Mobil ağ kullanıldığında bu, daha uzun bekleme süreleri, daha yüksek veri tüketimi ve artan pil
kullanımı anlamına gelmektedir.
Aynı Senaryo GraphQL ile
Aynı ekran için gerçekte yalnızca aşağıdaki bilgilere ihtiyaç duyulduğu varsayılsın:
• Kullanıcı için: id, gender, language
• Spotlar için: id, title, thumbnailUrl, birkaç vibe etiketi
• Promosyonlar için: id, title, ctaText
Bu durumda kullanılabilecek örnek bir GraphQL sorgusu şu şekildedir:
query HomeScreen {
me {
id
gender
language
}

featuredSpots(limit: 5) {
id
title
thumbnailUrl
vibes
}
promotions(limit: 2) {
id
title
ctaText
}
}
Bu yaklaşımın sonucunda:
• Yalnızca tek bir HTTP isteği gönderilir.
• Yalnızca gerekli alanları içeren görece küçük bir JSON yanıt elde edilir.
• GZip/Brotli gibi sıkıştırma yöntemleriyle birlikte, 1 MB yerine 100–200 KB aralığında
bir payload boyutuna ulaşmak çoğu senaryoda mümkündür.
Önemli Not: GraphQL Sihirli Bir Çözüm Değildir
GraphQL, tek başına tüm veri hacmini otomatik olarak küçültmez; asıl kontrol şema ve sorgu
tasarımındadır:
• Sorgu içerisine çok fazla ilişki, çok sayıda alan ve binlerce öğe dâhil edilirse, yanıt boyutu
yine önemli ölçüde artabilir.
• Performans açısından:
– Her ekran için mantıklı ve sınırlı kapsamlı sorgular tasarlamak,
– Sayfalama (limit, offset, cursor ) kullanmak,
– Liste ekranlarında önce hafif alanları, detay ekranlarında ise daha ağır alanları çağır-
mak
önerilen yaklaşımlar arasındadır.

API’yi Versiyonlamadan Evrimleştirmek
İstemci gereksinimleri zaman içerisinde değişmektedir. GraphQL’in en önemli avantajlarından
biri, API’nin yeni sürümler tanımlamaya gerek kalmaksızın evrimleşmesine olanak tanı-
masıdır.
Örneğin başlangıçta yalnızca name alanına sahip bir User tipi tanımlanmış olsun:
type User {
name: String
}
Daha sonra, istemci tarafında ortaya çıkan yeni bir gereksinim kapsamında daha ayrıntılı ad
alanları tanımlanmak istenebilir:
type User {
fullName: String
nickname: String
name: String @deprecated(reason: "Use ‘fullName‘.")
}
Bu durumda:
• name alanı @deprecated ile işaretlenmiştir; böylece yeni istemcilerin fullName alanını
kullanması teşvik edilir.
• İstemci tarafında kullanılan araçlar (örneğin IDE eklentileri) geliştiricilere uyarı gösterir.
• Eski alanın artık hiçbir istemci tarafından kullanılmadığından emin olunduğunda, bu alan
şemadan tamamen kaldırılabilir.
Bu sayede, REST tabanlı mimarilerde sıkça karşılaşılan /v1, /v2 benzeri ağır versiyonlama
süreçleri yerine, şema düzeyinde kademeli bir evrim sağlanmış olur.

Mobil Geliştiriciler İçin Önerilen Uygulamalar
Mobil senaryolarda GraphQL kullanılırken dikkat edilmesi önerilen başlıklar aşağıda özetlenmiş-
tir:
• Ekran bazlı sorgular tasarlamak: Her ekran için mantıklı ve sınırlı kapsamlı bir Grap-
hQL sorgusu tanımlanmalıdır. Aşırı genel, tüm veriyi döndürmeye çalışan tek bir super-
query tasarımından kaçınılmalıdır.
• Over-fetching’i azaltmak: Yalnızca ilgili ekranda gerçekten kullanılan alanların isten-
mesi, gereksiz alanların sorgulardan tamamen çıkarılması önemlidir.
• Sayfalama kullanmak: Uzun listeler (örneğin haber akışı, mesaj listeleri vb.) mutlaka
sayfalama ile tasarlanmalıdır. Örnek: limit: 20 ve cursor tabanlı sayfalama.
• Hafif ve ağır payload’ları ayırmak: Liste ekranlarında hafif alanlara, detay ekranlarında
daha ağır ve kapsamlı alanlara yer verilmesi, performans açısından faydalıdır.
• Tek istekle birden fazla kaynağa erişmek: Aynı ekranda normalde farklı REST end-
point’lerinden elde edilecek verileri tek bir GraphQL sorgusunda toplamak, özellikle mobil
ortamlarda toplam gecikmenin azaltılmasına anlamlı katkı sağlar.
Mobil geliştirme perspektifinden bakıldığında GraphQL:
• Veri boyutunun kontrol edilmesini kolaylaştırmakta (örneğin 1 MB yerine 100 KB seviye-
leri),
• Fazla sayıdaki REST endpoint bağımlılığını azaltarak daha düşük gecikme ve daha az
ağ trafiği sağlamaya yardımcı olmakta,
• API’nin zaman içerisinde yeni sürümler tanımlamaya gerek kalmaksızın evrimleşmesine
olanak tanımaktadır.

GraphQL ile Tüm Veriyi Tek Seferde Çekmek
Mobil uygulama mimarisinde zaman zaman, “tüm veriyi tek bir GraphQL sorgusu ile çekip daha
sonra tamamen yerel (local) bellekten kullanmak” şeklinde bir yaklaşım gündeme gelebilmektedir.
Bu bölümde, söz konusu yaklaşımın hangi durumlarda uygun olabileceği ve hangi durumlarda
sakıncalı sonuçlar doğurabileceği ele alınmaktadır.
Tek Seferde Tüm Veriyi Çekme Yaklaşımı
Bu yaklaşımda temel fikir, aşağıdakine benzer kapsamlı bir sorgu ile uygulamanın ihtiyaç duya-
cağı hemen hemen tüm verilerin tek seferde elde edilmesidir:
query InitApp {
me { ...birçok alan }
allCategories { ...tüm alanlar }
allSpots { ...tüm alanlar }
allQuestions { ...tüm alanlar }
allFestivals { ...tüm alanlar }
# vb.
}
Elde edilen yanıt, genellikle tek bir büyük nesne (örneğin bir global store veya yerel önbellek)
içerisinde saklanmakta; uygulama ekranları açıldıkça yeni ağ isteği yapılmaksızın bu nesneden
okunmaktadır.
Teorik olarak bu yaklaşım, ilk açılışta nispeten daha fazla bekleme süresi karşılığında, sonrasında
hızlı ve tamamen yerel okuma imkânı sunmayı amaçlamaktadır. Ancak uygulamada bu model
her veri türü için uygun değildir.
Uygun Olduğu Durumlar
Aşağıdaki veri türleri için tüm veriyi tek seferde çekmek çoğu zaman anlamlı ve etkilidir:
• Küçük ve statik referans veriler: Cinsiyet seçenekleri, dil listeleri, sabit kategori türleri,
ikon eşlemeleri gibi boyutu düşük ve nadiren değişen sözlük (dictionary) verileri.
• Konfigürasyon ve özellik bayrakları (feature flags): Örneğin, belirli bir özelliğin aktif
olup olmadığı, minimum desteklenen uygulama sürümü gibi uygulama genelini ilgilendiren
ayarlar.
• Sınırlı boyutta sabit içerik katalogları: Örneğin 30–100 öğe aralığında kalan, içeriği
sık değişmeyen küçük kataloglar.
Bu tür veriler, uygulamanın ömrü boyunca sıkça başvurulan, ancak boyutu görece küçük olan
veri kümeleri olduğundan, tek seferde alınarak yerel bellekte veya kalıcı bir depoda (örneğin bir
key-value store) tutulmaları performans açısından avantaj sağlayabilir.

Sakıncalı Olduğu Durumlar
Aşağıdaki veri tipleri için tüm veriyi tek bir sorgu ile çekmeye çalışmak genellikle önerilmemek-
tedir:
• Büyüyen ve kullanıcı tarafından üretilen veriler: Sohbet geçmişleri, kullanıcıların
oluşturduğu içerikler, büyük favori listeleri veya zamanla artan kayıtlar gibi veri kümeleri,
kullanıcı sayısı ve kullanım süresi arttıkça hızla büyüyebilir. Tüm bu veriyi tek seferde
çekmek, başlangıçta kabul edilebilir görünse bile zaman içinde büyük boyutlara ulaşabilir.
• Sayfalama gerektiren uzun listeler: Haber akışı, keşfet sayfaları, geniş festival veya
mekân listeleri gibi içerikler için sayfalama (pagination) kaçınılmazdır. Tüm listeyi tek
seferde istemek, hem ağ hem de bellek açısından verimsizdir; kullanıcı çoğu zaman bu
içeriğin yalnızca küçük bir bölümünü görmektedir.
• Sık güncellenen veya gerçek zamanlıya yakın veriler: Kampanyalar, bildirimler,
canlı skorlar gibi verilerin tek bir başlangıç sorgusu ile indirilip uzun süre güncellenmemesi,
kullanıcının eski (geçerliliğini yitirmiş) verilerle karşılaşmasına neden olabilir. Her açılışta
aynı büyük sorguyu tekrar çalıştırmak ise ağ yükünü arttırır.
• Bellek ve performans etkileri: Tek bir büyük sorgu sonucunda:
– Büyük miktarda JSON verisinin ayrıştırılması (parse) ek CPU maliyeti oluşturur,
– Uygulamanın JavaScript bellek alanında çok büyük nesneler tutulmasına neden olur,
– Bu durum çöp toplama (garbage collection) sıklığını artırarak arayüzde takılmalara
(jank) yol açabilir.
GraphQL’in Amacı Açısından Değerlendirme
GraphQL’in temel kazanımı, istemciye yalnızca ihtiyaç duyduğu alanları ve ilişkileri seçme imkânı
vermesidir. Tüm veriyi tek bir sorgu ile istemek, bu ince taneli kontrol mekanizmasını kullan-
mamak anlamına gelir. Bu durumda, GraphQL kullanıyor olmanın getirdiği avantajlar önemli
ölçüde azalmaktadır.
Dolayısıyla, her şeyi tek bir sorguyla almak yerine, GraphQL sorgularını ekran bazlı, ihtiyaç
odaklı ve boyut olarak sınırlı şekilde tasarlamak, hem teknolojinin ruhuna hem de mobil
performans gereksinimlerine daha uygundur.

Önerilen Hibrit Yaklaşım
Pratikte en sağlıklı model, hibrit bir stratejidir:
• Başlangıç (bootstrap) sorgusu: Uygulama ilk açıldığında veya oturum açma sonrasında,
temel kullanıcı bilgileri, uygulama konfigürasyonu ve küçük referans veri setlerini içeren
sınırlı bir InitApp sorgusu çalıştırılabilir. Örneğin:
query InitApp {
me {
id
gender
language
premiumStatus
}
appConfig {
minVersion
features {
name
enabled
}
}
staticDictionaries {
genders { code label }
relationshipGoals { code label }
}
}
Bu sorgunun yanıtı, uygulamanın durum yönetiminde veya yerel saklama katmanında tu-
tulabilir ve ekranlar bu bilgilerden yararlanabilir.
• Ekran bazlı sorgular: Her ekran, yalnızca kendi ihtiyaç duyduğu veri alanlarını içeren,
uygun ölçüde küçük ve genellikle sayfalı (paginated ) sorgular ile desteklenmelidir. Örneğin:
– Ana ekran için featuredSpots(limit: 5) ve todayPromotions,
– Festival listesi için festivals(page: 1, limit: 20),
– Soru listesi için questionsByCategory(categoryId, limit: 50).
Bu sorguların sonuçları da uygun görüldüğünde yerel önbellekte saklanabilir ve arka planda
yenileme (refresh) mekanizmalarıyla güncel tutulabilir.

Temel Tip Kavramları
Bu bölümde GraphQL tip sisteminin temel kavramları açıklanmakta ve her kavram için kısa, pra-
tik örnekler verilmektedir. Örnekler, mobil uygulama geliştirme bağlamına yakın olacak şekilde
seçilmiştir.
Schema (Şema) Nedir?
Tanım: Schema, bir GraphQL API’sinden hangi verilerin istenebileceğini, hangi tiplerin ve alan-
ların mevcut olduğunu ve bu alanların tiplerini tanımlayan yapıdır. Kısaca, API’nin sözlüğü
olarak düşünülebilir. Örneğin bir ilişki uygulamasında basit bir şema parçası şu şekilde olabilir:
type User {
id: ID!
name: String!
gender: String
}
type Query {
me: User
}
Bu tanıma göre:
• Sistemde bir User tipi vardır.
• User tipinin alanları:
– id: ID! (her zaman dolu, null olamaz),
– name: String! (her zaman dolu),
– gender: String (boş olabilir).
• Query kök tipinde me adında bir alan vardır ve bu alan bir User döndürür.
Mobil geliştirici açısından, şema sayesinde:
• Hangi alanların sorgulanabileceği,
• Hangi alanların zorunlu, hangilerinin isteğe bağlı olduğu,
• Bir alanın tipinin ne olduğu
gibi bilgiler otomatik olarak öğrenilebilir.

Object Type Nedir?
Tanım: Object Type, belirli bir nesne türünü ve bu nesneye ait alanları tanımlayan tiptir. Nesne
yönelimli programlamadaki class User { name; age; } yapısına benzetilebilir. GraphQL’de
type User { ... } biçiminde ifade edilir.
Örneğin romantik mekan kartlarını temsil eden bir tip:
type Spot {
id: ID!
title: String!
city: String!
imageUrl: String
}
Bu tanım:
• Spot adında bir Object tipin olduğunu,
• id, title, city alanlarının zorunlu olduğunu,
• imageUrl alanının ise isteğe bağlı (null olabilir) olduğunu
ifade eder.
Basit bir sorgu örneği:
{
romanticSpots(limit: 2) {
id
title
city
}
}
Olası JSON yanıtı:
{
"data": {
"romanticSpots": [
{ "id": "1", "title": "Kaputaş Plajı", "city": "Antalya" },
{ "id": "2", "title": "Pokut Yaylası", "city": "Rize" }
]
}
}

Scalar Type Nedir?
Tanım: Scalar tipler, bir sorgunun uç (leaf) noktalarındaki basit değerleri temsil eder. Altında
başka alan içermezler. Örnekler: Int, Float, String, Boolean, ID.
Örneğin:
type User {
name: String!
age: Int
isPremium: Boolean!
}
Burada:
• name: String! (zorunlu),
• age: Int (isteğe bağlı),
• isPremium: Boolean! (zorunlu)
alanlarının her biri birer Scalar tiptir.
Sorgu:
{
me {
name
age
isPremium
}
}
Örnek yanıt:
{
"data": {
"me": {
"name": "Ömer",
"age": 30,
"isPremium": true
}
}
}
Scalar değerler tek parça olduğu için, bir scalar alanın altına ek alt alan seçimi yapılamaz (örneğin
me.name.something gibi bir sorgu geçersizdir).

Non-Null (!) Nedir?
Tanım: GraphQL’de tipler varsayılan olarak nullable kabul edilir; yani bir alan null döndüre-
bilir. Bir tipin sonuna ! eklendiğinde, o alanın veya argümanın null olmaması gerektiği ifade
edilir. Sunucu null döndürmeye çalışırsa, GraphQL çalıştırma aşamasında hata üretir.
Örneğin:
type User {
id: ID! # null olamaz
name: String! # null olamaz
bio: String # null olabilir
}
Bu tanımda:
• id ve name alanlarının her durumda bir değer döndürmesi gerekir,
• bio alanı boş bırakılabilir (örneğin yeni kayıt olan bir kullanıcı henüz biyografi girmemiş
olabilir).
Mobil arayüz tasarımında bu bilgi, zorunlu alanlar için boş değer kontrolü derdini azaltır ve
isteğe bağlı alanlar için uygun fallback metinlerin gösterilmesini mümkün kılar.

Liste ([]) Nedir?
Tanım: Köşeli parantezler ([]), ilgili alanın bir dizi (liste) döndürdüğünü veya bir argümanın
dizi beklediğini ifade eder. Örneğin: [String], [Episode!], [Spot!]!.
Örnek bir tanım:
type User {
id: ID!
favoriteSpotIds: [ID!]!
}
Burada favoriteSpotIds alanı için:
• Alan her zaman bir liste döndürür (alanın kendisi null olamaz),
• Liste boş olabilir ([]),
• Listenin içindeki her bir öğe ID! tipindedir ve null olamaz.
Geçerli ve geçersiz örnekler:
favoriteSpotIds: null # geçersiz (Non-Null liste)
favoriteSpotIds: [] # geçerli (hiç favorisi yok)
favoriteSpotIds: ["1", "2"] # geçerli
favoriteSpotIds: ["1", null] # geçersiz (ID! öğeleri null olamaz)
Bu davranış, mobil tarafta her zaman bir dizi bekleyebileceğiniz anlamına gelir; böylece listele-
melerde (örneğin FlatList) ek null kontrollerine daha az ihtiyaç duyarsınız.

Argüman (Field Parametresi) Nedir?
Tanım: GraphQL’de her alan, bir fonksiyon gibi davranarak, sıfır veya daha fazla sayıda isim-
lendirilmiş argüman alabilir. Bu argümanlar, sorgu sırasında alanın nasıl davranacağını belirler.
Örnek:
type Query {

searchSpots(city: String, limit: Int = 10): [Spot!]!
}
Bu tanımda:
• spot(id: ID!) alanı, id isimli zorunlu bir argüman alır; id için null değer verilemez.
• searchSpots(city, limit = 10) alanında:
– city isteğe bağlıdır,
– limit verilmezse varsayılan olarak 10 kabul edilir.
Sorgu örneği:
{
searchSpots(city: "˙Istanbul", limit: 5) {
id
title
}
}

Root Tipler: Query, Mutation, Subscription
Tanım: Root tipler, GraphQL şemasının giriş noktalarını temsil eder:
• Query: Okuma (sorgulama) operasyonlarının kök tipi,
• Mutation: Yazma/değiştirme operasyonlarının kök tipi,
• Subscription: Gerçek zamanlı veri akışı (örneğin WebSocket) için kök tip.
Örneğin:
type Query {
me: User
romanticSpots(limit: Int!): [Spot!]!
}
type Mutation {
addFavorite(spotId: ID!): User
}
Mobil geliştirici açısından:
• Query alanları liste ve detay ekranlarını beslemek için kullanılır,
• Mutation alanları, “favoriye ekle”, “beğen”, “yorum gönder” gibi işlem butonlarının arka-
sındaki operasyonlarda kullanılır,
• Subscription ise canlı sohbet, canlı skor gibi gerçek zamanlı senaryolarda devreye girer
(temel kullanım için zorunlu değildir).

Interface Nedir?
Tanım: Interface tipler, soyut tiplerdir ve kendisini implemente eden Nesne tiplerinin sahip ol-
ması gereken ortak alanları tanımlar. Böylece, farklı tipler arasında ortak bir kontrat oluşturulmuş
olur.
Örneğin:
interface Content {
id: ID!
title: String!
}
type Spot implements Content {
id: ID!
title: String!

type Festival implements Content {
id: ID!
title: String!
country: String!
}
Burada:
• Content interface’i, her içeriğin en az id ve title alanlarına sahip olacağını belirtir.
• Spot ve Festival tipleri, bu interface’i implemente ederek bu ortak alanları sağlamak
zorundadır.

title
}

sorgusu, homeFeed içerisinde hem Spot hem de Festival nesneleri bulunsa dahi, her biri için
id ve title alanlarına güvenle erişebildiğiniz anlamına gelir.

title

country
}

anlamak ve tipe özgü alanları sorgulamak için inline fragment ve __typename meta-alanı kul-
lanılır:
{

__typename
... on Spot {
title
city

}
... on Festival {
title
city
}
... on User {
name
}
}
}

"title": "Galata Kulesi",

Input Object Tipler
Input Object tipler, alan veya directif argümanlarının daha karmaşık, nesne benzeri yapılar
halinde gruplanmasını sağlayan tiplerdir. Özellikle, çok sayıda parametre alan sorgu filtreleri
veya mutation argümanları için faydalıdır.
Örneğin, romantik mekanları belirli filtrelere göre aramak istediğimizi varsayalım. Bireysel argü-
manlar yerine, tek bir filtre nesnesi tanımlanabilir:
input SpotFilterInput {
city: String
minRating: Float
maxBudget: Int
}
type Spot {
id: ID!
title: String!
city: String!
rating: Float
}
type Query {
searchSpots(filter: SpotFilterInput): [Spot!]!
}
Bu tanımda SpotFilterInput, searchSpots alanının filter argümanı için kullanılan bir
Input Object tipidir. İstemci tarafında bu argüman şu şekilde kullanılabilir:
{
searchSpots(filter: {
city: "˙Izmir"
minRating: 4.5
}) {
id
title
city
rating
}
}
Bu yaklaşımın avantajları şunlardır:
• İlgili argümanlar tek bir yapı altında mantıksal olarak gruplanır.

• Argüman sayısı arttığında şema tanımı daha derli toplu kalır.
• İstemci tarafında, tek bir nesne (filter) hazırlanarak sorguya parametre olarak verilebilir.
Input Object tipler yalnızca giriş (input) bağlamında kullanılır; çıktı (output) olarak bir alanın

Directifler
Directifler, @ karakteri ile başlayan ve şema veya sorgu parçalarının nasıl yorumlanacağını etki-
leyen açıklayıcı yapılardır. Tip sistemi directifleri, şemadaki tiplerin, alanların ve argümanların
doğrulanma veya yürütülme biçimini değiştirmek için kullanılabilir.
GraphQL spesifikasyonu, bazı yerleşik directifleri tanımlar. Bunlardan biri de, artık kullanılması
önerilmeyen alanların işaretlenmesi için kullanılan @deprecated directifidir:
type User {
fullName: String
name: String @deprecated(reason: "Use ‘fullName‘.")
}
Bu örnekte:
• name alanı hâlen şemada mevcuttur,
• Ancak @deprecated ile işaretlendiği için, yeni istemcilerin fullName alanını kullanması
teşvik edilir,
• GraphQL araçları (örneğin geliştirme ortamı eklentileri) name alanı sorgulandığında uyarı
gösterebilir.
@deprecated directifinin olası tanımı SDL ile şu şekilde ifade edilebilir:
directive @deprecated(
reason: String = "No longer supported"
) on FIELD_DEFINITION | ENUM_VALUE
Bu tanımda:
• reason argümanı, varsayılan değeri olan opsiyonel bir Stringtir,
• Directifin uygulanabileceği yerler, FIELDDEF IN IT ION veENUMV ALU Eilesßnßrlandßrßlmßtßr.
Benzer biçimde, uygulamaya özgü özel (custom) directifler de tanımlanabilir; bu directiflerin nasıl
yorumlanacağı, kullanılan GraphQL kütüphanesinin ve sunucu implementasyonunun sorumlulu-
ğundadır.

Dokümantasyon: Açıklamalar ve Yorumlar
GraphQL, şemadaki tipler, alanlar ve argümanlar için dokümantasyon eklenmesini teşvik eder.
Bu dokümantasyon iki ana mekanizma ile sağlanır: açıklamalar (descriptions) ve yorumlar (com-
ments).
Açıklamalar (Descriptions)
Açıklamalar, Markdown sözdizimi destekleyen metinlerdir ve şemadaki tiplerin, alanların ve ar-
gümanların üzerinde yer alır. SDL’de çoğunlukla üç tırnak işareti ile tanımlanırlar:
"""
A character from the Star Wars universe
"""
type Character {
"The name of the character."
name: String!
}
Benzer şekilde, bir Enum tipi için:
"""
The episodes in the Star Wars trilogy
"""
enum Episode {
"Star Wars Episode IV: A New Hope, released in 1977."
NEWHOPE
"Star Wars Episode V: The Empire Strikes Back, released in 1980."
EMPIRE
"Star Wars Episode VI: Return of the Jedi, released in 1983."
JEDI
}
Bu açıklamalar:
• Şemayı insanlar için daha anlaşılır ve kendini ifade eden hâle getirir,
• Introspection sorguları aracılığıyla istemci tarafında da erişilebilir,
• GraphiQL ve benzeri geliştirme araçlarında, otomatik tamamlama ve yardım metinleri
olarak görüntülenir.
Yorumlar (Comments)
Bazen şemaya, doğrudan tip veya alan açıklaması olmayan, yalnızca geliştiriciye yönelik notlar
eklemek gerekebilir. Bu tür notlar, yorum olarak eklenir ve GraphQL ayrıştırıcısı tarafından yok
27

-- 27 of 116 --

sayılır. SDL’de tek satırlık yorumlar # karakteri ile başlar:
# This line is treated like whitespace and ignored by GraphQL
type Character {
name: String!
}
Yorumlar, yalnızca şemada değil, istemci sorgularında da kullanılabilir:
{
hero {
name
# Queries can have comments!
friends {
name
}
}
}
Bu yorumlar, sunucu tarafından işlenmez; sorgunun okunabilirliğini artırmak amacıyla istemci
tarafında kullanılır.
Özetle:
• Union tipler, tek bir alanın birden fazla farklı Nesne tipinden birini döndürebilmesini
sağlar ve bu tipler arasında ortak alan tanımı yapmaz.
• Input Object tipler, özellikle filtreler ve mutation argümanları gibi karmaşık giriş verilerini
tek bir nesne altında toplamak için kullanılır.
• Directifler, tipler, alanlar ve argümanların doğrulanma ve yürütülme biçimini etkilemek
için şema veya sorgu üzerinde ek meta bilgi sağlar; @deprecated bunun yaygın bir örne-
ğidir.
• Açıklamalar ve yorumlar, GraphQL şemasının dokümantasyonunu zenginleştirir; açık-
lamalar istemci tarafından da görülebilen, yorumlar ise ayrıştırıcı tarafından yok sayılan
notlardır.

Sorgu Seviyesinde Temel-İleri Kavramlar
Bu bölümde, GraphQL sorgularında sık kullanılan ve pratikte büyük fayda sağlayan kapsamlı
kavramlar ele alınmaktadır.
Query Nedir, Nereden Başlar?
GraphQL’de veri okumak için yapılan operasyona query denir.
Her query, şemadaki root Query tipinden başlar:
type Query {
hero: Character
}
Örnek sorgu:
{
hero {
name
}
}
Olası cevap:
{
"data": {
"hero": {
"name": "R2-D2"
}
}
}
Önemli noktalar:
• Yanıt her zaman en üst seviyede data alanı içinde döner.
• Hata varsa ayrıca errors alanı bulunur.
• Sorgunun yapısı ile dönen verinin yapısı birebir aynıdır.
Field ve Alt Field Mantığı
GraphQL’de sorgu yazarken temel olarak şu yapılır: “şu field’ı ver, bunun altındaki şu field’ları
da ver”.
Örneğin, alt alanlarıyla birlikte bir sorgu:
29

-- 29 of 116 --

{
hero {
name
friends {
name
}
}
}
Burada:
• hero alanı bir Character döndürmektedir.
• friends alanı bir liste (dizi) döndürmektedir.
• Listedeki her bir friend için name alanı istenmektedir.
Notlar:
• Sorgu yazarken tekil veya liste için kullanılan sözdizimi aynıdır.
• Bir alanın tekil mi yoksa liste mi döndürdüğü, şemadaki tip tanımından anlaşılır (Character
vs. [Character!]!).

name
}
}

• Query string’ini çalışma zamanında string birleştirerek üretme ihtiyacı ortadan kalkar.

İşlem Tipi ve İşlem Adı (Operation Type ve Name)
GraphQL sorgularında, yapılan işlemin türü ve isteğe bağlı bir işlem adı belirtilerek sorgular
daha ifade gücü yüksek ve izlenebilir hale getirilebilir.
Temel işlem tipleri:
• query: Veri okuma operasyonları için,
• mutation: Veri değiştirme (yazma) operasyonları için,
• subscription: Gerçek zamanlı veri akışı için.
Basit sorgu kısaltması:
{
hero {
name
}
}

query HeroNameAndFriends {
hero {
name
friends {
name
}
}
}

• query anahtar sözcüğü, bunun bir okuma operasyonu olduğunu belirtir.
• HeroNameAndFriends işlemin adıdır ve hata ayıklama, loglama gibi durumlarda hangi
sorgunun çalıştığını anlamayı kolaylaştırır.
Bir GraphQL dokümanı içinde birden fazla operasyon tanımlanıyorsa işlem adı zorunludur; tek

name
friends {
name
}
}

Bu sorgu, aşağıdaki değişkenler ile birlikte gönderilebilir:
{
"episode": "JEDI"
}

}

Mutations’a Giriş
Mutation Kavramı ve Query ile Farkı
GraphQL’de veri okuma işlemleri query ile, veri üzerinde değişiklik yapan (ekleme, güncelleme,
silme vb.) işlemler ise mutation ile gerçekleştirilir.
• Query: Sunucudaki veriyi okur ve döner; veriyi değiştirmemelidir.
– Örnek: Bir kullanıcının favori mekânlarını listelemek.
• Mutation: Sunucudaki veriyi değiştirir (yeni veri ekler, mevcut veriyi günceller veya siler).
– Örnek: Bir mekânı favorilere eklemek, kullanıcı adını güncellemek, bir kaydı silmek.
Bu bağlamda, SQL analojisiyle:
• SELECT → Query,
• INSERT, UPDATE, DELETE → Mutation
şeklinde düşünülebilir.

banına yeni bir kayıt eklenmesi beklenir. Bu işlem, GraphQL tarafında bir mutation alanı olarak
tanımlanır.
Şema Tanımı

addFavorite(spotId: ID!): Favorite
}
type Favorite {

userId: ID!
spotId: ID!
}
Bu tanıma göre:
• addFavorite(spotId: ID!): Favorite alanı, spotId isimli zorunlu bir argüman
alır.
• Geriye, yeni oluşturulan favori kaydını temsil eden bir Favorite nesnesi döndürür.

Subscriptions’a Giriş
Subscription Kavramı
GraphQL’de veri okuma işlemleri query ile, veri üzerinde değişiklik yapan işlemler (ekleme,
güncelleme, silme vb.) mutation ile gerçekleştirilir. Buna ek olarak, istemcinin sunucudan ger-
çek zamanlı (veya gerçek zamana yakın) güncellemeler almasını sağlayan üçüncü bir
operasyon türü daha vardır: subscription.
Subscription, tek seferlik bir istek–yanıt modelinden farklı olarak, uzun ömürlü bir bağlantı
üzerinden sunucudan gelen olayları dinlemeye imkân tanır. Örneğin:
• Yeni bir yorum (review) oluşturulduğunda haberdar olmak,
• Canlı sohbet mesajlarını almak,
• Canlı skor veya metrik güncellemelerini izlemek.
Subscription Tipi ve Operasyon Tanımı
Subscription alanları, Query ve Mutation alanlarında olduğu gibi şema üzerinde tanımlanır,
ancak Subscription kök operasyon tipi kullanılır:
type Subscription {
reviewCreated: Review
}
İstemci tarafında bir subscription operasyonu, subscription anahtar kelimesi ile başlatılır:
subscription NewReviewCreated {
reviewCreated {
rating
commentary
}
}
Bu örnekte:
• NewReviewCreated operasyon adı,
• reviewCreated ise, yeni bir Review nesnesi yayımlandığında tetiklenecek subscription
alanıdır.
• Seçim kümesinde ({ rating commentary }) hangi alanların her güncellemede istemciye
gönderileceği belirtilmektedir.

Pub/Sub Mantığı ve Taşıma Katmanı
Çoğu gerçek dünyadaki uygulamada, subscription alanlarının arka planında bir pub/sub (pub-
lish/subscribe) sistemi bulunur. Tipik akış şu şekildedir:
• Örneğin bir createReview mutation’ı çalıştığında, sistem bir REVIEW_CREATED mesajı
yayımlar.
• reviewCreated subscription alanının resolver’ı bu olayı dinler.
• İlgili olaya abone olmuş istemcilere, yeni veriyi içeren mesaj gönderilir.
GraphQL spesifikasyonu, subscription’ların hangi taşıma protokolü ile gerçekleştirileceğini ta-
nımlamaz; bu tercih sunucuya bırakılmıştır. Uygulamada çoğunlukla:
• WebSocket veya
• Server-Sent Events (SSE)
kullanılmaktadır. İstemci kütüphanesinin de seçilen protokolü desteklemesi gerekmektedir.

Subscription dokümanları birden fazla operasyon içerebilir; ancak her subscription operasyo-
nunun tam olarak bir adet kök alanı (root field) olmalıdır. Aşağıdaki örnek bu nedenle
geçersizdir:
subscription {
reviewCreated {
rating
commentary
}

name
friends {
name
}
}
}
Buna karşılık, aynı dokümanda birden fazla ayrı isimlendirilmiş subscription operasyonu tanım-
lanabilir:
subscription NewReviewCreated {
reviewCreated {
rating

commentary
}
}
subscription FriendListUpdated($id: ID!) {

}

Bu durumda:
• Dokümanda birden fazla operasyon bulunduğu için her birine isim verilmesi zorunludur.
• İstek gönderilirken hangi operasyonun çalıştırılacağı, operasyon adı ile belirtilir.
Mimarî Zorluklar ve Kullanım Senaryoları
Subscription operasyonları, query ve mutation’a göre daha karmaşık bir mimarî gerektirir:
• Sunucu, subscription süresince GraphQL dokümanını, değişkenleri ve bağlam bilgisini (con-
text) canlı tutmak zorundadır.
• Yatay ölçekleme senaryolarında, her abonenin belirli bir sunucu instance’ına bağlanması
ve bu ilişkinin yönetilmesi gerekir.
• İstemci tarafında:
– Bağlantı koptuğunda otomatik yeniden abone olma,
– İlk query sonuçları ile subscription güncellemeleri arasındaki olası yarış durumlarını
(race condition) yönetme gibi ek mekanizmalar gerekebilir.
Bu nedenle, subscription’lar özellikle:
• Verinin sık ve artımlı (incremental) olarak değiştiği,
• İstemcinin bu değişiklikleri olabildiğince hızlı bir biçimde gösterme ihtiyacı duyduğu,
durumlar için uygundur (canlı sohbet, bildirim akışı, gerçek zamanlı gösterge panelleri, vb.).
Daha seyrek güncellenen veriler için ise aşağıdaki yaklaşımlar daha basit ve genellikle yeterli
olabilir:
• Periyodik polling (belirli aralıklarla tekrar sorgu çekmek),
• Kullanıcı etkileşimine bağlı yeniden sorgulama,
• Mobil push bildirimleri ile tetiklenen veri yenileme.

GraphQL Validation’a Giriş
Validation (Doğrulama) Kavramı
Bir GraphQL isteği sunucuya ulaştığında, iki temel aşamadan geçer:
1. Parse aşaması: İstek dokümanının söz dizimi (syntax) kontrol edilir.
2. Validation aşaması: Doküman, şemaya (schema) göre doğrulanır.
Validation aşamasında şu sorulara yanıt aranır:
• İstenen field’lar gerçekten ilgili tip üzerinde tanımlı mı?
• Selection set’ler tip kurallarına uygun mu (scalar vs. object ayrımı)?
• Fragment’ler doğru tipler üzerinde ve doğru biçimde kullanılıyor mu?
Bu sayede, hatalı sorgular çalıştırılmadan önce tespit edilir ve geliştiriciye anlamlı hata mesajları
döndürülür.
Var Olmayan Field İstemek
Bir field, yalnızca tanımlı olduğu tip üzerinde sorgulanabilir. Örneğin, şemada:
type Character {
name: String!
appearsIn: [Episode!]!
}
tanımı varsa ve favoriteSpaceship alanı Character tipi üzerinde tanımlı değilse, aşağıdaki
sorgu geçersizdir:
# INVALID: favoriteSpaceship does not exist on Character
query {
hero {
favoriteSpaceship
}
}
Bu durumda sunucu, örneğe benzer biçimde aşağıdaki türde bir hata döndürür:
{
"errors": [
{

"message": "Cannot query field \"favoriteSpaceship\" on type \"Character\".
}
]
}

GraphQL’de selection set’ler için iki temel kural vardır:
Obje Döndüren Field’lar için Alt Alan Seçilmesi Zorunluluğu
Eğer bir field, scalar veya enum dışındaki bir tip (örneğin bir Object veya Interface) dön-
dürüyorsa, o field için mutlaka bir selection set belirtilmelidir.
Örnek şema:
type Query {

}
Aşağıdaki sorgu geçersizdir:

Validation aşamasında bir hata tespit edildiğinde:
• İlgili istek için alan yürütmesi (field execution) başlatılmaz.
• Veritabanına veya arka uç (backend) sistemlerine herhangi bir erişim yapılmadan, hata bir

GraphQL sunucusu, bir isteği (request) doğrulayıp çalıştırdıktan (execute) sonra istemciye bir

• errors
• extensions

• ya her istemci, gereğinden fazla veri içeren genel bir uç noktayı çağırmak zorunda kalır

Sistem tasarımı veya deneyim tasarımı yapılırken, öncelikli hedef, iş alanını ve kullanıcı yolcu-

• Kullanıcı yolculukları çıkarılır,

• Hatalı veya olağandışı durumlar düşünülerek sistemin nasıl davranması gerektiği tartışılır.
Bu aşama, esasen düşünme ve keşif aşamasıdır. Amaç, iş alanının sınırlarını, kullanıcı ihtiyaçla-

Pratikte bu yaklaşım şu adımlarla somutlaşır:
1. Öncelikle, iş açısından kritik veya öncelikli olan tek bir senaryo veya tek bir uçtan uca akış

2. Sadece bu senaryoyu baştan sona destekleyecek kadar GraphQL tipi, alanı ve ilişki tanım-

3. Bu ilk model gerçek kullanım ve ekip geri bildirimi ile test edilir; isimlendirmelerin, ilişki-

4. Ardından, bir sonraki senaryo için şema ve model kademeli olarak genişletilir.

GraphQL’in HTTP Üzerinden Sunulması
Bu bölümde, GraphQL istek ve yanıtlarının HTTP protokolü üzerinden nasıl gönderilmesi ve
alınması gerektiğine ilişkin temel ilkeler özetlenmektedir. GraphQL spesifikasyonu belirli bir ta-
şıma protokolü dayatmasa da, pratikte yaygınlık ve altyapı desteği nedeniyle en sık kullanılan
protokol HTTP’dir. Buradaki öneriler, stateless nitelikteki query ve mutation işlemleri için ge-
çerlidir.
Genel Çerçeve ve Spesifikasyon
GraphQL dili, isteklerin nasıl taşınacağını değil, yalnızca sorgu yapısını ve yürütme semantiğini
tanımlar. Buna paralel olarak gelişmekte olan GraphQL-over-HTTP taslak spesifikasyonu, HTTP
üzerinden GraphQL API’lerinin nasıl sunulacağına dair ayrıntılı kurallar önermektedir. Bu taslak
zorunlu olmamakla birlikte, birlikte çalışabilirlik (interoperability) amacıyla pek çok sunucu ve
istemci kütüphanesi bu standartlara uyum sağlamaya yönelmektedir.
API Endpoint Tasarımı
REST yaklaşımı, URL’ler ile tanımlanan resource kavramına dayanırken, GraphQL’in temel mo-
deli bir varlık grafıdır. Bu nedenle GraphQL’de varlıklar URL ile değil, şema ve sorgu üzerinden
tanımlanır. Tipik olarak:
• Bir GraphQL servisi, tek bir HTTP endpoint’i üzerinden sunulur.
• Bu endpoint çoğunlukla /graphql ile biter ve ilgili servise ait tüm GraphQL istekleri bu
uç noktaya yönlendirilir.
Kimlik Doğrulama ve Yetkilendirme
Modern web çatıları, isteklerin bir dizi middleware veya filtre katmanından geçirilerek işlendiği
bir pipeline modeli kullanır. Bu bağlamda:
• GraphQL endpoint’i, kimlik doğrulama (authentication) katmanlarından sonra konumlan-
dırılmalıdır. Böylece diğer HTTP handler’larında olduğu gibi oturum ve kullanıcı bilgilerine
erişim sağlanır.
• Yetkilendirme (authorization) kararları ise, GraphQL yürütmesi sırasında, özellikle resol-
ver’lar içerisindeki iş mantığı katmanında verilmelidir. Bu sayede alan bazında yetkilen-
dirme yapılabilir ve kısmi veri ve hata içeren yanıtlar üretmek mümkün olur.

"data": ... , "errors": [ ... ], "extensions": ...
Burada:
• data, başarılı yürütülen sorgunun sonuç verilerini içerir.
• errors, doğrulama veya yürütme sırasında ortaya çıkan hata nesnelerinin listesini içerir.
• extensions, implementasyona bağlı olarak ek meta bilgilerin taşınabildiği isteğe bağlı
bir alandır.

• Hiç hata yoksa errors alanı bulunmamalıdır.

HTTP Status Kodları
Statü kodu seçimi, hem yanıtın içeriğine hem de medya tipine göre şu ilkelere dayanır:
• Yanıtta data anahtarı mevcut ve değeri null değilse, hata nesneleri bulunsa dahi HTTP
statü kodu 2xx olmalıdır. HTTP, “kısmi başarı” için ayrı bir kod tanımlamadığından bu
yaklaşım benimsenir.
• Doğrulama hataları gibi, GraphQL operasyonunun yürütülmesini engelleyen durumlarda,
sunucu genellikle 400 statü kodu döner.
• application/json ile yanıt veren eski sunucularda, geçerli GraphQL isteklerinin yürü-
tülememesi hâlinde 4xx veya 5xx kodları kullanılması mümkün olmakla birlikte, bu durum
belirsizlik yaratabileceği için genellikle 2xx + errors alanı tercih edilir.
• application/graphql-response+json medya tipi kullanıldığında, geçerli bir Grap-
hQL isteği yürütülememişse sunucunun 4xx veya 5xx statü kodu ile yanıt vermesi beklenir.

Özet
HTTP üzerinden GraphQL servisi sunarken önerilen başlıca ilkeler şöyle özetlenebilir:
• Kimlik doğrulama, GraphQL isteği doğrulanmadan önce HTTP pipeline’ında yapılmalı;
yetkilendirme ise GraphQL yürütmesi sırasında iş mantığı katmanında gerçekleştirilmelidir.
• GraphQL API’leri genellikle tek bir endpoint üzerinden sunulur ve bu endpoint çoğunlukla
/graphql ile biter.
• İstemci ve sunucu JSON formatını desteklemeli; istemci, yanıt medya tipini Accept he-
ader’ında özellikle application/graphql-response+json ile belirtmelidir.
• GraphQL istekleri varsayılan olarak POST metoduyla, Content-Type: application/json
header’ı eşliğinde gönderilmelidir; query operasyonları için isteğe bağlı olarak GET de des-
teklenebilir.
• Yürütülen GraphQL operasyonunun tam veya kısmi sonucu, JSON yanıt gövdesinde data
anahtarı altında; doğrulama ve yürütme sırasında oluşan hatalar ise errors anahtarı
altında sunulmalıdır.
• Geçerli bir GraphQL isteğine karşılık dönen yanıtta data alanı null değilse, HTTP statü
kodu 2xx olmalı; yürütülmeyen veya başarısız olan isteklerde ise kullanılacak statü kodu,
yanıtın medya tipine ve sunucu implementasyonuna göre 4xx/5xx ile farklılık gösterebilir.

GraphQL ile Dosya Yüklemenin Sınırları ve Önerilen Yaklaşım
Bu bölümde, dosya yükleme işlemlerinin neden GraphQL üzerinden gerçekleştirilmesinin uygun
olmadığı ve bunun yerine hangi yaklaşımın tercih edilmesi gerektiği özetlenmektedir.
GraphQL ile Dosya Yüklemenin Zorluğu
GraphQL temelde:
• Görece küçük boyutlu,
• JSON gibi metin tabanlı serileştirme formatlarıyla taşınan
istek ve yanıtlar için tasarlanmıştır. Buna karşılık, dosya yükleme işlemleri genellikle resim, PDF
veya video gibi binary verilerle çalışır.
JSON, ikili veriyi doğrudan taşıyamadığı için ek kodlama (örneğin base64) gerektirir. Bu yakla-
şım:
• Verimsizdir (mesaj boyutunu artırır),
• Büyük dosyalar için uygun değildir,
• Akış tabanlı (streamed) işleme imkânlarını kısıtlar.
HTTP tarafında ikili veri için yaygın çözüm multipart/form-data kullanımıdır; ancak bu
formatın GraphQL ile birleştirilmesi hem uygulama karmaşıklığını artırmakta hem de güvenlik
ve dayanıklılık açısından ek riskler getirmektedir.
Başlıca Riskler
Bellek Tükenmesi ve Değişken Tekrarı
GraphQL sorgularında aynı değişkenin birden fazla yerde kullanılmasına izin verilir. Eğer bu
değişken bir dosya yüklemesini temsil ediyorsa, alttaki akışın (stream):
• Birden fazla kez okunmaya çalışılması,
• Erken tüketilmesi veya yanlış yönetilmesi
sonucunda yanlış davranışlar veya bellek tükenmesi (memory exhaustion) oluşabilir.
Başarısız İsteklerde Akışların Açık Kalması
GraphQL yürütmesi, doğrulama ve çalıştırma olmak üzere iki fazda gerçekleşir. Eğer doğrulama
hataları veya yetkilendirme sorunları nedeniyle yürütme başlamadan istek başarısız olursa, ge-
len dosya akışları hiç tüketilmeden açık kalabilir. Sunucu bu akışları buffer’lıyor veya bellekte
tutuyorsa, bu durum kaynak sızıntılarına ve bellek kullanımının artmasına yol açar.

Saldırganlar çok büyük boyutlu dosyalar gönderebilir veya kullanılmayan değişken adları altında

Önerilen Yaklaşım: İmzalı URL (Signed URL) Kullanımı
En güvenli ve ölçeklenebilir çözüm, dosya yükleme işlemlerini GraphQL’in dışına taşımaktır.
Önerilen model şu adımlardan oluşur:
1. İstemci, bir GraphQL mutasyonu ile depolama sağlayıcısından (örneğin Amazon S3) dosya

2. İstemci, dosyayı bu imzalı URL’yi kullanarak doğrudan depolama servisine yükler (Grap-

3. Yükleme tamamlandıktan sonra:
• İkinci bir GraphQL mutasyonu ile ilgili dosyanın uygulama verisiyle ilişkilendirilmesi
sağlanır, veya
• Depolama tarafında otomatik tetiklenen bir süreç (örneğin Lambda fonksiyonu) bu

Bu yaklaşım:
• GraphQL sunucusunu doğrudan ikili veri trafiğinden izole eder,
• Sorumlulukları net biçimde ayırır (GraphQL: iş mantığı ve metadata; depolama servisi:
büyük dosya trafiği),
• Modern web mimarilerinde önerilen en iyi uygulamalarla uyumlu bir çözüm sunar.
Geçici yükleme alanının kısa ömürlü tutulması, saldırganların yalnızca imzalı URL alıp depo-
lama alanını doldurmasını önlemeye yardımcı olur. Dosya işlendikten sonra uygun kalıcı depoya
taşınmalı, kullanılmayan yüklemeler ise temizlenmelidir.

GraphQL’de Yetkilendirme (Authorization) ve İş Mantığı Kat-
manı
Bu bölümde, GraphQL tabanlı bir sistemde yetkilendirme (authorization) sorumluluğunun neden
doğrudan GraphQL katmanına değil, iş mantığı (business logic) katmanına devredilmesi gerektiği
ele alınmaktadır. Kimlik doğrulama (authentication) ile yetkilendirme (authorization) arasındaki
sorumluluk paylaşımı ve bu yaklaşımın pratik sonuçları tartışılmaktadır.
Kimlik Doğrulama ve Yetkilendirme Ayrımı
Bir GraphQL isteği sunucuya ulaştığında işlem hattı genel olarak şu şekilde yapılandırılmalıdır:
• Kimlik doğrulama (authentication): İstek, GraphQL yürütmesi başlamadan önce bir
middleware zincirinden geçer. Bu aşamada kullanıcının kimliği doğrulanır ve ilgili kullanıcı
bilgisi, GraphQL isteğine ait context nesnesine eklenir (örneğin context.user).
• Yetkilendirme (authorization): Kullanıcı doğrulandıktan sonra, kullanıcının belirli veri
alanlarına veya işlemlere erişim hakkının olup olmadığına karar verilmesi gerekir. Bu ka-
rarlar, GraphQL katmanına dağılmak yerine iş mantığı katmanında verilmelidir.
Bu ayrım sayesinde, doğrulama ve yetkilendirme sorumlulukları farklı aşamalara bölünerek hem
mimari netlik sağlanır hem de kuralların tek bir yerde yönetilmesi kolaylaşır.
• Aynı yetkilendirme kuralına ihtiyaç duyan tüm giriş noktalarında (farklı API’ler, farklı
servisler vb.) benzer kodların tekrar tekrar yazılması gerekir.
• Eğer bu kodlar zaman içinde senkronize tutulamazsa, aynı iş kuralı farklı yerlerde farklı
biçimlerde uygulanır ve kullanıcılar, hangi giriş noktasını kullandıklarına bağlı olarak farklı
sonuçlar görebilir.
Dolayısıyla, resolver içinde doğrudan yetkilendirme mantığı tutmak, tutarsızlıklara ve güvenlik
açıklarına zemin hazırlayabilir.

-- 79 of 116 --

Doğru Yaklaşım: Yetkilendirme Mantığının İş Katmanına Delegasyonu
Önerilen mimari, yetkilendirme kurallarının iş mantığı katmanında tek bir yerde tanımlanmasıdır.
GraphQL resolver’ları ise bu kuralları çağıran, nispeten ince geçiş noktaları olarak kalmalıdır.
İş mantığı katmanına mümkünse yalnızca bir token veya anahtar değil, tam olarak hydrate edilmiş
bir kullanıcı nesnesi aktarılması önerilir. Böylece:
• Kimlik doğrulama (token çözümleme) aşaması middleware katmanında,
• Yetkilendirme (bu kullanıcının neleri yapabileceğine karar verme) aşaması ise iş mantığı
katmanında gerçekleşir.
Tüm yetkilendirme mantığının iş mantığı katmanında tutulması genel olarak en sağlıklı yaklaşım
olsa da, şema seviyesinde bazı alanların hangi yetki kurallarına tabi olduğunu ifade etmek için
type system directive kullanımı düşünülebilir.
Özet
GraphQL ile yetkilendirme yapılırken aşağıdaki ilkeler benimsenmelidir:
• Yetkilendirme mantığı, GraphQL resolver’larının içine dağılmak yerine, iş mantığı katma-
nında tek bir kaynakta toplanmalıdır.
• Kimlik doğrulama, GraphQL yürütmesi başlamadan önce middleware aşamasında yapıl-
malı; GraphQL yürütmesi sırasında ise hangi alanlara erişileceğine ilişkin kararlar, iş man-
tığı katmanındaki kurallar temelinde verilmelidir.
• Şema üzerinde directive kullanımıyla, belirli alan ve tiplerin hangi genel yetki kurallarına
tabi olduğu deklaratif olarak belirtilebilir; ancak bu, yetkilendirme mantığının iş mantığı
katmanında tutulması ilkesini değiştirmez.

Liste Alanları ve Temel Kullanım

Offset Tabanlı ve Cursor Tabanlı Sayfalandırma
Sayfalandırma için farklı yaklaşımlar mümkündür:
• Offset tabanlı: friends(first: 2, offset: 2)
• Cursor tabanlı: friends(first: 2, after: $friendCursor)
Offset tabanlı yaklaşım, özellikle büyük veri kümelerinde performans ve güvenlik açısından zayıf-
lıklara sahiptir ve yeni kayıtlar eklendiğinde sayfa sınırları kayarak belirsizlik yaratabilir. Buna
karşılık, cursor tabanlı sayfalandırma:
• Daha stabil ve esnektir,
• Cursor’lar opak tutulduğunda (örneğin base64 ile kodlanmış) backend’in sayfalandırma
stratejisi zaman içinde değiştirilebilir,
• Offset tabanlı modeller de, uygun cursor tanımı ile bu yapı üzerinden uygulanabilir.
Sosyal Medya Akışlarında Sayfalandırma ve Yenileme Davranışı
Bu bölümde, sosyal medya uygulamalarında veri tabanının fiilen “dondurulmadığı”, ancak istem-
cinin (mobil veya web uygulamasının) sayfalandırma (pagination) ve yenileme (pull-to-refresh)
davranışlarıyla sanki sabit bir görünüm sunuyormuş gibi hareket ettiği özetlenmektedir. Özel-
likle cursor tabanlı sayfalandırma, kullanıcının “nerede kaldığı” bilgisini korurken, sistemin arka
planda akmaya devam etmesine imkân verir.
Aşağı Doğru Kaydırma: Eski Gönderilere Cursor Tabanlı Erişim
Kullanıcı, akışta aşağıya doğru kaydırdığında:
• Uygulama, arka uca genellikle first ve after parametreleri ile cursor tabanlı istekler
gönderir (örneğin GraphQL’de posts(first: 20, after: "CURSOR_100")).
• Her sayfa isteğinde, istemci bir önceki yanıtın endCursor değerini “nerede kaldığını” temsil
eden referans olarak kullanır.
• Veri tabanına yeni kayıtlar eklense dahi, istemci “şu kayıttan sonra gelenler” mantığıyla
ilerlediği için, aşağı doğru sayfalar tutarlı kalır; kullanıcı aynı gönderiyi tekrar görmez ve
kayıtları atlamaz.
Bu sayede, veri tabanı akışkan ve güncel kalırken, kullanıcının aşağıya doğru ilerlediği “kişisel
zaman çizgisi” cursor’lar sayesinde stabil bir şekilde korunur.

Yukarı Doğru Çekerek Yenileme: Pull-to-Refresh Davranışı
Kullanıcı akışın en üstündeyken “pull-to-refresh” (yukarı çekip bırakma) jestini kullandığında,
uygulama genellikle aşağıdaki iki yaklaşımdan birini benimser:
Tam Yeniden Yükleme
• İstemci, akışın en güncel hâlini almak için baştan yeni bir istek gönderir (örneğin posts(first: 20)).
• Eski liste, gelen yeni sonuçla tamamen değiştirilir.
• Sıralama veya puanlama mantığı değişmişse, kullanıcı daha önce gördüğü bazı gönderilerin
yer değiştirdiğini veya artık görünmediğini fark edebilir.
Bu davranış, kullanıcıya “liste sil baştan yenilenmiş” hissi verir; ancak veri tabanının dondurul-
ması söz konusu değildir, yalnızca yeni bir sorgu çalıştırılmıştır.
Birleştirerek Güncelleme
• Uygulama, en üstteki gönderinin cursor veya kimliğini (top cursor ) tutar.
• Yenileme jesti sonrasında, bu cursor’dan daha yeni olan gönderileri ister (örneğin “bu nok-
tadan daha yeni olanları getir”).
• Gelen yeni gönderiler mevcut listenin en üstüne eklenir; alt taraftaki eski gönderiler aynı
sırada kalmaya devam eder.
Bu yaklaşımda, ekran baştan boyanmak yerine, kullanıcı sadece “tepeye yeni satırlar eklendiğini”
görür.
Gerçekte Ne “Dondurulmuş” Gibi Davranır?
Tek “dondurulan” şey veri tabanının kendisi değildir; dondurulan, istemcinin nerede kaldığına
dair referansıdır. Kullanıcı aşağı kaydıkça:
• Uygulama, örneğin CURSOR_100 gibi bir değeri, “100 numaralı gönderiye kadar gösterildi”
anlamında referans olarak saklar.
• Sonraki isteklerde, after: "CURSOR_100" diyerek bu noktadan sonra gelenleri talep
eder.
Veri tabanına yeni kayıtlar eklense bile, istemci “bu cursor’dan sonraki dünya” ile ilgilendiği
için, kullanıcının aşağıya doğru sayfaladığı bölge tutarlı görünür. Bu sayede, sistem hem sürekli
güncel kalır, hem de kullanıcılar sayfalandırma sırasında tekrarlayan veya atlanan kayıtlarla
karşılaşmadan akıcı bir deneyim yaşayabilir.

GraphQL Şema Tasarımı: Versiyonlama ve Nullability
Bu bölümde, GraphQL şema tasarımında iki temel kavram özetlenmektedir: versiyonlama yak-
laşımı ve nullability (alanların boş/“null” olabilme durumu).
Versiyonlama Yerine Şemanın Evrimleştirilmesi
Geleneksel API tasarımlarında (özellikle REST):
• Endpoint’in döndürdüğü veri üzerinde istemci tarafında sınırlı kontrol vardır.
• Küçük bir değişiklik dahi mevcut istemcileri bozabileceğinden breaking change olarak kabul
edilir.
• Bu nedenle, değişiklikler genellikle /v1, /v2 gibi yeni API versiyonları açılarak yönetilir.
Bu yaklaşım, şu ikilemi doğurur:
• Sık sık, küçük versiyonlar yayımlamak: Esnek ama takip etmesi zor ve karmaşık bir API
yüzeyi.
• Seyrek ve büyük versiyonlar yayımlamak: Daha az karmaşa ama esnekliği düşük, yavaş
evrilen bir API.
GraphQL’de ise:
• İstemci yalnızca açıkça talep ettiği alanları alır.
• Sunucu, şemaya yeni tipler veya mevcut tiplere yeni alanlar ekleyebilir; eski istemciler bu
alanları istemediği sürece davranışları değişmez.
• Bu sayede yeni kabiliyetler genellikle kırıcı olmayan (non-breaking) eklemelerle sağlanır.
Sonuç olarak yaygın pratik:
• Kırıcı değişiklik yapmaktan mümkün olduğunca kaçınmak,
• Şemayı sürekli, küçük eklemelerle evrimleştirmek,
• Ayrı versiyon numaraları sunmak yerine, versionless (tek sürümlü) bir GraphQL API sun-
maktır.

Nullability: Varsayılan Olarak Nullable Alanlar
Pek çok tip sisteminde, null desteği şu şekilde modellenir:
• Temel tip (T) ve bunun nullable versiyonu (T?) ayrıdır.
• Varsayılan tip genellikle “null içermez” kabul edilir; null desteği ayrıca belirtilir.
GraphQL’de yaklaşım tersine çevrilmiştir:
• Her alan, varsayılan olarak nullable’dır.
• Bunun nedeni, dağıtık bir sistemde alan bazında çok sayıda hata kaynağı bulunabilmesidir:
– Veritabanı erişimi başarısız olabilir,
– Arka plandaki bir servis cevap vermeyebilir,
– Çalışma zamanı hatası (exception) oluşabilir,
– Alan bazında farklı yetkilendirme kuralları devreye girebilir.
Bu varsayılan tasarım sayesinde:
• Bir alanın alınması sırasında hata oluştuğunda, tüm isteği başarısız saymak yerine sadece
ilgili alan null döndürülebilir.
• İstek genel olarak başarılı kalırken, hata detayları errors alanında raporlanabilir.
GraphQL ayrıca non-null tipler de sunar:
• Örneğin String! veya Int!.
• Bu tipler istemciye şu garantiyi verir:
“Bu alan hiçbir zaman null olmayacak; eğer hata oluşursa, bunun bir üstindeki
parent alan null olarak işaretlenecek.”
Şema tasarlarken şu soruyu sormak önemlidir:
• Bu alan için hata durumunda null dönmek, iş mantığı açısından makul mü?
– Çoğu alan için cevap evet olacaktır ve varsayılan nullable davranış yeterlidir.
– Bazı kritik alanlar için ise null kabul edilemez; bu durumda non-null (!) kullanılarak
daha güçlü bir garanti verilir.
Özetle GraphQL, hem API yüzeyini versiyonsuz biçimde evrimleştirmeye uygun bir model sunar,
hem de varsayılan nullable alanlar üzerinden isteklerin tamamını patlatmadan, alan bazında
hataları tolere edebilen esnek bir hata modeli sağlar.

Caching ve ID
GraphQL’de caching yapabilmek için, her nesneye tutarlı bir id vermemiz yeterlidir. Bu id
değeri, istemcinin o nesneyi cache’te tanıması için kullanılan anahtar (cache key) olur.
Yani özetle: “Cache’lemek istiyorsam, her nesne için değişmeyen bir id üretirim ve bunu dışarı
açarım.”
Performance
GraphQL istekleri ilk bakışta zor cache’lenebilir görünse de, doğru tekniklerle REST kadar (hatta
bazı durumlarda daha da) verimli hale getirilebilir. Aşağıda, GraphQL API performansını iyileş-
tirmek için önerilen temel yaklaşımlar özetlenmektedir.
İstemci Tarafı Cache
GraphQL istemcileri (örn. Apollo, Relay) genellikle yerleşik cache mekanizmalarına sahiptir. Bu
cache sayesinde:
• Aynı nesne birden fazla sorguda geldiğinde, id ve __typename gibi alanlara bakarak tek
bir cache kaydı altında birleştirilebilir.
• Daha önce alınmış veriler tekrar sunucudan çekilmeden, doğrudan cache üzerinden okuna-
bilir.
• UI, ağ gecikmesine daha az bağımlı ve daha tutarlı bir şekilde güncellenebilir.
Sorgular için GET Kullanımı
GraphQL sunucuları varsayılan olarak POST isteklerini destekler; ancak sadece query operasyon-
ları için GET methodu da kullanılabilir.
• GET istekleri HTTP ve CDN katmanında cache’lenebilir olduğundan, performans açısından
avantaj sağlar.
• Sorgu, URL içindeki query parametresiyle gönderilebilir; ancak URL uzunluk sınırları
nedeniyle karmaşık sorgular için bu pratik olmayabilir.
Bu sorunu aşmak için persisted queries veya hashed queries kullanılabilir:
• İstemci, tam sorgu yerine sorgunun hash değerini gönderir.
• Sunucu, bu hash ile önceden kayıtlı sorguyu bulur, doğrular ve çalıştırır.
• Böylece:

-- 91 of 116 --

– Daha kısa URL’ler elde edilir,
– HTTP/CDN cache kullanımı kolaylaşır,
– Ağ üzerinden gönderilen veri miktarı azalır.
N+1 Problemi
GraphQL, her field için ayrı resolver yazmaya imkân tanıyarak sunucu kodunu temiz ve modüler
hale getirir. Ancak bu durum, ek bir önlem alınmadığında N+1 problemine yol açabilir.
• Örnek: Bir hero nesnesi, onun friends listesi ve her arkadaş için starships alanı
sorgulandığında:
– 1 istek: kahramanı (hero) almak,
– 1 istek: arkadaş listesini almak,
– N istek: her arkadaş için starship listesini almak,
şeklinde toplam N+2 arka uç isteği oluşabilir.
Bu problem genellikle batching ve caching teknikleriyle çözülür:
• Facebook’un DataLoader kütüphanesi gibi araçlarla, aynı tipteki birden çok istek kısa
bir zaman aralığında toplanır.
• Bu istekler tek bir WHERE id IN (...) sorgusu gibi toplu bir çağrıya dönüştürülerek
veritabanına veya microservice’e gönderilir.
• Böylece N ayrı çağrı yerine 1 toplu çağrı yapılarak gecikme ve yük azaltılır.
Demand Control (İstek Karmaşıklığını Sınırlama)
Bazı GraphQL şemaları, istemcilerin istemeden ya da kötü niyetle aşırı karmaşık ve maliyetli
sorgular gönderebilmesine izin verebilir. Bu tür durumlar, arka uç kaynakları üzerinde gereksiz
yük oluşturur.
Bunu önlemek için:
• Pagination: Büyük listeler daima sayfalara bölünmeli, sonsuz uzunlukta listeler tek istekte
döndürülmemelidir.
• Derinlik sınırlaması: Bir sorgunun ağaç derinliği (nesting seviyesi) belirli bir maksimum
değerin üzerinde kabul edilmemelidir.
• Genişlik / karmaşıklık analizi: Bir sorgunun toplam alan sayısı veya “karmaşıklık puanı”
hesaplanarak, belirli bir eşiğin üzerindeki istekler reddedilebilir.
• Oran sınırlama (rate limiting): Belirli bir kullanıcı veya API anahtarı için isteklere
kota ve hız limiti getirilebilir.

Security
GraphQL, diğer API türleri gibi kötü niyetli isteklerden ve aşırı yükten korunması gereken bir
katmandır. Özellikle karmaşık ve ağır sorgularla yapılabilecek denial of service tarzı saldırılara
karşı ek önlemler almak gerekir. Aşağıda GraphQL API’leri için temel güvenlik yaklaşımları
özetlenmektedir.
Transport Katmanı Güvenliği
GraphQL spesifikasyonu belirli bir taşıma protokolü zorunlu kılmaz, ancak çoğu uygulama sorgu
ve mutasyonlar için HTTP, subscription için WebSocket veya Server-Sent Events (SSE) kullanır.
Seçilen protokol ne olursa olsun:
• HTTP kullanılıyorsa HTTPS ile trafiğin şifrelenmesi,
• İstekler için makul timeout değerleri belirlenmesi,
• HTTP cache kullanılıyorsa hassas verilerin public yerine private veya hiç cache’lenmemesi
gibi genel API güvenlik uygulamaları GraphQL için de birebir geçerlidir.
Demand Control: İstek Yükünü Kontrol Etmek
Amaç, tek bir GraphQL sorgusunun sunucuyu aşırı yüklemesini veya bilerek istismar edilmesini
önlemektir. Bunun için birkaç tamamlayıcı teknik kullanılabilir.
Trusted Documents (Allowlist Yaklaşımı)
Eğer API yalnızca first-party istemciler tarafından kullanılıyorsa:
• Geliştirme ortamında sorguları geliştiriciler yazar.
• Build aşamasında bu sorgular sunucuya gönderilir ve bir allowlist (güvenilir sorgular listesi)
içinde hash veya ID ile saklanır.
• Üretim ortamında istemci, tam sorgu metni yerine sadece bu document ID veya hash’i
gönderir.
• Sunucu, sadece tanıdığı (allowlist’te olan) sorguları çalıştırır.
Böylece, istemcinin kafasına göre yeni GraphQL dokümanı göndermesi engellenmiş olur. Bu
teknik, üçüncü parti istemcilere açık public API’ler için uygun değildir; ancak kapalı ekosistemler
için güçlü bir savunma hattı sağlar.

GraphQL’in gücü, ilişkisel yapıları derin bir şekilde ifade edebilmesidir; ancak bu, kötü yazılmış
veya kötü niyetli sorgularla ağaç derinliğinin kontrolden çıkmasına sebep olabilir:
• Örneğin friends { friends { friends { ... } } } şeklinde recursive veya aşırı
iç içe geçmiş selection set’ler CPU ve bellek kullanımını arttırabilir.
Bu nedenle:
• Her sorgu için maksimum bir derinlik (max depth) belirlenmesi,
• Bu sınırı aşan dokümanların, daha validation aşamasında hata ile geri çevrilmesi,
• Özellikle iç içe listelerin derinliği için daha düşük bir ek limit uygulanması

Breadth ve Batch Limiting (Genişlik ve Toplu İstek Sınırı)
Sorgu ağaç derinliği düşük olsa bile, yatayda çok sayıda alan veya alias kullanılabilir:
• Örneğin aynı kök alan için friends1, friends2, . . . , friends100 şeklinde alias’larla
tekrar tekrar istek yapılması,
• Tek HTTP isteği içinde onlarca GraphQL sorgusunun batch hâlinde gönderilmesi gibi du-
rumlar, arka uç veri kaynaklarında ciddi yük oluşturabilir.
Bu yüzden çoğu GraphQL sunucusunda:
• Tek dokümandaki top-level field sayısı,
• Kullanılabilen field alias sayısı,
• Tek batch içinde izin verilen maksimum sorgu (operation) sayısı
için sınırlar (limits) konfigüre edilebilir.

• String tipindeki bir alan zararlı HTML veya script içerebilir.

• Kullanıcı girişlerinin, resolver içinde veya iş katmanında sanitize edilmesi,

GraphQL Federation
Genel Bakış
GraphQL federation, büyük bir GraphQL API’yi tek bir monolit olarak tutmak yerine, birden
fazla bağımsız GraphQL servisine (subgraph) bölüp bunları bir gateway üzerinden birleştiren
mimari bir yaklaşımdır.
Klasik microservice yaklaşımına benzer şekilde:
• Her domain için ayrı bir servis ve ayrı bir şema vardır,
• Bu servisler bağımsız geliştirilip deploy edilebilir,
• İstemci tarafında ise hâlâ tek bir birleşik GraphQL API görünür.
Federation Nedir?
Federation, dağıtık sistemleri organize etme biçimidir. Temel fikir:
• Her bileşen (servis / ekip) kendi alanından sorumludur,
• Kendi kararlarını verebilir, kendi hızında gelişir,
• Ortak ihtiyaçlar için bunların üzerinde bir koordinasyon katmanı bulunur.
Devlet benzetmesiyle:
• Eyaletler: Kendi iç kuralları olan bağımsız alanlar,
• Federal yapı: Eyaletleri bir arada tutan, dışarıya tek ses olarak çıkan katman.
GraphQL federation, bu fikri GraphQL API’lerine uygular.
Federated GraphQL
Federated bir GraphQL mimarisinde:
• Subgraph:
– Kendi GraphQL şemasına ve resolver’larına sahip bağımsız servis.
– Örneğin: Users, Orders, Products gibi domain bazlı servisler.
• Gateway:
– İstemci ile subgraph’lar arasındaki tek giriş noktasıdır.

– Tüm subgraph şemalarını birleştirerek tek bir unified schema sunar.
– Gelen sorguyu alt parçalara bölerek ilgili servislere yönlendirir ve sonuçları tek bir
yanıt hâlinde birleştirir.
Örnek bir unified şema:
type User {

name: String!
email: String

}
type Order {

• Siparişteki ürünler için Products subgraph’a gidilir.

• Schema composition aşaması, subgraph şemalarını birleştirirken:
– Tip çakışmalarını,
– Uyuşmayan alan tanımlarını,
– Kırık referansları
yakalayabilir.
• Böylece bir ekibin yaptığı hatalı şema değişikliği, diğer ekiplerin alanlarını bozmadan tespit
edilebilir.
Ölçeklenebilirlik ve Performans
• Her subgraph bağımsız ölçeklenebilir:
– Çok okunan Product servisi daha fazla replika ile,
– Daha az kullanılan Order servisi daha az kaynakla çalıştırılabilir.
• Gateway, cache ve diğer optimizasyonlarla genel performansı iyileştirebilir.

• Domain-Driven Design (DDD) prensipleriyle çalışan,

Bu nedenle sık önerilen yaklaşım:
• Önce monolitik bir GraphQL API ile başlamak,

mimarisine geçmektir.

Common HTTP Errors and Debugging in GraphQL over HTTP
Genel Bakış
GraphQL’i HTTP üzerinden kullanırken, özellikle geliştirme aşamasında çeşitli HTTP hata kod-
larıyla karşılaşmak yaygındır. Bu bölümde:
• Sık görülen HTTP ve GraphQL hata kodlarının ne anlama geldiği,
• Yaygın nedenleri,
• Nasıl debug edilebilecekleri,
• Ve GraphQL yanıtları için önerilen medya tipi (application/graphql-response+json)
özetlenmektedir.
400 Bad Request (İstek Hatalı / Parse Hatası)
Anlamı. Sunucu, isteği parse edememiştir. Ya JSON body geçersizdir ya da GraphQL sorgu
metni sentaks olarak hatalıdır.
Yaygın Nedenler.
• JSON body sentaks hataları (eksik süslü parantez, tırnak hatası vb.),
• "query" alanını içeren bir JSON yerine düz string göndermek (örneğin sadece "{ hero
{ name } }" gödermek),
• Uygulamanın desteklemediği bir Content-Type kullanmak (örneğin text/plain veya
application/graphql).
Debug Önerileri.
• JSON body’yi bir formatter veya linter ile doğrula,
• POST isteğiyle birlikte Content-Type: application/json gönderildiğinden emin ol,
• GraphQL sorgusunu bir IDE veya linter ile sentaks açısından kontrol et.

-- 104 of 116 --

405 Method Not Allowed (Yanlış HTTP Metodu)
Anlamı. İstek, bu endpoint için desteklenmeyen bir HTTP methodu ile gönderilmiştir. Çoğu
GraphQL sunucusu:
• Mutations için POST,
• Queries için POST (opsiyonel olarak GET)
bekler.
Yaygın Nedenler.
• PUT veya DELETE ile GraphQL isteği göndermek,
• Mutation’ı GET ile göndermek,
• Sunucu yalnızca POST desteklerken GET kullanmak.
Debug Önerileri.
• HTTP methodunu kontrol et:
– Mutations için her zaman POST,
– Queries için sunucunun GET desteği olup olmadığını kontrol et.
• Kullandığın sunucunun veya kütüphanenin dokümantasyonuna bakarak hangi methodlara
izin verildiğini doğrula.
500 Internal Server Error (Sunucu Hatası)
Anlamı. İstek sunucuya ulaşmış, ancak backend içinde beklenmeyen bir hata oluşmuştur.
Yaygın Nedenler.
• Resolver içinde yakalanmamış exception,
• Sunucu açılışında şema veya konfigürasyon hataları,
• Eksik veya yanlış yapılandırılmış middleware (auth, body parser vb.).
Debug Önerileri.
• Sunucu loglarını ve stack trace’leri incele,
• Resolver fonksiyonlarında hata yakalama ve logging stratejileri uygula,
• Uygulamanın başlatma aşamasında şema, middleware zinciri ve diğer konfigürasyonları
doğrula.

200 OK + errors (HTTP Başarılı, GraphQL Hatalı)
Anlamı. HTTP isteği başarılıdır (200), ancak GraphQL operasyonu sırasında hata oluşmuştur.
Yanıt tipik olarak şu şekildedir:
{
"data": { ... } veya null,
"errors": [ ... ]
}
İki Temel Durum.
• data var, errors var:
– Sorgu şemaya göre geçerlidir,
– Ancak çalışma sırasında hata oluşmuştur (örneğin hatalı input, yetki reddi, resolver
bug’ı).
• Sadece errors var, data yok:
– Sorgu validation aşamasında başarısız olmuştur (örneğin var olmayan alanı sorgulama
gibi).
Yaygın Validation Hataları.
• Şemada tanımlı olmayan bir alanı sorgulamak:
"Cannot query field \"foo\" on type \"Query\"."
• Yanlış argüman tipi veya eksik zorunlu argüman,
• Non-leaf tip için selection set eksik/yanlış,
• Aynı dokümanda birden fazla operasyon varken operationName belirtilmemesi.
Debug Önerileri.
• Yanıttaki errors dizisini dikkatle incele,
• Eğer hata, field veya tip isimleriyle ilgiliyse:
– Şemayı introspection veya IDE üzerinden gözden geçir,
– Field, argüman ve tip isimlerini şemaya uygun hâle getir.

Diğer HTTP Kodları (Implementation-Specific)
Bazı GraphQL sunucu implementasyonları, spesifikasyonun açıkça önermediği ek HTTP kodları
kullanabilir:
• 415 Unsupported Media Type:
– İstek, sunucunun anlamadığı bir Content-Type ile gönderilmiştir (örneğin text/plain).
• 422 Unprocessable Entity:
– Bazı implementasyonlar, GraphQL validation hataları için 200 + errors yerine 422
dönebilir.
Bu davranışlar, GraphQL over HTTP taslak spesifikasyonunda standart olarak önerilmez; kul-
lanılan kütüphaneye veya sunucuya göre değişebilir.

-- 107 of 116 --

GraphQL Yanıt Formatı ve Medya Tipi
Geleneksel olarak GraphQL sunucuları yanıtları:
• Content-Type: application/json
ile döndürmüştür. GraphQL over HTTP taslak spesifikasyonu ise, daha spesifik bir medya tipini
önermektedir:
• application/graphql-response+json
Neden Önemli?
• Bu medya tipi, payload’ın GraphQL response formatına uygun olduğunu açıkça belirtir,
• 2xx olmayan durum kodlarında bile, eğer application/graphql-response+json ise,
iç yapısının data/errors/extensions şeklinde GraphQL standardına uyduğu varsayı-
labilir,
• Başka proxy veya gateway’ler JSON dönebilir, ancak application/graphql-response+json
döndürmezler; bu da yanıtın gerçekten GraphQL katmanından geldiğini ayırt etmeyi ko-
laylaştırır.
İstemci Tarafında Önerilen Kullanım.
• Accept header’ında hem yeni hem eski tipi talep edebilirsin:
Accept: application/graphql-response+json, application/json;q=0.9
• Böylece:
– Öncelikle application/graphql-response+json tercih edilir,
– Sunucu bunu desteklemiyorsa application/json ile devam edilir.
