Bu doküman, React Native ile mobil uygulama geliştirirken sık sık karşımıza
çıkan ancak çoğu zaman dağınık kaynaklar hâlinde bulunan temel kavramları
tek bir yerde toparlamayı amaçlamaktadır. Metro paketleyicisinin rolünden

performans ve build sürelerini iyileştirme tekniklerine kadar uzanan bir yel-

Odak noktası, React Native geliştirme deneyimini soyut kavramlardan somut
kararlara indirgemektir: Hangi durumda hangi storage çözümünü kullanmalı,
hangi liste ayarları performansı etkiler, ağır işleri JS thread’ine yüklemeden
animasyonları nasıl akıcı tutabiliriz, native build süresini nasıl kısaltabiliriz
gibi sorulara kısa ve hatırlaması kolay cevaplar vermeye çalışır. Bu yönüyle
metin, hem yeni başlayanların kavram haritası oluşturmasına hem de deney-
imli geliştiricilerin günlük işlerinde hızlıca başvurabilecekleri bir referans olarak
tasarlanmıştır.

.native.js ve C oklu Platform Monorepo Mantığı
Problem
Ele alınan tipik senaryo şu biçimdedir:
• Elde tek bir modül bulunmaktadır: Container.
• Bu modülün:
– React Native (Android ve iOS),
– Web (React DOM),
– ve gerekirse Node ortamında
yeniden kullanılabilmesi hedeflenmektedir.
Bununla birlikte, pratikte şu durumla karşılaşılır:
• Android ve iOS, ortak bir React Native arayüzü mantığını paylaşırken,
• Web tarafında, çoğu zaman tamamen farklı bir implementasyona ihtiyaç
duyulur (örneğin <View> yerine <div> kullanımı, farklı CSS yapısı vb.).
Tam bu noktada .native.js uzantısı devreye girer:
• Native (iOS ve Android) platformlar için ortak bir dosya: Container.native.js,
• Web platformu için ise ayrı bir dosya: Container.js tanımlanır.
Böylece, çağıran kod tarafında import ifadesi her ortamda aynı kalır:
import Container from ’./Container’;
Ancak çalışılan platforma bağlı olarak, ilgili bundler arka planda farklı fizik-
sel dosyayı çözümleyip projeye dahil eder.

Metro (React Native Bundler’ı) Dosyaları Nasıl Seçer?
React Native tarafında (Metro) dosya çözüm sırası kabaca şu şekildedir:
Android:
1. Container.android.js (varsa),
2. Container.native.js (varsa),
3. Container.js.
iOS:
1. Container.ios.js,
2. Container.native.js,
3. Container.js.
Dolayısıyla:
• *.ios.js / *.android.js → platforma özgü dosyalar,
• *.native.js → her iki native platform için ortak fallback,
• *.js → en genel fallback.
Bundan şu sonuç çıkar:
• Yalnızca native için ortak bir implementasyon istiyorsanız, Container.native.js
yazmanız yeterlidir.
• Daha sonra Android’e özel bir davranış eklemek isterseniz, Container.android.js
ekleyebilirsiniz; Android bu dosyayı kullanırken, iOS tarafı hâlâ Container.native.js
üzerinden devam eder.

-- 3 of 41 --

Web Bundler (webpack, Vite vb.) Bu Dosyalara Nasıl
Bakıyor?
Web tarafında tipik bir resolve.extensions ayarı şu şekilde olabilir:
resolve: {
extensions: [’.web.js’, ’.js’, ’.ts’, ’.tsx’],
}
Bu durumda web bundler:
1. Container.web.js varsa onu seçer,
2. Yoksa Container.js dosyasını seçer,
3. Container.native.js dosyasını hiç görmez; çünkü uzantı listesinde
.native.js yoktur.
Bu nedenle dokümantasyonda sıkça kullanılan “bundler’ını .native.js dosyalarını
ignore edecek şekilde ayarla” ifadesinin pratiği aslında şudur:
• resolve.extensions listesine .native.js eklemezsiniz.
• Böylece web build’i .native.js dosyalarından haberdar bile olmaz; prod
bundle gereksiz React Native kodlarıyla şişmez.
Özetle:
• Native dünyası: .native.js ile ilgilenir.
• Web dünyası: .js (veya .web.js) ile ilgilenir, .native.js yokmuş gibi
davranır.
Örnek Klasör Yapısı
Aşağıda, ortak bir kütüphane için örnek klasör yapısı verilmiştir:
shared/
ui/
Container.js # Web implementasyonu
Container.native.js # RN (Android + iOS) implementasyonu
utils/
formatDate.ts # Her yerde aynı kullanılan pure logic

Container.native.js (React Native):
// shared/ui/Container.native.js
import React from ’react’;
import { View, StyleSheet } from ’react-native’;
export default function Container({ children }) {
return <View style={styles.container}>{children}</View>;
}
const styles = StyleSheet.create({
container: {
padding: 16,
backgroundColor: ’#111’,
},
});
Container.js (Web):
// shared/ui/Container.js
import React from ’react’;
export default function Container({ children }) {
return (
<div
style={{
padding: 16,
backgroundColor: ’#111’,
maxWidth: 600,
margin: ’0 auto’,
}}
>
{children}
</div>
);
}
Her iki tarafta da import ifadesi aynıdır:
import Container from ’shared/ui/Container’;
• React Native uygulaması → Container.native.js,
• Web uygulaması → Container.js.
Böylece çağıran taraf platformdan tamamen habersiz kalır: “Ben sadece
Container istiyorum; altında hangi platformda ne olduğunu bilmek zorunda
değilim.” Bu yaklaşım, özellikle tasarım sistemi ve component library tipi
yapılarda oldukça temiz bir desendir.

Node Tarafı Nerede Devreye Girer?
Node ortamı, varsayılan olarak .native.js uzantısını tanımaz. Zaten Node
tarafında çoğunlukla UI bileşenlerini değil, iş mantığı (pure logic) modüllerini
kullanırsınız.
Pratik bir pattern şöyle kurulabilir:
core/
domain/
price.ts # pure logic, Node + Web + RN ortak
ui/
Container.js # Web UI
Container.native.js# RN UI
• Node tarafı, örneğin
import { formatPrice } from ’core/domain/price’;
şeklinde yalnızca domain/logic modüllerini kullanır; UI veya .native dosyalarıyla
ilgilenmez.
• React Native ve Web tarafı hem domain hem ui katmanlarını kullanabilir.
Dolayısıyla .native.js, esas olarak UI katmanı için anlamlıdır; Node
açısından önemli olan, iş mantığı dosyalarının platformdan bağımsız ve “pure”
tutulmasıdır.

Neden Platform.select Yerine .native.js?
Her iki yaklaşım da platform farklarını yönetebilir; ancak kullanım amaçları
farklıdır.
Platform.select / Platform.OS ile: Tüm mantığı tek dosyada tutarsınız:
const styles = {
container: {
backgroundColor: Platform.OS === ’ios’ ? ’red’ : ’green’,
},
};
Bu, küçük farklar için uygundur; ancak:
• Dosya büyüdükçe if (Platform.OS === ...) blokları çoğalarak kodu
karmaşıklaştırabilir.
• Web ile tamamen farklı bir UI gerektiğinde, tek dosya içinde her şeyi
yönetmek zor ve “kirli” bir hale gelir.
• Web bundler, React Native import’larını da görüp tree-shaking ve bundle
boyutu açısından sorunlara yol açabilir.
.native.js ile:
• Web kodu ile Native kodu dosya bazında tamamen ayrıştırılır.
• Native uygulama:
– Metro, yalnızca React Native import’larını ve .native.js dosyalarını
görür.
• Web uygulama:
– Web bundler, yalnızca DOM tabanlı kodu (.js veya .web.js) görür;
RN kodlarından haberdar olmaz.
Özetle:
• Küçük ve yerel farklar için → Platform.select,
• Tamamen farklı implementasyon / farklı UI durumlarında → .ios/.android/.native
dosyaları tercih edilmelidir.

“Web Bundler’ını .native.js’i Ignore Edecek Sekilde Ayarla”
Ne Demek?
Pratik bir webpack örneği:
// webpack.config.js
module.exports = {
// ...
resolve: {
extensions: [’.web.js’, ’.js’, ’.jsx’, ’.ts’, ’.tsx’],
},
};
Burada dikkat edilmesi gereken nokta:
• Uzantılar arasında .native.js yoktur.
Dolayısıyla webpack, import Container from ’./Container’ ifadesini gördüğünde:
1. Önce Container.web.js arar,
2. Yoksa Container.js arar,
3. Container.native.js dosyasını tamamen yok sayar.
Vite veya Rollup gibi diğer bundler’larda da benzer şekilde resolve.extensions
veya resolve ayarları üzerinden aynı mantık uygulanır.
Bunun faydaları:
• Web bundle’a, React Native’e özgü component ve kodlar dahil edilmez.
• Bundle boyutu küçülür; kullanılmayan RN kodları web tarafına taşınmamış
olur.

Metro ve metro.config.js
Metro, React Native projelerinde kullanılan JavaScript paketleyicisidir (bundler).
Uygulamanın JavaScript kodlarını ve varlıklarını (örneğin resimler, fontlar vb.)
bir araya getirerek cihaza iletir. Web dünyasındaki Webpack veya Vite ne ise,
React Native tarafında da aynı görevi Metro üstlenir. Proje kök dizininde yer
alan metro.config.js dosyası, Metro’nun davranışını özelleştirmek için kul-
lanılır.

Gesture Responder System
Gesture Responder System, React Native uygulamalarında dokunma ve jest
(gesture) yaşam döngüsünü yöneten düşük seviye mekanizmadır. Tek veya çoklu
dokunuşlar sırasında:
• Kaydırma (scroll),
• Sürükleme (drag / pan),
• Basma (tap)
gibi etkileşim türlerini ayırt ederek, hangi bileşenin ilgili dokunuşa responder
(dokunuşun sahibi) olacağına karar verilmesini sağlar. Bu karar, bileşen hiy-
erarşisini doğrudan bilmeden, bileşenlerin kendi aralarında protokol üzerinden
“anlaşması” ile verilir.
Yüksek Seviye Kullanım: Touchable* ve Pressable
Düşük seviye responder API’leri karmaşık olduğundan, React Native günlük
kullanım için aşağıdaki yüksek seviye soyutlamaları sunar:
• TouchableOpacity, TouchableHighlight,
• Pressable vb.
Bu bileşenler, Gesture Responder System üzerine inşa edilmiş “dokunulabilir”
(tappable) arayüz elemanlarıdır. Web tarafında buton/link kullanılan yerlerde,
React Native projelerinde genellikle bu bileşenler tercih edilir. Böylece:
• Görsel geri bildirim (highlight, opacity değişimi),
• ˙Işlemi iptal edebilme (parmağı kaydırarak vazgeçme)
gibi davranışlar tutarlı ve otomatik şekilde sağlanır.
˙Iyi Kullanıcı Deneyimi ˙Için Temel ˙Ilkeler
Gesture Responder System üzerine kurulu bileşenler tasarlanırken özellikle iki
ilkeye dikkat edilmelidir:
1. Geri bildirim (feedback): Kullanıcı dokunduğunda hangi elemanın
tepki verdiği ve parmağını çektiğinde ne olacağı, renk değişimi, ölçek
(scale) veya benzeri efektlerle açıkça gösterilmelidir.
2. ˙Iptal edilebilirlik: Kullanıcı parmağını etkileşim bölgesinin dışına kaydırarak
işlemi iptal edebilmelidir. Bu, yanlışlıkla dokunmalara karşı güven hissi
sağlar.

Düşük Seviye Yaşam Döngüsü ve PanResponder
Daha özel drag/sürükleme senaryolarında, düşük seviye yaşam döngüsü call-
back’leri kullanılarak bir bileşenin responder olup olmayacağı belirlenebilir:
• onStartShouldSetResponder, onMoveShouldSetResponder: Dokunuşu
sahiplenme isteğinin belirtildiği aşamalar.
• onResponderGrant, onResponderMove, onResponderRelease: Dokunuş
verisini aktif olarak işleyen aşamalar.
• onResponderTerminationRequest, onResponderTerminate: Dokunuşun
başka bir bileşene devredilmesi veya iptal edilmesi ile ilgili aşamalar.
Bu düşük seviye akışla doğrudan çalışmak yerine, React Native çekirdeğinde
yer alan PanResponder aracı sıklıkla tercih edilir. PanResponder, başlangıç/
hareket / bırakma mantığını paketleyerek özellikle:
• Kart sürükleme,
• Basit swipe veya pan jestleri,
• Özel slider bileşenleri
gibi etkileşimlerde daha ergonomik bir API sunar.
Modern Yaklaşım: react-native-gesture-handler
Daha karmaşık, performans kritik veya çok sayıda jestin kullanıldığı ürün se-
viyesindeki uygulamalarda, topluluk tarafından yaygın olarak:
• react-native-gesture-handler (jest algılama),
• Reanimated (animasyonların UI thread üzerinde çalıştırılması)
yığını tercih edilmektedir. Bu yığın, Gesture Responder System’in üzerine
kurulu olmakla birlikte, platformun yerel (native) jest sistemleriyle daha sıkı
entegre çalışır ve modern “drag / swipe / bottom sheet” benzeri arayüzlerin
geliştirilmesinde fiili standart haline gelmiştir.

Mobil Uygulama Güvenliği için Kısa Rehber
Mobil tarafta hiçbir uygulama mutlak anlamda “tam güvenli” değildir. Güvenlik,
her zaman bir risk azaltma ve maliyet dengesi meselesidir: Verinin ne kadar has-
sas olduğu, kaç kullanıcının etkileneceği ve bir ihlal durumunda oluşacak zararın
boyutu, hangi önlemlerin alınması gerektiğini belirler. Harcanan emek ve disi-
plin arttıkça, saldırganın işini zorlaştıran katman sayısı da artar.
Hassas bilgiler söz konusu olduğunda, API key ve secret gibi değerlerin asla
doğrudan uygulama koduna gömülmemesi gerekir. react-native-config veya
react-native-dotenv benzeri kütüphaneler, derlenmiş bundle içinden hala
okunabildiği için bu değerleri gerçekten “gizlemez”; yalnızca yönetimini ko-
laylaştırır. Doğru model, her zaman bir orchestration katmanı kullanmaktır:
istemci, önce sizin kontrolünüzdeki backend’e veya serverless fonksiyona is-
tek gönderir; asıl üçüncü parti API’lerle konuşan ve secret’ları tutan taraf da
yalnızca bu sunucu olur. Böylece client, kritik bilgilere hiçbir zaman doğrudan
erişemez.
Verinin saklanma şekli de en az ne saklandığı kadar önemlidir. Diske yazılan
ve uygulama kapandıktan sonra da kalan persisted veriler, kullanım açısından
konforlu olsa da saldırı yüzeyini büyütür. Sadece RAM’de tutulan unpersisted
veriler ise daha güvenli olmakla birlikte, her uygulama açılışında yeniden elde
edilmeyi gerektirir. React Native ekosisteminde yaygın olarak kullanılan Async-
Storage, web’deki localStorage hissine yakın, pratik fakat güvenlik açısından
sınırlı bir çözümdür. Kullanıcı tercihleri, tema ayarları veya önemsiz bayrak-
lar için uygundur; ancak erişim token’ları, şifreler, secret’lar ve benzeri hassas
veriler için tercih edilmemelidir.
Gerçekten hassas bilgiler için, platformların sunduğu güvenli depolama altyapıları
kullanılmalıdır. iOS tarafında Keychain Services, Android tarafında ise En-
crypted Shared Preferences ve Keystore mekanizmaları bu amaçla tasarlanmıştır.
JavaScript katmanından erişim için expo-secure-store veya react-native-keychain
gibi kütüphaneler kullanılabilir. Bunun yanında, Redux state’ini veya benzer
global durumları persist ederken yanlışlıkla token, kredi kartı bilgisi veya kişisel
veri gibi alanları AsyncStorage’a yazmamak; ayrıca loglama, crash raporlama
(Sentry, Crashlytics vb.) gibi kanallara da bu tip verileri sızdırmamak kritik
önemdedir.
Kimlik doğrulama ve deep linking tarafında da benzer hassasiyet gereklidir.
myapp://products/1 gibi scheme tabanlı deep link’ler, mobil işletim sistem-
lerinde merkezi bir kayıt otoritesine bağlı olmadıkları için başka bir uygulama
tarafından da aynı scheme ile “sahiplenilebilir”. Bu nedenle, deep link üzerinden
erişim token’ı, şifre veya benzeri gizli bilgiler taşımak güvenli kabul edilmez;
yalnızca içerik kimliği, slug veya benzeri non-sensitive veriler bu yolla iletilme-
lidir. iOS’ta universal links kullanmak, içerik odaklı yönlendirmelerde ek bir
güven katmanı sağlar.
OAuth2 akışları söz konusu olduğunda, özellikle mobil ortamda URL scheme
çatışmaları ve yönlendirme zafiyetleri nedeniyle PKCE (Proof of Key Code Ex-
change) mekanizması neredeyse zorunlu bir ek güvenlik katmanı haline gelmiştir.

React Native’de Performans
React Native’de performansın temel hedefi, ekranda saniyede en az 60 kare
üreterek arayüzün “native” hissini korumaktır. Bu, her frame için yaklaşık
16,67 ms süreniz olduğu anlamına gelir. Bu kısa süre içinde hem JavaScript
tarafındaki hesaplamalar hem de native view güncellemeleri tamamlanamazsa
kareler düşer, kullanıcı da takılma ve gecikme hisseder.
JS Thread ve UI Thread Farkı
JavaScript thread’i; React bileşenlerinin render edilmesi, state güncellemeleri,
iş mantığı ve API çağrılarının yürütüldüğü yerdir. Bu thread uzun süre meşgul
olduğunda animasyonlar donar, dokunma olayları geç yanıt verir ve uygulama
ağır hissedilir. Büyük bir component ağacında pahalı bir setState dalgası
200 ms sürdüğünde, arka arkaya birçok frame kaybedebilirsiniz.
UI (main) thread ise native tarafın sorumluluğundadır. Özellikle native-stack
ile yapılan navigation animasyonları ve ScrollView gibi native scroll bileşenleri
burada çalışır. Bu nedenle, JS tarafı kısa süreli kilitlense bile sayfa geçişleri
veya kaydırma hareketi görece akıcı kalabilir; yalnızca scroll event’lerinin JS’ye
iletilmesi gecikir.
Yaygın Performans Sorunları ve Kısa C özümler
Geliştirme modunda (dev = true) ek kontrol ve uyarılar nedeniyle JS thread’i
belirgin biçimde yavaşlar; bu yüzden gerçek performans her zaman release build
üzerinde ölçülmelidir.
Uygulama çapında yoğun console.log kullanımı da, production bundle’da
bile gereksiz iş yükü oluşturabilir; bunu engellemek için babel-plugin-transform-remove-console
gibi araçlar kullanmak faydalıdır.
Büyük listelerde performans problemlerinin sık görülen nedeni, her öğenin
ölçülmesinin pahalı olmasıdır. FlatList kullanırken getItemLayout tanımlamak
bu maliyeti ciddi biçimde azaltır; daha uç senaryolarda FlashList veya benzeri

React Native Build Hızlandırma Rehberi
Büyük React Native projelerinde özellikle Android tarafında native build süreleri
dakikalara çıkabilir. Aşağıdaki teknikler, geliştirme ortamında build sürelerini
kısaltmak için kullanılabilecek, nispeten ileri seviye (yarı-native) optimizasyon-
lardır.
1. Sadece Tek ABI Derlemek (Android)
Android, varsayılan olarak dört mimari için derleme yapar: armeabi-v7a, arm64-v8a,
x86, x86 64. Geliştirme sırasında genellikle tek bir emulator veya tek bir fiziksel
cihaz kullandığınız için, yalnızca aktif mimariyi derlemek ciddi hız kazandırır.
React Native CLI ile:
yarn react-native run-android --active-arch-only
Doğrudan Gradle ile:
./gradlew :app:assembleDebug -PreactNativeArchitectures=x86,x86_64
gradle.properties üzerinden varsayılan ayar:
# Hepsini build et (default)
reactNativeArchitectures=armeabi-v7a,arm64-v8a,x86,x86_64
Release APK/AAB üretirken tüm ABI’lar için derleme yapmayı unutmamak
gerekir; aksi halde bazı cihazlarda uygulama çalışmayabilir.
2. Gradle Configuration Cache (Android)
Gradle build süreci, configuration (tüm build.gradle dosyalarının okunması
ve yapının kurulması) ve execution (derleme ve task’lerin çalıştırılması) fa-
zlarından oluşur. Configuration cache etkinleştirildiğinde, ilk build’te yapılan
konfigürasyon hesaplaması cache’e yazılır ve sonraki build’lerde bu faz büyük
ölçüde atlanır.
android/gradle.properties içine:
org.gradle.configuration-cache=true
Özellikle Android/native tarafa sık sık dokunulan projelerde önemli zaman
kazancı sağlayabilir.

FlatList Performans Cheat Sheet
Temel Kavramlar
FlatList, altında VirtualizedList motorunu kullanarak büyük listeleri verimli
şekilde render eder. Performansla ilgili temel kavramlar şunlardır:
• VirtualizedList: FlatList’in üzerinde çalıştığı sanallaştırılmış liste altyapısıdır;
yalnızca görünür pencereyi ve yakın çevresini render eder.
• Memory consumption: Aynı anda bellekte tutulan item sayısı ve bu
item’lara ait view verisinin toplam maliyetidir; çok yükselirse crash riski
artar.
• Responsiveness: Kullanıcının dokunuş, kaydırma gibi etkileşimlerine ne
kadar hızlı yanıt verildiğidir.
• Blank areas: Scroll sırasında liste henüz render edemediği için ekranda
boş gri/beyaz alanların görülmesidir.
• Viewport: Ekranda o anda fiilen görünen bölgedir.
• Window: FlatList’in bellekte tuttuğu görünür pencere; viewport’tan
geniş bir alan kapsar (üst ve alt buffer).
Önemli FlatList / VirtualizedList Props
removeClippedSubviews Viewport dışındaki item’ları native view hiyerarşisinden
detach eder. Ana thread üzerindeki yükü azalttığı için dropped frame sayısını
düşürebilir; ancak özellikle iOS’ta transform ve position: absolute kom-
binasyonlarında içerik kaybolması gibi yan etkilere yol açabilir. Bellek kazancı
sınırlıdır; view’lar detach edilir ama tamamen deallocate edilmez.
maxToRenderPerBatch (varsayılan: 10) Her batch’te kaç item’ın render
edileceğini belirler. Değeri artırmak, scroll sırasında boş alan görme olasılığını
azaltır (fill rate artar); ancak aynı anda daha fazla iş JS thread’e yükleneceği
için responsiveness düşebilir.
updateCellsBatchingPeriod (varsayılan: 50 ms) Batch’ler arasındaki bek-
leme süresini (milisaniye) tanımlar; yani yeni batch’lerin ne kadar sık tetik-
leneceğini kontrol eder. maxToRenderPerBatch ile birlikte kullanılarak “daha
sık, az item” ya da “daha seyrek, çok item” gibi stratejiler kurulabilir. C ok
seyrek batch, blank area riskini artırırken; çok sık batch JS thread’i tıkayarak
responsiveness’i olumsuz etkileyebilir.
initialNumToRender (varsayılan: 10) ˙Ilk render’da kaç item’ın çizileceğini
tanımlar. Bu değeri, ekranı baştan tam dolduracak şekilde ayarlamak ilk yükleme
performansına büyük katkı sağlar. C ok düşük tutulursa daha ilk ekranda dahi
boş alanlar görülebilir.

JS Yükleme Performansı için Mini Not
JavaScript kodunun parse edilmesi ve çalıştırılması hem zaman hem de bellek
tüketir. Uygulama büyüdükçe, gereksiz kodu başlangıçta yüklemek yerine ihtiyaç
oldukça (lazy) yüklemek, özellikle açılış süresi ve ilk etkileşim performansı açısından
büyük fayda sağlar. React Native, belirli optimizasyonları zaten sunar; fakat
doğru ayarlarla bu sürece siz de aktif katkıda bulunabilirsiniz.
1. Hermes Kullanın ( Önerilen)
Hermes, yeni React Native projelerinde varsayılan JavaScript engine olarak gelir.
Özellikle release build’lerde:
• JS kodunu önceden bytecode’a çevirir (AOT).
• Bytecode’u ihtiyaç oldukça belleğe yükler.
• Her seferinde plain JS parse etmek yerine bytecode üzerinden çalıştığı için
startup süresi belirgin biçimde kısalır.
2. Büyük Component’leri Lazy-Load Edin
Ekran (screen) seviyesindeki ağır component’ler veya ilk açılışta görülme olasılığı
düşük, kalın bağımlılıklara sahip modüller, React.lazy ve Suspense ile lazy-
load edilmelidir. Böylece bu component’lerin kodu, gerçekten ilk kez render
edilene kadar bundle’dan çekilmez; yeni ekranlar eklendiğinde dahi uygulama
açılış süresi daha az etkilenir.
3. Module Side Effect’lerinden Kaçının
Lazy-load ve inline require, modüllerin çalıştırılma sırasını değiştirir. Eğer bir
modül:
• global seviyesinde değişiklik yapıyor,
• event’lere subscribe oluyor,
• polyfill veya patch uyguluyorsa,
bu davranışlar side effect olarak değerlendirilebilir. Bu tür side effect’ler,
modülü lazy-load etmeye başladığınızda, beklenmedik bug’lara veya crash’lere
yol açabilir. ˙Ideal olan, modüllerin yalnızca değer export etmesi; global etkilerin
ise component lifecycle içinde, kontrollü noktalarda çalıştırılmasıdır.

duyulduğunda yüklenir. Bu yöntem senkron çalışır; React.lazy / import()
gibi async değildir.

React Native’de Zamanlayıcılar (Timers)
React Native, tarayıcı ortamındaki zamanlayıcı API’lerini büyük ölçüde aynı
şekilde sunar. Temel zamanlayıcı fonksiyonları şunlardır:
• setTimeout / clearTimeout
• setInterval / clearInterval
• setImmediate / clearImmediate
• requestAnimationFrame / cancelAnimationFrame
Bu fonksiyonlar, JS event loop’u ve React Native bridge mekanizması ile
birlikte çalışarak, farklı zamanlama ihtiyaçlarına cevap verir.
requestAnimationFrame vs. setTimeout(fn, 0)
requestAnimationFrame(fn)
• Bir sonraki paint/frame öncesinde çalıştırılmak üzere planlanır.
• Ekran yenileme döngüsüne senkron olduğu için, animasyon ve frame’e
bağlı hesaplamalar (örneğin konum/görsel güncelleme) için idealdir.
• Amaç: “Bir sonraki çizimden hemen önce bu işi yap” demektir.
setTimeout(fn, 0)
• “Bu blok biter bitmez, mümkün olan en kısa sürede çalıştır” gibi düşünülebilir.
• Frame’e senkron değildir; yalnızca bir sonraki zamanlama aralığında, JS
boşalınca tetiklenir.
• Modern cihazlarda çok sık tetiklenebilir; örneğin iPhone 5S gibi bir cihazda
saniyede 1000+ kez bile çağrılması mümkün olabilir.
Özetle:
• requestAnimationFrame: frame’e senkron, çizim öncesi animasyon/hesaplama
için uygundur.
• setTimeout(fn, 0): sadece “hemen sonra” çalışır; fakat ekranın çizim
ritmiyle doğrudan bağlantısı yoktur.

setImmediate Nedir?
setImmediate, içinde bulunduğunuz JS bloğunun çalışması tamamlandıktan
hemen sonra, native tarafa gönderilecek batched response’tan önce çalıştırılır.
Yani:
• Mevcut senkron JS işi bittiğinde,
• Native katmana kontrollü bir geçiş yapılmadan hemen önce devreye girer.
Önemli bir detay:
• setImmediate içinden tekrar setImmediate çağırırsanız, araya native taraf
girmeden ardışık şekilde JS içinde çalışmaya devam edebilir.
React Native’in Promise implementasyonu, asenkronluk sağlamak için içerde
setImmediate benzeri mekanizmalar kullanır; böylece “bir sonraki tick” mantığıyla,
mevcut çağrı yığını boşaldıktan sonra çalışacak işler planlanmış olur.
Bu kısa not, React Native’de farklı zamanlayıcıların ne zaman tetiklendiğini
zihinde netleştirmek ve özellikle animasyon ile genel asenkron işler arasında
doğru aracı seçmek için bir referans olarak kullanılabilir.

React Native Codegen
1. Codegen’in ˙Işlevi
• Codegen, modül ve bileşen arayüzlerinin TypeScript/Flow ile tanımlanması
ve bu tanımlardan hareketle köprü (bridge) katmanının otomatik
olarak üretilmesini sağlayan bir araçtır.
• Geliştirici yalnızca bir spec (sözleşme) dosyası oluşturur; örneğin:
– Modül adı: MathModule,
– Fonksiyon imzaları: add(a, b): number vb.
• Codegen, bu tip tanımlarını esas alarak:
– Android ve iOS için native modül iskeletlerini,
– Ortak C++ köprü (glue) katmanını
derleme aşamasında otomatik biçimde üretir.
2. Gereklilik Gerekçesi
• Codegen kullanılmadığı durumda:
– Aynı modül arayüzünün JavaScript, Android ve iOS kod tabanlarında
ayrı ayrı ve manuel olarak tanımlanması gerekmektedir.
– Herhangi bir imza veya tip değişikliğinde, bu değişikliğin üç farklı
platformda da tutarlı biçimde güncellenmesi zorunlu hâle gelir; bu
da bakım maliyetini ve hata olasılığını artırır.
• Codegen kullanıldığında:
– Modül arayüzü için tek bir hakikat kaynağı (TypeScript/Flow
spec dosyası) oluşturulmuş olur.
– Android ve iOS köprü katmanları, bu merkezi tip tanımından otomatik
olarak türetildiği için, tip uyuşmazlığı, imza farklılığı veya yazım
hatası kaynaklı sorunlar önemli ölçüde azaltılır.

React Native Headless JS
1. Headless JS nedir?
• Headless JS = Uygulama arka plandayken JavaScript çalıştırma mekaniz-
masıdır.
• Tipik kullanım senaryoları:
– Arka planda veri senkronizasyonu yapmak,
– Push bildirimi geldiğinde belirli bir işlem gerçekleştirmek,
– Kullanıcı arayüzü kapalıyken dahi müzik/medya çalmaya devam et-
mek.
• Temel kural: Headless JS içinde UI’ya dokunulmaz; ekran değiştirme,
component render etme gibi işlemler yapılmaz, yalnızca arka plan işleri
yürütülür.
2. JS tarafı: Task tanımlama
Headless JS task’i JavaScript tarafında AppRegistry üzerinden kaydedilir:
import { AppRegistry } from ’react-native’;
AppRegistry.registerHeadlessTask(’SomeTaskName’, () =>
require(’./SomeTaskName’),
);
˙Ilgili dosyada (örneğin SomeTaskName.js) asenkron task tanımı yapılır:
module.exports = async (taskData) => {
// network isteği, timer, log vb.
// Dikkat: UI yok!
};
• Task tamamlandığında (Promise resolve olduğunda), React Native JS mo-
toru tekrar “uyku” durumuna geçer.

3. Native tarafı (Android Service) ana fikir
Android tarafında, Headless JS task’ini başlatmak için HeadlessJsTaskService
sınıfından türetilen küçük bir servis yazılır:
public class MyTaskService extends HeadlessJsTaskService {
@Override
protected @Nullable HeadlessJsTaskConfig getTaskConfig(Intent intent) {
Bundle extras = intent.getExtras();
if (extras != null) {
return new HeadlessJsTaskConfig(
"SomeTaskName", // JS’te register edilen isim
Arguments.fromBundle(extras), // taskData
5000, // timeout (ms)
false // foreground’da çalışsın mı?
);
}
return null;
}
}
Manifest’e servis tanımı eklenir:
<service android:name=".MyTaskService" />
Daha sonra native tarafta bu servis başlatıldığında:
Intent service = new Intent(getApplicationContext(), MyTaskService.class);
Bundle bundle = new Bundle();
bundle.putString("foo", "bar");
service.putExtras(bundle);
getApplicationContext().startForegroundService(service);
JS motoru ayağa kalkar, SomeTaskName task’i çalıştırılır ve iş bittikten sonra
tekrar durdurulur.

4. Retry (yeniden deneme) mekanizmasının özü
• Varsayılan olarak Headless JS task’leri otomatik retry yapmaz.
• Yeniden deneme davranışı isteniyorsa:
– Native tarafta HeadlessJsRetryPolicy (örneğin LinearCountingRetryPolicy(3,
1000)) tanımlanır,
– JS tarafında hata durumunda özel bir hata fırlatılır.
JS tarafında örnek kullanım:
import { HeadlessJsTaskError } from ’HeadlessJsTask’;
module.exports = async (taskData) => {
const ok = /* ... */;
if (!ok) {
throw new HeadlessJsTaskError(); // retry tetikler
}
};
5. Dikkat edilmesi gereken temel noktalar
• Headless task, varsayılan konfigürasyonda foreground durumdayken çalıştırılırsa
uygulama crash edebilir; bu davranış ancak açıkça allowInForeground =
true ile değiştirilir (performans açısından genellikle önerilmez).
• BroadcastReceiver üzerinden başlatılıyorsa, cihazın uykuya geçmesini
engellemek için HeadlessJsTaskService.acquireWakeLockNow(context)
çağrısı yapılmalıdır.
• Tipik kullanım: Örneğin ağ bağlantısı yeniden geldiğinde, uygulama arka
plandayken sessizce veri senkronizasyonu yapmak gibi senaryolardır.

6. “Bu npm kütüphaneleriyle yapılamaz mıydı?” sorusu
Bu soru iki yönlü cevaplanabilir:
Evet: C oğu durumda npm kütüphaneleri bu işi sizin yerinize yapar.
• Örneğin:
– react-native-background-fetch,
– react-native-push-notification,
– react-native-track-player
gibi kütüphaneler:
– Android tarafında Headless JS + native service,
– iOS tarafında ilgili background modlarını (BGTask, background fetch,
push, audio vb.)
sizin adınıza kullanır.
Geliştirici yalnızca:
BackgroundFetch.configure(...);
TrackPlayer.setupPlayer(...);
gibi JS API’lerini çağırırken, kütüphane yazarı arka planda: HeadlessJsTaskService,
HeadlessJsTaskConfig ve AppRegistry.registerHeadlessTask(...) gibi
detayları halletmiş olur. Bu nedenle, sanki yalnızca “npm paketiyle çözülmüş”
gibi görünür; aslında bu paketler JS + native + Headless JS kombinasyonudur.
Hayır: Saf npm (sadece JS) ile tam arka plan işi yapılamaz.
• JS tarafındaki setTimeout, setInterval, Promise, async/await gibi
mekanizmalar, yalnızca uygulama foreground’dayken ve JS engine çalışırken
etkindir.
• Uygulama arka plana geçtiğinde, hatta tamamen kapatıldığında veya işletim
sistemi tarafından sonlandırıldığında:
– “Her 15 dakikada bir sync yap”,
– “Ağ bağlantısı gelince sessizce veri çek”,
– “WiFi değişince otomatik bir işlem yap”
gibi senaryolar yalnızca saf JS ile gerçekleştirilemez; bu noktada:
– Android’de Headless JS + Service,
– iOS’ta background modes + native kod
devreye girer.

7. Headless JS ne zaman gerçekten gerekli olur?
Aşağıdaki durumlarda hazır npm paketleri yetersiz kalabilir ve doğrudan Head-
less JS kullanımı gündeme gelebilir:
1. C ok spesifik bir native SDK’nın sarmalanması gerektiğinde (örneğin yalnızca
Android’e özgü bir telekom SDK’sı ve belirli event’lerde özel arka plan
task’leri tetiklenmesi isteniyorsa),
2. Mevcut kütüphanelerin sağladığı davranışın ihtiyaçları tam olarak karşılamadığı,
özel “background sync” senaryolarının yazılmak istendiği durumlarda,
3. Kendi React Native kütüphanenizi geliştirdiğiniz ve diğer projelerin de
npm üzerinden kullanacağı bir çözüm sunduğunuz durumlarda.
Bu tür durumlarda, native tarafta MyTaskService extends HeadlessJsTaskService
ve JS tarafında AppRegistry.registerHeadlessTask kullanarak kendi arka
plan işçinizi oluşturursunuz.
