Introduction
Bu doküman, mobil geliştirme dünyasına React Native ekseninden adım atmak
isteyen geliştiriciler için pratik bir giriş rehberi olarak tasarlanmıştır. Amaç,
yalnızca teknik detayları art arda sıralamak değil; modern mobil uygulama
geliştirme sürecini uçtan uca zihinde canlandırabilecek bütüncül bir çerçeve
sunmaktır.
Metin boyunca; bileşen temelli düşünme biçimi, navigation yaklaşımları, du-
rum yönetimi, performans optimizasyonu, yerel (native) modüllerle entegrasyon,
paket yönetimi, dağıtım (build / release) süreçleri ve platforma özgü (Android
/ iOS) temel kavramlar sistematik biçimde ele alınmıştır. Böylece okuyucunun
yalnızca “nasıl kod yazacağını” değil, aynı zamanda yazdığı kodun platform
tarafında neye dönüştüğünü ve yaşam döngüsünün hangi aşamalardan geçtiğini
görmesi hedeflenmiştir.
Rehber, deneyim seviyesinden bağımsız olarak, iki tür ihtiyaca cevap vermeyi
amaçlar:
• Mobil geliştirmeye yeni başlayanlar için: Temel kavramları, en sık kul-
lanılan araçları ve React Native ekosistemini sindirilebilir bir dille tanıtmak.
• Halihazırda geliştirme yapanlar için: Dağınık bilgi kırıntılarını bir araya
getiren, kavramsal boşlukları dolduran, “neden böyle tasarlanmış?” sorusuna
yanıt veren bir referans çerçevesi oluşturmak.
Bu çerçevede doküman, bir “hızlı başlangıç”tan ziyade, mobil geliştiricinin
kafasında sağlam bir zihinsel model inşa etmeyi amaçlayan bir giriş katmanı
olarak okunmalıdır. Detaylı dokümantasyonların ve resmi kaynakların öncesinde,
nereden başlayacağını bilemeyen geliştiriciye pusula işlevi görmesi hedeflenmiştir.

1. Activity / ViewController Lifecycle
Mobil uygulamalarda ekranlar kullanıcıdan bağımsız olarak işletim sistemi
tarafından yönetilir. Bu durum, web’deki SPA (Single-Page Application)
modelinden tamamen farklıdır.
Android Activity Lifecycle
onCreate() → UI oluşturulur, bağımlılıklar yüklenir.
onStart() → Ekran görünür hale gelir.
onResume() → Kullanıcı ekranı aktif kullanır.
---
onPause() → Kısmen görünür (popup, başka uygulama).
onStop() → Tamamen görünmez, background’a alınır.
onDestroy() → OS tarafından tamamen öldürülür.
iOS ViewController Lifecycle
viewDidLoad()
viewWillAppear()
viewDidAppear()
viewWillDisappear()
viewDidDisappear()
deinit()
Bu metodlar, ekranın ne zaman oluşturulacağı, görünür olacağı veya yok
edileceği konusunda kritik kontrol noktaları sağlar.
2. Background → Foreground Geçişleri
Mobil işletim sistemleri agresif davranır:
• RAM sıkıştığında arka plandaki Activity’leri öldürür.
• CPU yükünde animasyonları durdurur.
• Uzun süre background’da kalan uygulamaları askıya alır (suspend).
Bu süreçte oluşan tipik geçişler:
onPause → Kullanıcı görmüyor.
onStop → Uygulama tamamen arka planda.
OS Kill → Gerekirse uygulama tamamen yok edilir.
Uygulama foreground’a döndüğünde tekrar onCreate veya onResume çalışabilir.

UI State Yalnızca RAM’de Tutulmamalıdır
Modern mobil ve web uygulamalarında UI state yalnızca RAM üzerinde (com-
ponent içi değişkenlerde) tutulursa, uygulama kapandığında veya işletim sistemi
süreci sonlandırdığında tüm state kaybolur. Bu nedenle kritik uygulama durum-
larının mutlaka kalıcı bir depolama alanına yazılması gerekir.
1. RAM Neden Güvenilir Değildir?
• Uygulama arka plana alınıp öldürülebilir. Android ve iOS düşük
bellek durumunda uygulama süreçlerini kapatır.
• Refresh / reload / navigation durumlarında state sıfırlanır. Web’de
F5, mobilde cold start, ekran döndürme gibi durumlarda component life-
cycle yeniden oluşur.
• OS memory reclaim yapabilir. ˙Işletim sistemi RAM baskısı olduğunda
uygulamayı kapatabilir.
Bu nedenle component içinde tutulan state (React useState, Angular com-
ponent properties, context, MobX, Zustand, NgRx store vb.) kalıcı değildir.
2. Hangi State’ler Kalıcı Olmalıdır?
Aşağıdaki state türleri mutlaka persistent storage’a yazılmalıdır:
• Authentication bilgileri: access token, refresh token, session metadata.
• Kullanıcı tercihleri: dil, tema, font boyutu, izinler.
• Offline-first veriler: önbelleğe alınmış API sonuçları, gönderilmemiş
formlar, lokal mesajlar.
• Kullanıcı ilerleme verileri: onboarding tamamlandı mı, chat geçmişi,
notlar, tamamlanmış adımlar.

React Native’de Lifecycle Nasıl C alışır?
React Native, üç katmanlı bir mimaridir:
Android (Java/Kotlin)
iOS (Objective-C/Swift)
JavaScript (React)
Native taraf Activity/ViewController yaşam döngüsünü yönetirken, JS tarafı

React Native, native yaşam döngüsü değişikliklerini JavaScript tarafına şu API

AppState

inactive → Geçiş aşaması.
Örnek Kullanım

const sub = AppState.addEventListener("change", state => {
if (state === "active") console.log("Foreground");
if (state === "background") console.log("Background");
});
return () => sub.remove();
}, []);

React Native’de JS → Native Geçişi
React Native, JavaScript (JS) kodu ile platforma özgü (iOS/Android) native
API’ler arasında iki farklı mimari kullanır:
1. Eski Mimari: Bridge Tabanlı Model
2. Yeni Mimari: JSI + TurboModules + Fabric
Aşağıda iki mimari arasındaki fark ve geçiş sürecinin teknik detayları an-
latılmaktadır.
1. Eski Mimari: Bridge Tabanlı Yapı
React Native 0.69 ve öncesinde JavaScript kodu ile native taraf arasındaki
iletişim asenkron bir Bridge üzerinden sağlanır.
1.1. Temel Akış
1. JS kodu bir native fonksiyon çağırır:
NativeModules.Location.getCurrentPosition();
2. Bu çağrı JSON formatına dönüştürülür ve Bridge’e iletilir:
{
"module": "Location",
"method": "getCurrentPosition",
"params": []
}
3. Bridge bu mesajı Native Modules Message Queue’suna aktarır.
4. Native taraf ilgili modülü ve metodu çağırır.
5. Native sonuç döndürmek isterse aynı Bridge üzerinden tekrar JSON mesajı
olarak JS’e geri gönderir.
1.2. Eski Mimari’nin Sınırlamaları
• JSON serileştirme/deserileştirme maliyetlidir.
• JS ve Native arasında thread hop çok fazladır.
• Büyük veri taşımak (örneğin: video frame’leri) verimsiz ve yavaştır.
• Senkron (synchronous) fonksiyon çağrıları mümkün değildir.

Yeni Mimari: JSI + TurboModules + Fabric
Modern React Native mimarisinde Bridge ortadan kaldırılmıştır. JS ile native
taraf arasında doğrudan C++ tabanlı bir arayüz (JSI) kullanılır.
JSI (JavaScript Interface)
JSI, JavaScript kodunun C++ seviyesindeki objelere ve fonksiyonlara doğrudan
erişmesini sağlayan yeni düşük seviye API’dir.
Temel Akış
1. JS kodu bir TurboModule fonksiyonunu çağırır:
Location.getCurrentPosition();
2. Bu çağrı JSON’a dönüştürülmez; doğrudan C++ pointer üzerinden
ilgili native fonksiyon invoke edilir.
3. Native fonksiyon sonucu yine JSI üzerinden JS runtime’a döner.
Yeni Mimari’nin Avantajları
• JSON yoktur, serialization overhead ortadan kalkar.
• Sıfıra yakın gecikme (low-latency) ile çağrı yapılır.
• Zero-copy veri aktarımı mümkündür (özellikle medya verilerinde).
• Senkron fonksiyon çağrıları mümkündür.
• Daha yüksek FPS, daha akıcı animasyonlar ve gesture performansı sağlanır.

React Native’in Limitasyonları
React Native, SPA modeli nedeniyle bazı durumlarda zayıftır:
• AppState gecikebilir.
• Bazı Android cihazlarda background event’leri engellenir.
• WebSocket bağlantıları background’da kopabilir.
• Kamera/video background’da otomatik kapanmaz.
• Timer’lar background’da çalışmaz.
Dikkat Gerektiren Durumlar
• Screen/audio recording → onPause’da durdurulmalı.
• WebSocket → background dönüşünde yeniden bağlanmalı.
• Kamera → background’da mutlaka kapatılmalı.
• Form state → unmount olursa kaybolmamalı.
• Location tracking → background permission gerekli.
Kavramsal Benzetme
React Native:
• Native lifecycle = Angular Component Lifecycle
• App background = Browser tab inactive
• Bridge = Zone.js benzeri event scheduler
Bu nedenle React Native uygulamaları web SPA gibi görünse de, arka
planda tamamen native lifecycle ile yaşar.

React Native’de AppState Nedir?
React Native’deki AppState, uygulamanın işletim sistemi tarafından hangi yaşam
döngüsü durumunda olduğunu (foreground, background, inactive) takip eden bir
API’dir. Mobil uygulamalar sürekli olarak OS tarafından duraklatılabilir, arka
plana alınabilir veya öldürülebilir. Bu nedenle uygulama durumlarını yönetmek
için AppState kritik bir yapıdır.
1. AppState Durumları
• active
Uygulama ekranda görünür durumdadır ve kullanıcı etkileşimi mümkündür.
• background
Uygulama arka plandadır; kullanıcı tarafından görülmez. OS bu aşamada
uygulamayı öldürebilir.
• inactive
iOS’ta sık görülen bir ara durumdur. Arama ekranı, kontrol menüsü,
bildirim overlay’i gibi geçici durumlarda oluşur.
2. Örnek Kullanım
import { AppState } from ’react-native’;
AppState.addEventListener(’change’, (state) => {
console.log("Yeni durum: ", state);
});
3. AppState Neden Gereklidir?
Mobil işletim sistemleri (iOS ve Android) bellek ve CPU yönetimi gereği uygu-
lamaları sık sık durdurur, dondurur veya tamamen sonlandırır. AppState bu
geçişleri JS tarafında yönetebilmek için kullanılır.
• OS uygulamayı arka plana attığında işlemler durdurulabilir, timer’lar
temizlenebilir, socket bağlantıları kapatılabilir.
• Uygulama tekrar active olduğunda veriler yeniden yüklenebilir, API
çağrıları tazelenebilir, sayfa yenilenebilir.
• Background durumunda kaynak tüketimi azaltılabilir: konum servis-
leri, sensör okuma, animasyonlar, audio/video oynatma vb.

Memory Pressure & Low Memory Kill
Mobil cihazlarda RAM sınırlıdır. Bu nedenle iOS ve Android işletim sistemleri,
bellek kritik seviyelere düştüğünde uygulamalara baskı yapar, uyarır ve gerekirse
uygulama sürecini tamamen sonlandırır. Bu duruma memory pressure denir.
1. Low Memory Warning
iOS, bellek seviyesi kritikleştiğinde UIApplicationDidReceiveMemoryWarningNotification
üretir. Bu uyarı geliştiriciye:
• gereksiz cache’lerin temizlenmesi,
• büyük resimlerin bellekten atılması,
• geçici objelerin serbest bırakılması
gerektiğini bildirir.
Uyarı dikkate alınmazsa işletim sistemi kısa süre içinde uygulamayı “Ter-
minated due to memory pressure” hatasıyla sonlandırır.
2. Background Task Kill
Uygulama arka plana geçtiğinde OS yalnızca birkaç saniyelik yaşam hakkı tanır.
Bu süreden sonra uygulama:
• iOS’ta suspended durumuna alınır,
• Android’de LMK (Low Memory Killer) tarafından sınıflandırılır.
Bellek ihtiyacı artarsa OS en önce arka plandaki uygulamaları öldürür.
3. App Process’inin OS Tarafından Kesilmesi
˙Işletim sistemi aşağıdaki durumlarda süreci zorla sonlandırabilir:
• bellek yetersizliği,
• yüksek CPU tüketimi,
• battery saver modları,
• kontrolsüz timer veya animasyon kullanımı,
• memory leak.
Android loglarında genellikle LowMemoryKill mesajı, iOS crash loglarında
jetsamEvent görülür.

4. Large Image Footprint
Diskte 3MB olan bir görsel RAM’de decode edildiğinde 20–50MB yer kaplaya-
bilir. Örneğin:
1080 × 1920 × 4byte ≈ 8.3M B
Ekranda birden fazla büyük görsel bulunması, uygulamanın kısa sürede
memory pressure altında kalmasına yol açar. Bu nedenle:
• downsampling,
• WebP/AVIF formatları,
• FastImage,
• uygun cache stratejileri
kullanılmalıdır.
5. Memory Leak Detection
Memory leak’ler zamanla native heap’in büyümesine ve OS tarafından uygula-
manın öldürülmesine neden olur.
Android Araçları
• LeakCanary
• Android Studio Memory Profiler
iOS Araçları
• Xcode Instruments (Leaks, Allocations, Memory Graph)
• Malloc Stack Logging
Özet
Bellek baskısı altında işletim sistemi uygulamayı önce uyarır, gerekirse
arka planda dondurur ve en sonunda tamamen sonlandırır. Büyük
resimler, bellek yönetimi hataları ve memory leak’ler mobil uygula-
malarda en sık görülen sorunlardır.

React Native’de Memory Leak Detection
React Native uygulamalarında memory leak’ler hem JavaScript tarafında (Her-
mes heap) hem de native taraflarda (iOS/Android) oluşabilir. Bu nedenle
leak tespiti yalnızca JS seviyesinde değil, platformun kendi araçlarıyla birlikte
yapılmalıdır.
1. React Native’e Özel Araçlar
1.1 react-native-performance
Yeni mimari (JSI + TurboModules) ile uyumlu modern bir ölçüm aracıdır.
Aşağıdaki leak türlerini tespit edebilir:
• JSI tabanlı referans sızıntıları,
• Component remount sonrası büyüyen heap,
• Render sonrası anormal memory trendleri.
1.2 react-native-memory-leak-listener
JavaScript tarafındaki GC yapılmamış nesneleri dinleyerek basit leak tespiti
sağlar. Yalnızca JS heap üzerinde etkili olup native leak’leri yakalayamaz.
2. Android ˙Için Araçlar
2.1 LeakCanary ( Önerilen)
Android’de React Native uygulamaları için en güvenilir leak tespit aracıdır.
Aşağıdaki leak türlerini otomatik olarak yakalayabilir:
• Activity / Fragment leak,
• Context leak,
• View ve ViewGroup leak,
• React Native Surface / ViewManager leak,
• GestureHandler veya Navigation container leak.
2.2 Android Studio Memory Profiler
Java/Kotlin heap dağılımını, retained memory miktarını ve object lifecycle anal-
izini gösterir.

3. iOS ˙Için Araçlar
3.1 Xcode Instruments
React Native ile en uyumlu ve en detaylı leak tespit aracıdır. Özellikle şu
modüller kullanılmalıdır:
• Leaks (retain cycle tespiti)
• Allocations (heap büyümesini izleme)
• Memory Graph (strong reference zincirleri)
Tespit edilebilen leak türleri:
• UIView / UIViewController leak
• RCTBridge ve RCTModule leak
• Timer ve NotificationCenter leak
• Image decode kaynaklı leak
4. Hermes Runtime Memory Tools
Hermes 0.12+ ile JS heap snapshot alınabilir. Bu araçlar:
• JavaScript heap büyümesini,
• detached JS object’leri,
• reference cycle’ları,
• allocation timeline’ı

Heap snapshot, Flipper içinden de alınabilir.
6. Önerilen Mimari Yaklaşım
• Android: LeakCanary kullanılması önerilir.
• iOS: Xcode Instruments ile derin leak analizi yapılmalıdır.
• JS Tarafı: Hermes Heap Snapshot tercih edilmelidir.
• Yeni Mimari: JSI leak’leri için react-native-performance faydalıdır.
Özet
React Native’de leak tespiti yalnızca JS seviyesinde yapılamaz. Uygu-
lama hem native heap hem de JavaScript heap üzerinde leak üretebileceği
için, Android ve iOS’un kendi memory araçlarıyla birlikte analiz
yapılması en doğru yöntemdir.

Mobile Threading Model
Mobil cihazlarda CPU çekirdeği, RAM ve batarya kısıtlı olduğundan, PC’deki
gibi sınırsız thread açmak mümkün değildir. Bu nedenle işletim sistemi, thread’lerin
kullanımını sıkı şekilde kontrol eder ve geliştiricinin doğru thread modelini ben-
imsemesi gerekir.
1. UI Thread (Main Thread)
• Tüm kullanıcı etkileşimleri (touch, scroll, gesture)
• View çizimi ve layout hesaplamaları
• Animasyonlar
• Event dispatch işlemleri
UI thread bloklanırsa uygulama donar ve Android tarafında ANR (Applica-
tion Not Responding) oluşur.
2. Worker Threads
CPU-bound işler için kullanılır:
• JSON parse
• veri işleme (filter, sort, map)
• image processing
• encryption ve compression
Sayıları sınırlıdır; aşırı thread oluşturmak battery drain ve context-switch
maliyeti yaratır.
3. Background Tasks / Job Schedulers
Uzun süreli işler OS tarafından yönetilir:
• Android: WorkManager, JobScheduler, ForegroundService
• iOS: BGTaskScheduler, URLSession background transfers
Bu mekanizmalar enerji dostu şekilde zamanlanır.
React Native Özelinde Thread Modeli
React Native, hem JavaScript tarafını hem de native tarafı yürüttüğü için çok
katmanlı bir thread modeline sahiptir.

1. JS Thread
JavaScript kodunun çalıştığı ana thread’dir.
• React komponentlerinin render mantığı
• state/props hesaplamaları
• timer’lar
• API callback’leri
JS thread’in bloklanması, UI’nın geç tepki vermesine neden olur.
2. UI Thread
Native arayüzün işlendiği thread’dir.
• touch/gesture event’leri
• layout ve render işlemleri
• native animasyonlar
3. Native Modules Thread
Native modüllerin (kamera, dosya sistemi, konum, kontaklar, Bluetooth vb.)
çalıştığı thread’dir.

ANR — Application Not Responding
Android işletim sistemi, UI thread’in 5 saniye boyunca yanıtsız kalması du-
rumunda uygulamayı ANR (Application Not Responding) hatasıyla son-
landırır. Bu durum, kullanıcıya “uygulama yanıt vermiyor” uyarısı olarak yansır
ve süreç çoğunlukla uygulamanın öldürülmesiyle sonuçlanır.
1. ANR’a Neden Olan Durumlar
ANR’ın temel nedeni UI thread’in bloklanmasıdır. UI thread tıkandığında:
• touch/scroll event’leri işlenemez,
• view redraw işlemleri durur,
• layout hesaplamaları yapılamaz,
• event loop kilitlenir.
React Native uygulamalarında aşağıdaki işlemler ANR’a sıkça neden olur:
• Büyük JSON parse işlemleri
• Geniş array üzerinde map/filter/reduce
• Kriptografik işlemler (hash, encrypt, PBKDF2)
• Büyük görsel decode işlemleri
• Senkron depolama erişimi (blocking I/O)
2. ANR’ın Temel Mekanizması
Android OS, UI thread’in 5 saniye boyunca input event’lerine yanıt ver-
memesini kritik bir hata olarak kabul eder. Bu süre içinde:
• yeni frame üretilemez,
• kullanıcı eylemleri kuyruğa alınır ancak işlenmez,
• “looper” tıkanır.
Sonuç olarak OS, kullanıcı deneyimini korumak adına uygulamayı sonlandırır.

Mobile Networking Limitations
Mobil cihazlarda ağ iletişimi, masaüstü veya sunucu ortamlarına göre çok daha
değişken, kırılgan ve maliyetlidir. Bu nedenle mobil uygulamalar hem bağlantı
kalitesindeki belirsizliklere hem de enerji ve veri tüketimi kısıtlarına göre tasar-
lanmalıdır.
1. Unstable Network (Kararsız Bağlantı)
Mobil cihazlar sürekli olarak:
• Wi-Fi ↔ Mobil veri değişimi,
• zayıf sinyal bölgeleri,
• tünel/metro kesintileri,
• hücre değişimi (cell handoff)
gibi bağlantı dalgalanmalarına maruz kalır.
Bu durum, HTTP çağrılarında:
• paket kaybı,
• DNS hataları,
• response gecikmesi,
• TCP yeniden aktarımı
gibi maliyetlere yol açar.
2. Captive Networks
Bazı Wi-Fi ağları internet erişimi sağlamadan önce kullanıcıya bir doğrulama
sayfası gösterebilir (örneğin otel, havaalanı). Bu durumdaki ağlara:
captive network
denir.
Uygulama internete bağlı görünse bile:
• istekler engellenir,
• DNS cevapsız kalır,
• HTTP redirect oluşur,
• TLS handshake başarısız olur.
28

-- 28 of 168 --

3. Packet Loss (Paket Kaybı)
Mobil ağlarda paket kaybı PC ortamlarına göre çok daha yüksektir. Paket kaybı,
özellikle TCP üzerinde:
• yeniden iletim (retransmission),
• artan gecikme,
• düşük throughput,
• bağlantı kopması
gibi etkiler yaratır.
4. Timeout Problemleri
Mobil ağlarda:
• yüksek gecikme (latency),
• dengesiz bağlantı,
• zayıflayan sinyal,
• cihazın uyku moduna geçmesi
timeout hatalarına neden olur. Bu durum mobil uygulamalarda doğru time-
out konfigürasyonunu kritik hale getirir.
5. Cold Start DNS Lookup
DNS çözümü mobilde:
• captive network müdahalesi,
• DNS caching’in olmaması,
• rota değişimi,
• farklı hücreye geçiş
gibi nedenlerle soğuk başlangıç (cold start) maliyeti yaratabilir. DNS lookup
süresi uygulamanın ilk isteğini ciddi şekilde yavaşlatır.
Mobil cihazlarda DNS çözümlemesi (Domain Name System lookup), masaüstü
ortamlara kıyasla çok daha değişken ve maliyetlidir. Cold start DNS lookup,
cihazın belirli bir domain için daha önce yapılmış bir çözümlemeye ait DNS cache
bulunmaması durumunda, DNS’in sıfırdan çözümlenmesi gerektiği senaryodur.
Bu durum özellikle mobil ağda ilk API isteğinin yavaşlamasına neden olur.

1. Cold Start DNS Lookup Neden Oluşur?
• Captive Network Müdahalesi
Otel, havalimanı veya restoran Wi-Fi ağları interneti engelleyip kullanıcıyı
giriş (login) sayfasına yönlendirmek için DNS trafiğini ele geçirir. Bu
işlem sırasında DNS cache temizlenir ve cihazın tekrar DNS çözümlemesi
gerekir.
• DNS Cache’in Olmaması
Mobil cihazlar bağlantı değişimlerinde (Wi-Fi kapatıp açmak, uçak mod-
una geçmek, SIM değişimi vb.) DNS cache’i temizleyebilir. Bu durumda
ilk istek her zaman cold start DNS maliyetini taşır.
• Rota Değişimi (Routing Change)
Mobil ağlarda cihazın baz istasyonları arasında geçişi (handoff) ile bir-
likte network route yeniden oluşturulur. Yeni route, yeni DNS resolver
kullanabilir ve cache geçersiz olur.
• Hücre Değişimi (Cell Handoff )
4G/5G ağlarında cihaz farklı bir hücreye geçtiğinde ağ bağlamı (net-
work context) yeniden kurulur. Bu süreç DNS çözümlemesinin tekrar
yapılmasına neden olur.
2. DNS Lookup Maliyeti
DNS çözümlemesi mobilde genellikle:
• 50–150 ms arasında (iyi bağlantı),
• 300–1000 ms arasında (zayıf bağlantı, captive ağ),
ek gecikme yaratabilir.
Bu nedenle mobilde ilk yapılan API isteğinin toplam süresinin büyük bir
kısmı DNS çözümlemesinden kaynaklanır.
3. Cold Start DNS Probleminin Etkileri
• uygulama açılış süresinin artması,
• login veya ilk fetch operasyonlarının gecikmesi,
• timeout’ların daha kolay tetiklenmesi,
• kullanıcı deneyiminde yavaşlık algısı.

Cold start DNS gecikmesini azaltmak için aşağıdaki stratejiler kullanılabilir:

Uygulama açılır açılmaz düşük maliyetli bir “ping” isteği gönderilerek
DNS çözümünün önceden yapılması sağlanabilir.

HTTP/2 veya HTTP/3 kullanılarak aynı bağlantı üzerinden birden fazla

DNS hataları için farklı resolver’lara fall-back mekanizması uygulanabilir.
Özet

Bu nedenle mobilde:

Web uygulamalarından farklı olarak mobil uygulamalar, işletim sistemi tarafından
herhangi bir anda durdurulabilir veya tamamen sonlandırılabilir. Bu durum,

nenden fazla bellek kullanması durumunda:
• memory warning bildirir (iOS),
• arka plandaki uygulamaları dondurur,
• gerekli görülürse uygulama sürecini öldürür.

LowMemoryKiller.

işlemler uygulamanın öldürülmesine neden olabilir:
• uzun süreli setInterval veya döngüler,
• yoğun CPU kullanan işlemler,
• büyük JSON parse operasyonları,
• büyük dosya veya resim işleme,
• sürekli network istekleri.
iOS, arka planda 30 saniyeden fazla CPU veya ağ işlemi yapan uygulamaları
sonlandırır.
3. Idle Timer
Kullanıcı uzun süre etkileşimde bulunmazsa uygulama:
• önce suspended durumuna alınır,
• ardından bellek ihtiyacı oluştuğunda OS tarafından öldürülür.
Bu davranış özellikle Android’de çok daha agresiftir.

Doze Mode’a Uygun Algoritmalar
Doze Mode, mobil cihazlarda enerji tasarrufu sağlamak amacıyla arka plan
çalışmasını ciddi şekilde kısıtlar. Bu nedenle mobil uygulamalarda timer ta-
banlı döngüler, uzun süreli işlemler ve düşük öncelikli bildirimler Doze Mode
sırasında çalışmaz veya işletim sistemi tarafından ertelenir. Bu durumlara uyum
sağlamak için özel algoritma tasarımları kullanılmalıdır.
1. Timer Değil → Planlı Task (OS-Managed Scheduling)
Doze Mode’da:
• setTimeout ve setInterval çalışmaz,
• JavaScript event loop durdurulur,
• native zamanlayıcılar dondurulur.
Bu nedenle timer tabanlı yaklaşımlar güvenilir değildir.
C özüm: OS Tarafından Yönetilen Görevler
Android: WorkManager veya JobScheduler iOS: BGTaskScheduler
Bu sistemler, işletim sisteminin uygun gördüğü enerji dostu zamanlarda
uygulama görevlerini çalıştırır.
“Timer yerine planlı görev (scheduled task) kullanılmalıdır.”
2. Push Değil → High Priority Push
Doze Mode sırasında normal öncelikli bildirimler gecikebilir. Bunun nedeni:
• FCM bağlantısının düşük güç moduna alınması,
• heartbeat paketlerinin gönderilememesi,
• non-urgent bildirimlerin ertelenmesidir.
Bu durumda standart bildirimler zamanında ulaşmaz.
C özüm: High Priority Push Notifications High priority FCM bildirim-
leri Doze Mode’u geçici olarak uyandırabilir ve bildirimin anında teslim edilmesini
sağlar.
“Normal push bildirimleri yerine yüksek öncelikli push kullanılmalıdır.”

3. Long Tasks Değil → Batching + Snapshot
Doze Mode uzun süreli arka plan işlemlerini durdurur:
• büyük upload işlemleri,
• büyük JSON parse operasyonları,
• CPU yoğun hesaplamalar,
• uzun süreli senkronizasyon işlemleri.
Bu tür işlemler enerji politikaları nedeniyle kesintiye uğrar veya uygulamanın

• uzun hesaplama → parçalı batch işlemleri.
Bu yaklaşım Doze Mode’un CPU ve ağ kısıtlamalarını aşmayı sağlar.
“Uzun işlemler yerine küçük parçalara ayrılmış batch + snapshot
yaklaşımı kullanılmalıdır.”
Özet
Doze Mode sırasında timer’lar çalışmaz, normal push bildirimleri

işlem modelleri ile tasarlanmalıdır.

GPU vs CPU Rendering Pipeline
Mobil kullanıcı arayüzlerinde akıcı animasyonlar ve yüksek performanslı render-
ing için CPU ve GPU arasındaki iş bölümü kritik öneme sahiptir. UI thread
(main thread) tıkandığında jank, frame drop ve kullanıcı deneyiminde gecik-
meler oluşur. Bu nedenle animasyonlar mümkün olduğunca GPU’ya taşınmalı
ve CPU yükü azaltılmalıdır.
1. Rendering Pipeline
Mobil rendering pipeline genel olarak üç temel aşamadan oluşur:
1. Layout ve Measure (CPU) View hiyerarşisinin hesaplanması, ölçüler,
konumlar ve tarz özellikleri CPU tarafından işlenir.
2. Painting (CPU → GPU Hazırlığı) View’ların çizim komutları oluşturulur.
Shadow, border, efekt gibi özellikler maliyet yaratabilir.
3. Compositing (GPU) GPU, çizim katmanlarını (layers) birleştirerek ekran
çıktısını oluşturur. Bu aşama donanım hızlandırmalıdır.
2. Jank Neden Oluşur?
UI thread üzerindeki fazlalık yük aşağıdaki durumlara yol açar:
• frame sürelerinin 16.6ms’yi aşması (60 FPS için sınır),
• ağır layout hesaplamaları,
• senkron işlemler,
• CPU üzerinde yapılan animasyon hesaplamaları.
Sonuçta kullanıcı tarafında jank (takılma) ve frame drop görülür.
3. GPU’ya Taşınması Gereken ˙Işler
Modern mobil arayüzlerde animasyon hesaplamalarının CPU’da yapılmaması
gerekir. Aşağıdaki işlemler GPU üzerinde daha verimlidir:
• off-main-thread animations Ana thread’i bloklamadan animasyonların
ayrı bir pipeline’da çalıştırılması.
• hardware compositing GPU katman birleşimi sayesinde daha akıcı geçişler.
• opacity, transform, scale, translate, rotate Bu özellikler GPU tarafından
doğal olarak hızlandırılır.

4. Maliyetli Görsel Efektler
Aşağıdaki özellikler CPU ve GPU üzerinde yüksek maliyet oluşturur:
• shadow rendering Soft shadow (blur radius yüksek) GPU üzerinde pa-
halıdır.
• blur efektleri Gaussian blur gibi işlemler çoğunlukla CPU veya shader
düzeyinde ağır hesaplama gerektirir.
• overdraw Ekranın aynı pikselinin birden fazla kez çizilmesi; gereksiz view
katmanları, opacity overlay’leri, nested view yapıları performans kaybına

React Native’de rendering iki ana thread’e bölünmüştür:
• UI Thread Layout, ölçüm ve native UI rendering işleri.

JS thread bloklanırsa animasyonlar gecikir ve jank oluşur.
Bu nedenle React Native ekosisteminde GPU hızlandırmalı animasyonlar
için özel kütüphaneler kullanılır.

6.1 Reanimated 2 Reanimated 2, worklet yapısı sayesinde animasyon hesapla-

Avantajları:
• JS thread tıkanıklığından etkilenmez,
• 60 FPS animasyon garantiye daha yakın,

6.2 Skia (react-native-skia) Skia bir GPU tabanlı 2D rendering en-
gine’dir. React Native içinde:
• custom drawing,
• shader efektleri,
• GPU hızlandırmalı canvas çizimi,
için kullanılır. Yüksek performanslı animasyonlar ve görsel efektlerde CPU
→ GPU offload sağlar.

13. App Size Optimization
Mobil uygulamalarda paket boyutunun büyümesi; indirilme oranının düşmesine,
ilk açılış süresinin uzamasına ve cihazda daha fazla disk kullanımı oluşmasına

R8, Proguard’ın modern sürümü olup aşağıdaki işlemleri yapar:
• Shrinking — Kullanılmayan sınıf ve metotların silinmesi
• Minification — Sembollerin küçültülmesi (ClassA → a)
• Obfuscation — Kodun okunamaz hale getirilmesi

Bu optimizasyonlar uygulama boyutunda genellikle %10–25 küçülme sağlar.

Hermes, mobil cihazlar için optimize edilmiş bir JavaScript motorudur. Varsayılan
JSC motoruna göre daha küçüktür ve daha hızlı çalışır.
• JS bundle boyutunu azaltır
• Soğuk başlangıç süresini hızlandırır
• RAM kullanımını düşürür
Aktifleştirildiğinde ortalama 3–8 MB kazanç sağlar.

ICU, dünya dilleri, takvimleri ve sayı formatlarını içeren büyük bir kütüphanedir.
Uygulamada yalnızca birkaç dil kullanılacaksa, diğer diller çıkartılarak boyut
küçültülebilir.
• TR/EN gibi sınırlı dil desteği ile 2–7 MB tasarruf elde edilebilir.

14. Mobile-Specific Concurrency (Basit Anlatım)
Mobil cihazlarda CPU ve RAM sınırlı olduğu için, aynı anda çok fazla iş yapmak
uygulamanın donmasına, animasyonların takılmasına ve veri tutarsızlıklarına
neden olur. Bu nedenle concurrency, mobil geliştirmede kritik bir konudur.
Aşağıda mobilde bilinmesi gereken temel kavramlar sade bir şekilde açıklanmıştır.
1. Job Schedulers (Arka Planda C alıştırıcılar)
Android (WorkManager): Uygulama kapalı olsa bile belirli işleri çalıştırmak
için kullanılır.
iOS (BackgroundTasks API): iOS uygun gördüğünde kısa süreli arka
plan çalışma süresi verir (10–30 saniye). Süre bitince işlemi durdurur.
2. Rate Limiting
Uygulamanın çok hızlı bir şekilde arka arkaya istek göndermesini engeller. Örneğin,
her saniyede en fazla bir API isteği yapılması.
3. Debounce
Bir işlem ancak kullanıcı belirli bir süre durduğunda çalıştırılır. Örnek: Arama
kutusunda kullanıcı yazmayı bıraktıktan 300ms sonra API çağrısının yapılması.
4. Throttle
Bir işlemin belirli aralıklarla çalıştırılmasını sağlar. Örnek: Scroll sırasında her
pikselde değil, her 200ms’de bir event tetiklenmesi.
5. Batching (Toplu Gönderim)
Birden fazla küçük işin tek bir büyük iş olarak gönderilmesidir. Örnek: 10 log
kaydının 10 ayrı istekte değil, tek bir POST isteğinde gönderilmesi.
6. Parallel vs Serial Queues
Serial Queue: ˙Işler sırayla çalışır. UI işlemleri ve React Native JS thread bu
yapıdadır.
Parallel Queue: Birden fazla iş aynı anda çalışabilir, ancak sayısı sınırlıdır.
Arka plan resim işleme, network istekleri gibi işler burada yapılır.
7. Thread Pool Limitleri
Mobil cihazlarda sistem tarafından yönetilen sınırlı sayıda worker thread vardır
(genellikle 4–8). C ok fazla thread açmak performansı düşürür.
53

-- 53 of 168 --

8. Promise Queue Congestion (React Native)
React Native’de JS thread tektir. Aynı anda çok fazla iş (örneğin 20 API isteği)
çalıştırılınca JS thread kilitlenir ve UI donmaya başlar.
9. Concurrency Hatalarının Sonuçları
• Race Conditions — Aynı veri farklı işlemlerce çakışır.
• Freezes — Uygulama kısa süreli donar.
• JS Thread Block — Tüm UI işlemleri durur.

Mobil cihazlarda CPU, RAM ve enerji sınırlıdır. Bu nedenle concurrency mi-

1. Thread Nedir?
• Thread, CPU’nun çalıştırdığı bağımsız bir yürütme birimidir.
• Mobil cihazlarda genellikle 4–8 çekirdek bulunur; aynı anda sınırlı sayıda

• Queue, işler için bekleme sırasıdır.
• Thread, queue içindeki işleri sırayla alır ve yürütür.
Örnek queue:
[ is1, is2, is3, is4 ]
Queue = işleri tutar, thread = işleri çalıştırır.

• React Native JS Thread ve UI thread bu yapıdadır.

Touch Latency, Input Delay & Gesture Conflicts
Mobil platformlarda her dokunma etkileşimi, kullanıcının parmağı ekrana değdiği
andan UI’nin tepki verdiği ana kadar bir dizi aşamadan geçer. Bu zincire touch
pipeline denir.
Touch Pipeline ˙Işleyişi
1. Touch Down
Kullanıcının parmağı ekrana temas eder. Cihaz bu ham olayı (raw touch
event) işletim sistemine iletir.
2. Hit Test
OS + framework, bu dokunuşun hangi UI elemanına ait olduğunu belirler.
( Örneğin: Button mı? ScrollView mı? Pan gesture mı?)
3. Gesture Recognizer
Dokunuşun bir gesture olup olmadığı anlaşılır: tap, press, pan, swipe,
long press, scroll vb.
4. Responder Chain
Hangi bileşenin olaya yanıt vereceği belirlenir. Eğer birden fazla aday
varsa, çakışma çözümü yapılır (conflict resolution).
5. Render Update
Gesture onaylanır, ilgili UI güncellenir ve ekrana yeniden çizim yapılır.
Bu aşamaların tamamı tek bir kritik hatta bağlıdır: UI Thread. UI thread
bloklandığında tüm dokunma zinciri gecikir (input delay).
Sonuç Olarak Ortaya C ıkan Problemler
• Pan vs Scroll Conflict
Hem yatay hem dikey gesture aynı anda algılanmaya çalıştığında hangi
yönün baskın olacağı belirlenemez. Örneğin: Yatay slider içinde dikey
scroll yapmak.
• Nested Scroll Issues
˙Iç içe scroll yapılmış yapılarda (ör. yatay kart slider + dikey sayfa), hangi
scroll view’ın dokunuşu devralacağı üzerinden çatışma oluşur.
• Press Delay / Input Gecikmesi
UI thread meşgulse:
– buton geç yanıt verir,
– dokunuş algılanmaz gibi görünür,
– uzun basma yanlışlıkla devreye girer,
– kullanıcı “uygulama dondu” zanneder.

Offline-First Conflict Resolution
Mobil uygulamalarda offline-first yaklaşımı, kullanıcının internetsizken de veri
oluşturabilmesini veya veriyi düzenleyebilmesini sağlar. ˙Internet bağlantısı geri
geldiğinde bu lokal değişiklikler sunucuya gönderilir.
Eğer aynı veri hem cihazda hem sunucuda farklı şekillerde değişmişse bir
çatışma (conflict) oluşur. Bu çatışmanın nasıl çözüleceğine conflict resolution
denir.
Aşağıda çatışmayı çözmek için kullanılan başlıca yöntemler sade bir şekilde
açıklanmıştır.
C özüm Stratejileri
• Vector Clocks: Her değişikliğe mantıksal bir zaman etiketi eklenir. Böylece
hangi değişikliğin daha yeni veya daha eski olduğu kolayca anlaşılır. Daha
yeni olan genelde kazanır.
• CRDT (Conflict-Free Replicated Data Types): Aynı verinin bir-
den fazla cihazda farklı şekillerde değişebilmesine izin veren ve internet
geldiğinde bu değişiklikleri otomatik olarak birleştiren veri yapılarıdır.
Google Docs tarzı sistemler bu yöntemi kullanır.
• Last-Write-Wins: En basit kuraldır: En son yapılan değişiklik kazanır.
Bazı durumlarda veri kaybı oluşturabilir ama küçük uygulamalarda çok
yaygındır.
• MVCC (Multi-Version Concurrency Control): Verinin birden fazla
versiyonu tutulur. Local değişiklik eski bir versiyon üzerine yapılmışsa,
sunucudaki yeni versiyonla karşılaştırılır ve çatışma tespit edilir.
• Optimistic Merge: Local değişiklik ile server değişikliğinin çarpışmadığı
varsayılır. Eğer farklı alanlara dokunmuşlarsa otomatik olarak birleştirilir.
• Custom Conflict Rules: Bazı alanlar için özel kurallar tanımlanabilir:
fiyat → server kazansın, açıklama → local kazansın, tarih → daha yeni
tarih kazansın, gibi uygulamaya özel stratejiler.

React Reconciler (JS)
Tanım
React Reconciler, kullanıcı arayüzünün hangi bölümlerinin yeniden oluşturulması
(re-render) gerektiğine karar veren mantık katmanıdır. UI’yı çizmez; yalnızca
hangi değişikliklerin yapılacağına karar verir.
Görevi
Bir component yeniden render edilince:
1. Eski Fiber ağacı (previous tree)
2. Yeni Fiber ağacı (next tree)
karşılaştırılır (diffing). Bu karşılaştırmanın sonucunda şu mutations oluşturulur:
• node ekleme
• node silme
• node güncelleme
• node taşıma
Bu değişiklik listesi, Renderer katmanına iletilir.
Reconciler vs Renderer
React Native’de Reconciler yalnızca karar mekanizmasıdır. Gerçek UI güncellemelerini
yapan ise:
• Fabric Renderer
• veya (eski mimaride) UI Manager
Reconciler:
• Ne değişti?
• Hangi component yeniden render edilmeli?
• Hangi node silinmeli / güncellenmeli / eklenmeli?
Renderer:
• Reconciler’ın bildirdiği değişiklikleri native UI’ya uygular.
• View oluşturur, günceller, layout hesaplatır.
Neden ”Reconcile” Denir?
C ünkü yeni UI ile eski UI uzlaştırılır (reconcile edilir). Amaç: minimum değişiklik
ile UI’yı güncellemek.

JSI (JavaScript Interface)
JSI, React Native’in yeni mimarisinde JavaScript kodu ile C++ katmanı arasında
yüksek performanslı bir köprü sağlayan düşük seviyeli bir arabirimdir. Eski
Bridge mimarisindeki JSON tabanlı mesajlaşmayı ortadan kaldırarak, pointer
düzeyinde doğrudan erişim sunar.
Amaç
JSI’nin temel amacı, JavaScript tarafının:
• C++ objelerine,
• TurboModule fonksiyonlarına,
• Native bileşen durumlarına,
• Hermes (veya JSC) motoru içindeki runtime yapısına
doğrudan, hızlı ve senkron şekilde erişebilmesini sağlamaktır.
Öne C ıkan Özellikler
• Serileştirme gerektirmez: Eski Bridge gibi JSON stringify/parse yok-
tur.
• Senkron çağrılar: Native fonksiyonlar anında cevap verebilir.
• Düşük gecikme: UI-thread ile JS-thread arasında minimum overhead.
• TurboModules desteği: Tüm yeni native modüller JSI üzerinde çalışır.
• HostObjects desteği: C++ objeleri JS tarafında gerçek JS objesi gibi
kullanılabilir.
• Cross-runtime esnekliği: Hermes, JSC ve gelecekteki tüm motorlardan
bağımsız çalışır.
JSI, JavaScript motorunun runtime yapısına erişebilir ve bu runtime üzerinden
C++ fonksiyonlarını JavaScript fonksiyonu olarak export eder.
Bu süreçte:
1. JS tarafı bir fonksiyon çağırır.
2. JSI runtime, bu çağrıyı C++ HostFunction’a yönlendirir.
3. HostFunction native kodu işletir.
4. Sonuç JSI üzerinden JS’e geri döner.

Fabric Renderer Nedir?
Tanım
Fabric Renderer, React Native’in yeni mimarisinde Reconciler tarafından
üretilen UI değişikliklerini (mutations) gerçek Native UI bileşenlerine uygu-
layan modern çizim motorudur. Eski mimarideki UI Manager katmanının
tamamen yerini alır.
Temel Görevleri
Fabric, Reconciler’ın belirlediği UI güncellemelerini şu adımlarla yönetir:
1. Yeni UI node’larını oluşturur (UIView, ViewGroup vb.)
2. Yoga ile layout hesaplamalarını gerçekleştirir
3. Shadow Tree’yi güncelleyip Native Tree ile senkron tutar
4. Stil, prop ve event güncellemelerini native katmana uygular
5. Commit Phase sırasında tüm değişiklikleri ekrana yansıtır
Özellikleri
• Senkron iletişim: JSI üzerinde çalıştığı için Bridge gerekmez.
• Düşük gecikme: JSON serileştirmesi olmadığı için hızlıdır.
• Shadow Tree yönetimi: Hesaplama ve rendering için ayrılmış bir veri
yapısı.
• Yoga destekli layout: Flexbox tabanlı ölçüm ve konumlandırma.
• Concurrent Rendering uyumlu: React 18 ile tam entegredir.
• Platform bağımsız: iOS ve Android için tek tasarım.
Shadow Tree ve Native Tree
Fabric, iki ayrı veri yapısı kullanır:
• Shadow Tree: UI’nın hesaplama ve layout için kullanılan hafif, sanal
temsilidir.
• Native Tree: Platformun gerçek UI node’larıdır (UIView, CALayer,
ViewGroup vb.).
Fabric bu iki ağacı sürekli olarak senkron tutar.

Eski Mimari ile Karşılaştırma
• Bridge + UIManager → asenkron, JSON tabanlı, yavaştı
• Fabric → senkron, pointer-level, yüksek performanslı
• Bridge batching yapardı; Fabric anında commit edebilir.
• Eski mimari jank yaratırdı; Fabric low-latency rendering sağlar.
Sonuç
Fabric Renderer, React Native’in modern mimarisinin merkezinde yer alır. JSI
üzerinden yüksek hızlı etkileşim, Yoga tabanlı layout, Shadow Tree yönetimi ve
concurrent rendering desteği ile, önceki nesil UI Manager’ı tamamen geçersiz
kılarak çok daha performanslı bir UI oluşturma süreci sağlar.

Hybrid Thread Models (React Native)
Modern React Native (Fabric + TurboModules) mimarisi, eski köprü (bridge)
modelinden farklı olarak çoklu thread üzerinde çalışan hibrit bir yapı kul-
lanır. Bu yapı, yanlış thread üzerinde yapılan state güncellemeleri veya işlemler
sonucunda jank, race condition veya crash oluşturabilir.
React Native’deki Temel Thread’ler
• JS Thread — Uygulamanın JavaScript mantığının çalıştığı alan.
• UI Thread — Native UI güncellemelerinin işlendiği ana thread.
• Shadow Tree Thread — Fabric layout ve ölçüm hesaplamalarının yürütüldüğü
thread.
• Native Modules Threads — Klasik native modüllerin çalışabileceği
ayrı background thread’ler.
• TurboModules Threads — JSI tabanlı TurboModules için concurrent
native thread’ler.
• JSI Background Threads — C++ tabanlı JSI işlemleri için kullanılan
ek background worker thread’ler.
Bu çoklu thread yapısı performans artışı sağlasa da eşzamanlılık (concur-
rency) sorunlarını önemli ölçüde arttırır.
Neden Kritik?
Eski mimaride yalnızca iki ana thread vardı:
• JS Thread
• UI Thread
Yeni mimaride:
• Layout hesaplamaları ayrı bir thread’de,
• Native modüller farklı queue’larda,
• TurboModules senkron/async çalışabilir,
• JSI pointer tabanlı memory modeli kullanır,
Bu nedenle thread güvenliği çok daha kritiktir. Yanlış thread’de işlem yap-
mak race condition, invalid memory access ve segfault gibi hatalara yol açabilir.

GPU Render Pass Optimization
Mobil GPU’lar (özellikle PowerVR, Adreno, Mali) tile-based deferred ren-
dering (TBDR) mimarisini kullanır. Bu mimaride ekran, küçük tiles (karo)
bloklarına bölünür ve her tile bağımsız olarak işlenir. Her tile için GPU birden
fazla “render pass” yürütür. Bu pass’lerin her biri ek maliyet demektir.
Maliyeti Artıran Render ˙Işlemleri
• Shadows (Gölge Hesaplamaları): Her gölge, genellikle Gauss blur
kernel’i ile hesaplanır. Bu işlem tile başına onlarca ek örnekleme (samples)
gerektirir. Mobil GPU’larda bu işlem, toplam render süresini katlayabilir.
• Nested Views (˙Iç ˙Içe Yapılar): Her nested view, potansiyel:
– ek clipping,
– ek compositing,
– ek overdraw
yaratır. Overdraw, aynı pikselin birden fazla kez çizilmesidir.
• Animated Clipping: Border radius, mask path veya clip rect animasy-
onla değişiyorsa, GPU her frame bunu yeniden hesaplamak zorundadır.
Bu, özel bir “offscreen pass” oluşturur ve oldukça pahalıdır.
Neden Bu ˙Işlemler Pahalıdır?
• Render pass sayısı arttıkça tile başına işlenen fragment sayısı artar.
• Blending, gölgeler ve clip işlemleri, GPU’nun fill-rate limitlerine çarpar.
• Her ek pass, hafıza bant genişliğini (memory bandwidth) tüketir.
• Mobil GPU’lar CPU kadar güçlü değildir; özellikle fragment shader yükü
arttığında “thermal throttling” devreye girer.
Sonuç: Performans Kaybı
Bu maliyetlerin birikimi şunlara yol açar:
• frame drop (FPS düşüşü),
• jank (UI takılması),
• yüksek GPU tüketimi → batarya tüketimi,
• termal throttling → performansın sürekli düşmesi,
• gecikmeli animasyonlar ve input lag.

˙Işlemci Mimarileri, Sistem ˙Imajları ve Sanallaştırma
Emülatör performansını ve stabilitesini belirleyen en kritik teknik bileşenlerden
biri, is, lemci mimarisi ve kullanılan sistem imajı (system image) türüdür.
Bu bölümde:
• geliştirici makinesinin işlemcisi (Intel/AMD/Apple Silicon),
• Android emülatör sistem imajlarının mimarileri (x86, x86 64, ARM),
• sanallaştırma katmanı (virtualization layer)
arasındaki ilişkiyi kavramsal olarak açıklıyoruz.
Host Makine ˙Işlemcisi (Intel, AMD, Apple Silicon vb.)
Geliştirici bilgisayarınızın markası (MSI, Lenovo, Dell vb.) teknik olarak ikincil

• Intel x86 64 veya AMD x86 64 işlemciler (masaüstü / laptopların

• ARM tabanlı işlemciler (özellikle Apple Silicon – M1, M2, M3).

• Intel/AMD bir makinede, x86/x86 64 tabanlı sistem imajları genellikle en

• ARM tabanlı bir makinede (örneğin Apple Silicon), ARM veya ARM

oluşturabilir.

iki önemli boyuta sahiptir:
1. API seviyesi (Android 12, 13, 14 vb.).
2. Mimari türü (ABI):
• x86
• x86 64
• armeabi-v7a (32-bit ARM)
• arm64-v8a (64-bit ARM)

x86 / x86 64 ˙Imajlar
• Intel/AMD masaüstü/laptop işlemcileri ile doğal olarak uyumludur.
• Donanım hızlandırması (hardware acceleration) ile birlikte çok daha yüksek
performans sunar.
• Geliştirme ortamı Windows veya klasik x86 64 tabanlı macOS ise, günlük
geliştirme için ilk tercih genellikle x86 64 imajlardır.
ARM Tabanlı ˙Imajlar
• Gerçek Android cihazların çoğu ARM tabanlı olduğu için, ülatör bu mi-
mariyi emüle edebilir.
• x86 bir host makinede ARM imajı çalıştırmak, ek emülasyon katmanı
gerektirdiği için belirgin performans kaybına neden olur.
• Apple Silicon gibi ARM tabanlı host sistemlerde ise, ARM imajlar doğal
olarak daha verimlidir.
Geliştirici açısından özet kural:
• Host işlemci x86 64 ise ⇒ mümkün olduğu kadar x86 64 system image
kullan.
• Host işlemci ARM (örneğin Apple Silicon) ise ⇒ ARM tabanlı imajlar
daha doğal ve genellikle daha hızlıdır.
Sanallaştırma Katmanı (Virtualization Layer)
Android emülatör, sistem imajını çalıştırmak için işletim sisteminin sunduğu
sanallaştırma özelliklerini kullanır. Bu katman:
• BIOS/UEFI seviyesinde donanım sanallaştırmasının (Intel VT-x, AMD-
V) açık olmasına bağlıdır.
• Windows tarafında Hyper-V, WSL2, WHPX gibi hypervisor’lar ile etk-
ileşim içindedir.
Pratik Etkiler
• Eğer BIOS/UEFI’de sanallaştırma kapalıysa, emülatör ya hiç başlamaz
ya da son derece yavaş çalışır.
• Bazı hypervisor’lar (Hyper-V, WSL2 vb.) aktifken, eski tip hızlandırma
sürücüleri (ör. HAXM) ile çakışma oluşabilir.
• Emülatörle ilgili “aşırı yavaş”, “takılıyor” veya “başlamıyor” türü sorun-
larda, çoğu zaman kök sebep işlemci gücünden ziyade sanallaştırma
yapılandırmasıdır.

Bilgisayarınızın Markası (MSI, Lenovo vb.) Neden ˙Ikincil Önemdedir?
Laptop veya masaüstünüzün MSI, Lenovo, Dell vb. olması, teknik anlamda

• Altındaki işlemci mimarisi (Intel/AMD x86 64 veya ARM),
• BIOS/UEFI’de sanallaştırmanın açık olup olmadığı,
• ˙Işletim sistemindeki hypervisor yapılandırması (Hyper-V, WSL2 vb.),
• AVD’de seçtiğiniz system image’in bu mimariyle uyumu.
Dolayısıyla “MSI kullanıyorum, emülatör neden yavaş?” sorusunun cevabı

• yanlış seçilmiş system image (ARM imaj + x86 64 host),
• kapalı sanallaştırma,
• aşırı düşük RAM/GPU ayarı
gibi konfigürasyon problemleridir.
Önerilen Minimal Bilgi Seti

1. Kendi makinesinin işlemci mimarisi nedir? (Intel/AMD x86 64 mi, yoksa

2. AVD oluştururken x86 64 imaj seçmek, x86 64 host üzerinde en iyi per-

5. ARM system image’ler genellikle gerçek cihaza yakın mimari sunar, fakat

Native Kütüphaneler ve .so Dosyaları (ABI Uyumunun Önemi)
Android tarafında native kütüphaneler, derlendiğinde genellikle .so (shared
object) uzantılı dosyalar üretir. Bu dosyalar:
• C/C++ (NDK) ile yazılmış kodların derlenmiş hâlidir,
• belirli bir işlemci mimarisi (ABI) için üretilmiştir,
• APK/Bundle içinde lib/<ABI>/libXXX.so şeklinde paketlenir.
React Native projelerinde, birçok üçüncü parti kütüphane (örneğin depo-
lama, şifreleme, görüntü işleme, ses/video işleme kütüphaneleri) arka planda
.so dosyaları kullanır. Geliştirici doğrudan .so ile uğraşmasa da, emülatör ve
cihaz mimarisi ile .so dosyalarının mimarisi uyumlu değilse, uygulama çalışma
zamanında hata verir.
APK ˙Içinde .so Yerleşimi Derlenmiş bir APK içindeki tipik dizin yapısı:
lib/
armeabi-v7a/
libexample.so
arm64-v8a/
libexample.so
x86/
libexample.so
x86_64/
libexample.so
Buradaki her klasör, farklı bir ABI (Application Binary Interface) için der-
lenmiş aynı kütüphanenin versiyonunu barındırır.
• armeabi-v7a → 32-bit ARM
• arm64-v8a → 64-bit ARM
• x86 → 32-bit x86
• x86 64 → 64-bit x86
Emülatör Mimarisinin .so ile ˙Ilişkisi
Android runtime, uygulama başlarken libXXX.so dosyalarını, emülatörün/cihazın
mimarisi ile eşleşen klasörden yüklemeye çalışır:
• Emülatör x86 64 ise, önce lib/x86 64 klasörüne bakar.
• Gerçek bir telefon arm64-v8a ise, lib/arm64-v8a klasörünü kullanır.
Eğer ilgili ABI için .so dosyası yoksa veya yanlış mimari bir dosya yüklenmeye
çalışılıyorsa:

modern mobil uygulama geliştirme sürecini anlaşılır, yapılandırılmış ve bütüncül
bir forma dönüştürmeyi amaçlamıştır. Bileşen yapısı, durum yönetimi, navi-
gasyon, yerel modül entegrasyonu, platform farklılıkları ve dağıtım süreçleri gibi
başlıklar, tek tek “ipucu” düzeyinde değil, birlikte anlamlı bir resim oluşturacak
şekilde ele alınmıştır.
Bu rehberin temel iddiası, iyi bir mobil geliştiricinin yalnızca kütüphane
API’lerini ezbere bilmesi değil; aynı zamanda:
• uygulamanın yaşam döngüsünü,
• JS katmanıyla native katman arasındaki ilişkiyi,
• Android ve iOS ekosistemlerinin temel beklentilerini,
• performans, bakım ve ölçeklenebilirlik kaygılarını
kavramsal düzeyde anlayan bir profesyonel olması gerektiğidir. Bu nedenle
metin, örnek ve kavramları birbirinden kopuk değil, aynı mimari resmin parçaları
olarak sunmuştur.
Pratikte bu dokümandan alınabilecek üç ana çıktı şunlardır:
1. Mobil projeye başlarken hangi kavramları en azından “yeterince” bilmek
gerektiğine dair net bir kontrol listesi.
2. React Native projelerinde karşılaşılan tipik sorunları (build, izinler, na-
tive entegrasyon, performans vb.) okurken çekinmeyecek seviyede temel
platform okuryazarlığı.
3. Uzun vadede derinleşmek istenebilecek alanlar için (native köprüler, mi-
mari desenler, performans optimizasyonu, growth ve ürün odaklı geliştirme
vb.) yön gösteren bir başlangıç noktası.
Sonuç olarak, bu rehberin, mobil geliştirme dünyasına girerken “dağınık ve
parça parça bilgi” hissini azaltan; yerine, nereye bakması gerektiğini bilen, kendi
öğrenme yol haritasını daha bilinçli çizebilen bir geliştirici profiline doğru atılmış
ilk adım olacağı varsayılmaktadır.

Hermes JS engine Meta’nın mobil için optimize ettiği JavaScript
motoru. Android ve iOS’ta React Native
uygulamalarında daha hızlı açılış, daha düşük
bellek kullanımı hedefler; yeni projelerde sıkça
varsayılan motor olarak kullanılır.
JavaScriptCore
(JSC)
JS engine Safari / iOS tarafının yerleşik JS motoru. Her-
mes kullanmazsan iOS’ta RN kodunu genelde
JSC çalıştırır. Eski RN projelerinde varsayılan
motordur.
Nitro JIT derleyici
adı
JSC’nin içindeki JIT derleyicinin ismi (Safari’de
kullanılır). iOS’ta sandbox ve güvenlik kısıtları
nedeniyle tam gücüyle her zaman kullanılamaz;
bu yüzden RN’de JS performansı Safari’den
farklı davranabilir.
Turbo Modules Native modül
mimarisi
Klasik “bridge” modeline göre daha hızlı ve tipi
belli yeni native modül sistemi. JSI üzerinden
doğrudan erişim sağlar; daha az kopyalama
ve daha iyi performans sunar. Üçüncü parti
RN kütüphaneleri yavaş yavaş bu mimariye
geçmektedir.
JSI (JavaScript
Interface)
Köprü katmanı JavaScript motoru (Hermes, JSC vb.) ile C++
dünyasını bağlayan ince bir soyutlama. En-
gine’den bağımsız tek bir arayüz üzerinden na-
tive kodla konuşmayı sağlar. Turbo Modules ve
Fabric bu katman üzerine inşa edilmiştir.
Fabric Yeni UI / ren-
derer sistemi
React Native’in yeni render altyapısı. Shadow
tree ve layout işlemlerini daha modern bir mi-
mariyle yöneterek özellikle büyük listeler ve
karmaşık arayüzlerde daha tutarlı, performanslı
ve deterministik davranmayı amaçlar.
Metro Bundler React Native’in varsayılan JS bundler’ı. Type-
Script/JavaScript dosyalarını tek bir bundle’a
çevirir, import’ları çözer, hot reload / fast re-
fresh sağlar. Web tarafındaki webpack veya vite
gibi düşünülebilir.
Headless JS Arka plan JS
modeli
UI olmadan, arka planda saf JS fonksiyon-
ları çalıştırmaya izin verir. Örneğin back-
ground sync, push bildirim işleme gibi “ekran
kapalıyken” kısa süreli işleri yürütmek için kul-
lanılır.
Flipper Debug aracı RN ve diğer mobil uygulamaları debug etmek
için masaüstü aracı. Network log’larını, lay-
out yapısını, React devtools’u, AsyncStorage
içeriğini görmeni sağlar; Hermes, Fabric vb. için
eklentiler (plugin) sunar.
