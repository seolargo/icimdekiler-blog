Giriş
Mobil uygulama geliştirme, kullanıcı arayüzü akıcılığı, düşük gecikme, enerji
verimliliği ve yüksek kararlılık gerektiren sınırlı bir çalışma ortamında yürütülmektedir.
Bu nedenle Android ve iOS işletim sistemlerinin süreç (process), iş parçacığı
(thread), zamanlama (scheduling) ve eşzamanlılık (concurrency) modellerinin
doğru anlaşılması, bir mobil geliştiricinin performans, stabilite ve kullanıcı deney-
imi açısından sağlıklı kararlar verebilmesi için kritik önem taşır.
Bu dokümanın amacı, mobil geliştirmede sıkça karşılaşılan jank, ANR, frame
drop, UI donması, memory leak veya race condition gibi problemlerin temelinde
yatan işletim sistemi davranışlarını teknik olarak açıklamaktır. Burada ak-
tarılan kavramlar; React Native, Flutter, Swift/Objective-C, Kotlin/Java gibi
tüm modern mobil geliştirme ortamlarında geçerli olup, farklı framework’lerin
thread modellerini anlamayı ve doğru uygulamayı kolaylaştırır.
Dokümanın kapsamı şu başlıklar etrafında şekillendirilmiştir:
• Process ve thread mimarisi
• Bellek yönetimi, heap–stack ayrımı
• Event loop modelleri (JavaScript event loop, Android Looper, iOS Run-
Loop)
• ˙Işletim sistemi zamanlayıcıları ve öncelik seviyeleri
• Asenkron çalışma paradigmaları (Promises, async/await, coroutines)
• Mobilde uzun işlemlerin doğru yönetimi
• Native ve hibrit framework’lerde thread kullanım stratejileri
Bu bilgiler, mobil geliştiricilerin platform davranışlarını daha iyi kavrayarak
uygulama mimarisini bilinçli şekilde tasarlamasını ve performans sorunlarını
kökünden çözebilmesini hedeflemektedir.

User Space ile Kernel Arasındaki Etkileşim: Sis-
tem C ağrıları
Modern işletim sistemlerinde uygulamalar doğrudan donanım kaynaklarına erişim
sağlayamaz. Güvenlik, yalıtım (isolation) ve kararlılık gereksinimleri nedeniyle,
uygulama kodu user space adı verilen sınırlı bir çalışma ortamında yürütülür.
Donanıma veya çekirdek seviyesindeki servis bileşenlerine erişim, yalnızca sys-
tem call (sistem çağrısı) adı verilen kontrollü bir mekanizma üzerinden gerçekleştirilir.
Bu bölümde user space ile kernel space arasındaki etkileşim, adım adım açıklanmaktadır.
System Call Kavramı
Bir system call, kullanıcı modunda çalışan bir sürecin (ör. mobil bir uygula-
manın) işletim sisteminden belirli bir hizmet talep etmesini sağlayan kontrollü
bir geçiş mekanizmasıdır. Dosya açma, soket oluşturma, bellek tahsis etme veya
zaman bilgisini okuma gibi işlemler, doğrudan donanıma erişim gerektirdiğinden
kernel tarafından yürütülür. Dolayısıyla user space uygulamaları bu işlemleri
doğrudan gerçekleştiremez; bunun yerine ilgili system call fonksiyonunu çağırır.
System Call Sürecinin Genel Akışı
User space ile kernel space arasındaki etkileşim aşağıdaki aşamalardan oluşur:
1. Kullanıcı Kodunun C ağrısı: Uygulama, bir kütüphane fonksiyonu
(open(), read(), send() vb.) aracılığıyla bir system call’ı tetikler. Bu
çağrı, user space içinde çalışan standart C kütüphanesi (Android’de bionic
libc, iOS’ta libSystem) tarafından hazırlanır.
2. CPU Mod Değişimi: System call tetiklendiğinde işlemci, özel bir CPU
komutu (ARM mimarisinde svc, x86-64 mimarisinde syscall) aracılığıyla
user mode ortamından kernel mode ortamına geçiş yapar. Böylece kernel
ayrıcalık seviyesine sahip olur ve donanım kaynaklarına erişebilir.
3. Kernel System Call Handler’ının C alışması: ˙Işletim sistemi, system
call numarasına karşılık gelen işleyiciyi (sys open, sys read, sys write
vb.) çağırır. Bu işleyici, kernel alt sistemlerinden ilgili bileşene (örneğin
dosya sistemi yöneticisi, bellek yöneticisi veya ağ yığını) isteği yönlendirir.
4. Alt Sistemlerin ˙Işlemi Gerçekleştirmesi: Kernel alt sistemleri gerekli
işlemleri yürütür. Örneğin bir dosya açma isteği, önce sanal dosya sis-
temi (VFS) katmanından geçer ve ardından ilgili dosya sistemi sürücüsüne
iletilir.
5. Kernel’den User Space’e Dönüş: ˙Işlem tamamlandığında kernel, sonucu
(başarı durum kodu, dosya tanımlayıcısı, hata kodu vb.) user space’e
döndürür ve işlemci tekrar user mode seviyesine geçiş yapar.

Performans ve Gecikme Açısından System Call’lar
System call’lar, user space içinde gerçekleştirilen sıradan fonksiyon çağrılarına
kıyasla daha yüksek maliyetlidir. Bunun temel nedenleri şunlardır:
• CPU mod değişimi (user mode → kernel mode),
• Kernel yığınlarının ve veri yapıların kurulması,
• Gerekli kernel alt sistemlerinin çalıştırılması,
• Donanım sürücülerinin tetiklenmesi ve olası kesmelerin (interrupt) işlenmesi.
Bu nedenle, dosya işlemleri, ağ iletişimi veya yüksek sayıda system call gerek-
tiren işlemler kullanıcı arayüzü (UI) iş parçacığında çalıştırılmamalıdır. Aksi
halde uygulamada gecikmeler, kare düşmeleri ve ANR (Application Not Re-
sponding) gibi sorunlar ortaya çıkabilir.
Güvenlik Açısından System Call Mekanizması
System call’lar aynı zamanda işletim sisteminin güvenlik politikasının uygu-
landığı kritik noktalardır. Kernel, her talebi gerçekleştirmeden önce:
• çağrıyı yapan sürecin kullanıcı kimliğini (UID/GID),
• süreç izinlerini,
• sandbox veya SELinux gibi ek güvenlik politikalarını
denetler. Bu sayede, uygulamaların izinsiz şekilde kamera, mikrofon, ağ
arabirimi veya dosya sistemi gibi kaynaklara erişmesi engellenmiş olur.
Genel Değerlendirme
System call mekanizması, modern işletim sistemlerinin güvenli, yalıtılmış ve
denetimli çalışmasını sağlayan temel yapı taşlarından biridir. Mobil uygulama
geliştirme açısından bu mekanizmanın anlaşılması, performans optimizasyonu,
gecikme analizi, izin yönetimi ve hata ayıklama gibi konularda doğrudan avan-
taj sağlar. Özellikle arayüz iş parçacığının bloklanmasının neden ANR veya
kare düşmesine yol açtığı, bu etkileşim modeli incelendiğinde açık bir biçimde
anlaşılmaktadır.

-- 4 of 111 --

C ekirdek Zamanlayıcısı: ˙Iş Parçacıklarının Plan-
lanması
Bir işletim sisteminin temel sorumluluklarından biri, sınırlı işlemci kaynaklarını
(CPU çekirdekleri) birden fazla süreç ve iş parçacığı arasında etkin biçimde
paylaştırmaktır. Bu görevi yerine getiren bileşen kernel scheduler (çekirdek za-
manlayıcısı) olarak adlandırılır. Scheduler, hangi iş parçacığının ne zaman ve
ne kadar süreyle çalışacağını belirleyerek sistemin yanıt verebilirliğini, perfor-
mansını ve adaletini (doğru kaynak paylaşımı) sağlar.
Zamanlayıcının Temel Görevleri
Bir kernel zamanlayıcısının temel fonksiyonları aşağıdaki şekilde özetlenebilir:
• ˙Iş Parçacığı Seçimi: Her CPU çekirdeğinde çalışmak üzere bir iş parçacığı
(thread) seçer. Bu seçim, öncelik (priority), geçmiş çalışma süresi ve plan-
lama politikasına göre yapılır.
• Zaman Dilimi Tahsisi (Time Slicing): C oklu görev ortamlarında
her iş parçacığına belirli uzunlukta bir zaman dilimi atanır. Bu süre
dolduğunda scheduler devreye girerek başka bir iş parçacığı çalıştırabilir.
• Adaletin Sağlanması (Fairness): Yoğun CPU kullanan iş yüklerinin
tüm sistemi bloke etmesini engeller; düşük öncelikli görevlerin de belirli
aralıklarla çalışmasını garanti eder.
• Yanıt Verebilirlik: Özellikle kullanıcı arayüzü veya gerçek zamanlı iş
yüklerinde, düşük gecikme (low latency) kritik olduğundan scheduler bu
tür iş parçacıklarına daha yüksek öncelik tanıyabilir.
• Bloklanmış ve Hazır Durumların Yönetimi: I/O bekleyen veya kilit
(lock) nedeniyle bloklanan iş parçacıklarının durumunu takip eder ve uy-
gun an geldiğinde tekrar çalıştırılmak üzere hazır kuyruğuna alır.
Zamanlayıcı Kuyrukları ve Durum Geçişleri
Her iş parçacığı, çekirdek tarafından tanımlanan belirli durumlar arasında geçiş
yapar. Zamanlayıcı, bu geçişleri takip ederek uygun anda işlemciyi tahsis eder.
• Running (C alışıyor): ˙Iş parçacığı bir CPU çekirdeğinde aktif olarak
çalışmaktadır.
• Ready (Hazır): C alışmaya hazır durumdadır fakat CPU tahsisi bekle-
mektedir.
• Blocked (Bloklanmış): I/O, kilit veya başka bir dış kaynağı beklemek-
tedir; CPU kullanamaz.

• Waiting (Bekliyor): Zamanlayıcı tarafından belirli bir olayın veya za-
manın gelmesi beklenmektedir.
Scheduler, yalnızca running ve ready durumlarındaki iş parçacıklarını CPU
üzerinde planlar. Bloklanmış iş parçacıkları, gerekli olay gerçekleştiğinde tekrar
hazır kuyruğuna alınır.
Planlama Politikaları
Modern işletim sistemlerinde birden fazla planlama politikası bulunur. Mobil
işletim sistemleri (Android ve iOS), tipik olarak aşağıdaki yaklaşımları benimser:
• Completely Fair Scheduler (CFS): Linux tabanlı Android’de kul-
lanılan varsayılan mekanizmadır. Tüm iş parçacıklarına hesaplanan oranda
CPU süresi vererek adil bir çalışma sağlar. Kırmızı-siyah ağaç (red-black
tree) tabanlı bir veri yapısı kullanır.
• Gerçek Zamanlı (Real-Time) Planlama: Ses işleme (audio), video oy-
natma veya düşük gecikme gerektiren senaryolarda daha yüksek öncelikli,
kesintisiz çalışması gereken iş parçacıkları için kullanılır.
• Background/Foreground Ayırımı: Mobil ortamda kullanıcı arayüzünü
temsil eden iş parçacıkları (foreground) düşük gecikme için öncelikli değerlendirilirken,
arka plan görevleri (background) düşük öncelik alarak batarya ve perfor-
mans dengesini korur.
Mobil Uygulamalar Açısından Önemi
Scheduler davranışının anlaşılması, mobil uygulama geliştirmede birçok perfor-
mans sorununu açıklığa kavuşturur:
• UI Thread Bloklanması: Kullanıcı arayüzü iş parçacığı diğer CPU
yoğun görevler nedeniyle planlanamazsa, uygulama gecikme, donma veya
ANR hatası üretebilir.
• Yoğun CPU Kullanımı: Arka planda yüksek hesaplama yapan iş parçacıkları,
foreground görevlerinin zamanında çalışmasını engelleyebilir.
• Enerji Tüketimi: Sürekli uyanan ve CPU talep eden iş parçacıkları
scheduler tarafından sıkça çalıştırılır; bu durum batarya tüketimini artırır.
• C ok C ekirdekli ˙Işlemcilerde Dağıtım: Kernel, iş parçacıklarını birden
fazla çekirdeğe dağıtarak paralel yürütme sağlar; bu, performansı belirgin
şekilde etkiler.

Process ve Thread Temelleri
Mobil işletim sistemlerinde bir uygulama, process adı verilen yalıtılmış bir çalışma
ortamında yürütülür. Her process, kendisine ait adres alanına, dosya tanımlayıcılarına
ve işletim sistemi tarafından atanan kimlik bilgilerine sahiptir. Bir process
içerisinde bir veya birden fazla thread (iş parçacığı) bulunabilir. Thread’ler
aynı adres alanını paylaşır, bu nedenle veri paylaşımı hızlı olmakla birlikte yarış
durumları (race condition) ve senkronizasyon hataları oluşabilir.
UI / Main Thread
Mobil işletim sistemlerinde kullanıcı arayüzü işlemleri özel bir iş parçacığında
yürütülür; bu iş parçacığı UI thread veya main thread olarak adlandırılır. Ekranın
çizilmesi, dokunma olaylarının işlenmesi ve arayüz güncellemeleri yalnızca bu
thread üzerinden gerçekleşebilir.
Neden Önemlidir?
UI thread’in bloklanması, kare düşüşü (jank), donma (freeze) ve Android’de
ANR (Application Not Responding) hatalarına yol açabilir. Ağ istekleri, dosya
işlemleri veya ağır hesaplamalar gibi zaman alan işlemler UI thread’de yürütüldüğünde
kullanıcı deneyimi doğrudan olumsuz etkilenir.
Mobil Geliştirici ˙Için C ıkarımlar
• Ağ istekleri, disk erişimi ve büyük veri işleme gibi pahalı operasyonlar her
zaman arka plan thread’lerinde yürütülmelidir.
• UI güncellemeleri yalnızca main thread üzerinden yapılmalıdır.
• Uzun süren işlemleri main thread’e taşımamak için asenkron yapılar (An-
droid’de coroutines, iOS’ta GCD ve async/await, React Native’de asenkron
JS API’leri) kullanılmalıdır.
• Aynı hafıza alanını paylaşan thread’ler arasında senkronizasyon mekaniz-
malarına dikkat edilmelidir.
Sonuç
Process ve thread kavramlarının anlaşılması, mobil uygulamalarda performans
sorunlarının kök nedenlerini görmeyi mümkün kılar. Özellikle UI thread’in blok-
lanmaması, akıcı ve kullanıcı dostu bir deneyim sağlamak açısından kritik öneme
sahiptir.

Bellek Yönetimi (Memory Management)
Mobil cihazlarda bellek miktarı sınırlıdır ve uygulamalar belirli bir RAM tah-
sis alanı içinde çalışmak zorundadır. ˙Işletim sistemi, her uygulamaya ayrı bir
sanal bellek alanı sağlar; ancak bu alan fiziksel bellek üzerinde paylaşımlı olarak
yönetilir. Uygulamalar, büyük veri yapıları, yüksek çözünürlüklü görseller veya
gereksiz referanslar nedeniyle bellek kullanımını kontrol edemediklerinde sis-
tem performans kaybı yaşayabilir veya uygulama işletim sistemi tarafından son-
landırılabilir.
Heap ve Stack Ayrımı
Uygulama belleği, çalışma zamanı davranışını etkileyen iki temel bölgeye ayrılır:
stack ve heap. Bu iki bellek alanı farklı amaçlarla kullanılır, farklı perfor-
mans özelliklerine sahiptir ve mobil uygulama geliştirme açısından farklı riskler
doğurur.
Stack Belleği
Stack, fonksiyon çağrılarının sırayla yığıldığı (push/pop) hızlı bir bellek bölgesidir.
Aşağıdaki özelliklere sahiptir:
• Fonksiyon çağrıları sırasında otomatik olarak büyür ve fonksiyon tamam-
landığında otomatik olarak küçülür.
• Yerel (local) değişkenler ve fonksiyon parametreleri bu alanda saklanır.
• Bellek tahsisi ve serbest bırakılması işletim sistemi veya çalışma zamanı
tarafından otomatik yönetildiğinden çok hızlıdır.
• Boyutu sınırlıdır; büyük veri yapılarının burada saklanması stack overflow
hatasına yol açabilir.
Örnek (Basit):
function sum(a, b) {
let c = a + b; // ’c’ değişkeni stack’te saklanır
return c;
}
Örnek (Mobil Geliştirici ˙Için): React Native veya Android Kotlin’de küçük
karşılaştırma değişkenleri, sayısal değerler ve fonksiyon parametreleri stack üzerinde
tutulur:
// Kotlin örneği
fun calculateDistance(x: Int, y: Int): Int {
val diff = x - y // ’diff’ stack üzerinde
return diff
}

Heap Belleği
Heap, çalışma zamanında dinamik olarak oluşturulan nesnelerin saklandığı es-
nek bir bellek bölgesidir. Aşağıdaki özelliklere sahiptir:
• Boyutu stack’e göre çok daha büyük fakat erişim süresi daha yavaştır.
• Nesneler new, Object(), class örnekleri gibi dinamik oluşturma işlemleriyle
burada tutulur.
• Bellek serbest bırakılması otomatik değildir; Android’de Garbage Collec-
tor (GC), iOS’ta ise ARC (Automatic Reference Counting) tarafından
yönetilir.
• Yanlış referans yönetimi, memory leak oluşumuna neden olabilir.
Örnek (Basit):
let user = {
name: "Omer",
age: 28
};
// ’user’ nesnesi heap üzerinde saklanır
Örnek (Mobil Geliştirici ˙Için — React Native):
const [items, setItems] = useState([]);
// ’items’ dizisi heap’tedir
// doğru yönetilmezse bileşen kapandıktan sonra bile RAM’de kalabilir
Örnek (Android Kotlin):
data class User(val name: String, val age: Int)
val currentUser = User("Omer", 28)
// currentUser nesnesi heap üzerinde tutulur
Heap ve Stack’in Karşılaştırılması
• Stack hızlıdır, çünkü tahsis ve serbest bırakma işlemleri deterministiktir
(fonksiyon çağrı sırasına göre).
• Heap daha yavaştır, çünkü bellek yönetimi karmaşıktır ve GC/ARC
gibi mekanizmalar devreye girer.
• Stack kısa ömürlü veriler içindir, heap uzun ömürlü nesneleri saklar.
• Heap aşırı büyürse uygulama Out of Memory ile kapanabilir.

Mobil Geliştirici ˙Için Pratik Sonuçlar
• Büyük veri yapıları (Array, Map, görseller) heap’te tutulduğu için perfor-
mansı etkiler.
• Gereksiz referanslar temizlenmezse heap büyür ve GC baskısı oluşur, bu
da jank (kare düşmesi) yaratır.
• UI thread üzerinde heap tahsisi yapmak (özellikle büyük nesneleri oluşturmak)
kısa süreli donmalara yol açabilir.
• Stack üzerinde büyük veri yapıları oluşturulamaz; bu nedenle büyük res-
imler, JSON’lar veya diziler her zaman heap üzerinde yer alır.
Sonuç
Stack ve heap ayrımını anlamak, mobil uygulamaların bellek kullanımını doğru
analiz etmek için gereklidir. Yanlış yönetilen heap nesneleri performans düşüşü,
bellek sızıntısı ve uygulama çökmesine neden olabilirken, stack’in sınırlı yapısı
büyük veri yapılarının burada tutulmasının imkânsız olduğunu gösterir. Bu
ayrımın bilinmesi, performans odaklı mobil gelişimde temel bir gerekliliktir.
Memory Leak Kavramı
Memory leak, uygulamanın artık ihtiyaç duymadığı nesnelere referans tutmaya
devam etmesi sonucunda, bu nesnelerin çöp toplayıcı (GC) veya otomatik refer-
ans sayımı (ARC) tarafından serbest bırakılmaması anlamına gelir. Leak birikimi,
uygulamanın RAM kullanımının sürekli artmasına ve nihayetinde işletim sistem-
inin uygulamayı sonlandırmasına neden olabilir.
Out of Memory (OOM) Durumu
Bir uygulama, işletim sistemi tarafından belirlenen bellek sınırını aşarsa Out
of Memory (OOM) hatası oluşur. Android ve iOS, düşük bellek durumunda
uygulamayı arka planda veya ön planda sonlandırabilir. Bu durum genellikle
büyük boyutlu resimler, sıkıştırılmamış veri yapıları veya kontrolsüz caching
stratejileri sonucunda ortaya çıkar.

Arka Plan C alışma Kısıtlamaları (Background Lim-
itations)
Mobil işletim sistemleri enerji tüketimini, güvenliği ve kullanıcı deneyimini opti-
mize etmek amacıyla uygulamaların arka planda çalışma yetkilerini sınırlandırır.
Bu kısıtlamalar, uygulamaların kontrol dışı şekilde sistem kaynaklarını tüketmesini
engeller. Android ve iOS, arka plan çalışma politikalarını farklı mekanizmalarla
uygular; ancak temel amaç her iki platformda da aynıdır: pil tasarrufu, perfor-
mans dengesi ve kullanıcı kontrolü.
Arka Plan C alışmanın Türleri
Bir uygulamanın arka planda çalışma ihtiyacı genel olarak üç kategoriye ayrılır:
• Zamanlanmış görevler: Belirli aralıklarla veri senkronizasyonu veya
bakım işlemleri.
• Sürekli çalışan görevler: Müzik çalma, konum takibi, sağlık ölçer vb.
• Kısa süreli arka plan işler: Bir ekran kapandıktan hemen sonra yürütülen
hızlı işlemler, örneğin küçük bir dosya yükleme.
Her mobil işletim sistemi bu kategorilere farklı seviyelerde izin verir.
Android’de Arka Plan Kısıtlamaları
Android işletim sistemi, özellikle sürüm 6.0 (Marshmallow) sonrasında pil ver-
imliliğini artırmak amacıyla arka planda çalışan uygulamalara yönelik kapsamlı
kısıtlamalar getirmiştir. Bu kısıtlamalar, geliştiricilerin arka plan işlerini doğru
planlama araçlarıyla yürütmesini zorunlu kılar. Bu bölüm Android’in temel arka
plan mekanizmalarını, davranışlarını ve uygulama geliştirme açısından sonuçlarını
detaylandırmaktadır.
Doze Mode
Android 6.0 ile tanıtılan Doze Mode, cihaz uzun süre hareketsiz durumdayken
enerji tüketimini azaltmak için arka plan aktivitelerini kısıtlar. Cihaz belirli
koşullarda Doze durumuna girer:
• Ekran kapalı,
• Cihaz uzun süre sabit bir yüzeyde hareketsiz,
• Sarj edilmemekte,
• Kullanıcı etkileşimi bulunmamakta.
Doze modunda aşağıdaki davranışlar geçerlidir:

• Zamanlayıcılar (AlarmManager) ertelenir.
• Ağ erişimi büyük ölçüde engellenir.
• Arka plan servisleri çalıştırılamaz.
• Uygulamalar yalnızca maintenance window denilen kısa aralıklarda sınırlı
işlem yapabilir.

• Ağ erişimi geciktirilir veya tamamen engellenir.

• Sağlık/fitness sensör takibi,

Android’de Zamanlanmış Bildirimlerde Gecikme
Problemi
Android işletim sisteminde zamanlanmış bildirimlerin veya alarmların belirlenen
saatten gecikmeli olarak tetiklenmesi sık görülen bir durumdur. Özellikle No-
tifee gibi üçüncü taraf kütüphaneler aracılığıyla oluşturulan Timestamp Trigger
tabanlı bildirimlerin yaklaşık 10–30 dakika öteye kayması, işletim sisteminin güç
tasarrufu mekanizmalarından kaynaklanmaktadır. Bu bölüm, gecikmenin temel
nedenlerini ve çözüm yöntemlerini açıklamaktadır.
Gecikmenin Temel Nedenleri
Zamanlanmış bir bildirimin bir gün sonra veya belirli aralıklarla yaklaşık 15
dakika ileri kayması genellikle aşağıdaki işletim sistemi davranışlarından kay-
naklanır.
1. Doze Mode Sebebiyle Exact Alarmın Ertelenmesi
Android 6.0 ile tanıtılan Doze Mode, cihaz uzun süre hareketsiz kaldığında za-
manlayıcıları, ağ erişimini ve arka plan yürütmesini sınırlayan bir güç tasarrufu
mekanizmasıdır. Doze sırasında:
• Sadece setExactAndAllowWhileIdle tipi alarmlar çalıştırılır.
• Normal alarmlar inexact hale getirilerek uygun bir bakım penceresinde
(maintenance window) tetiklenir.
• Bu bakım pencereleri genellikle 10–30 dakika aralıklarla gerçekleşir.
Bu nedenle exact olarak ayarlanamayan alarmlar yaklaşık 15 dakika gecikme
ile tetiklenebilir.
2. ˙Inexact Alarm Kullanımı
Bazı cihazlarda Notifee veya AlarmManager, üretici (OEM) tarafından uygu-
lanan güç optimizasyonu nedeniyle setExact komutunu setInexact biçimine dönüştürür.
Bu durumda Android şu politikayı uygular:
“Belirtilen zaman yaklaşık bir hedeftir; alarm uygun sistem duru-
munda çalıştırılacaktır.”
˙Inexact alarmlar sistem tarafından tipik olarak 5–20 dakika ileriye kaydırılır.
3. Repeating (Tekrarlayan) Alarmlar Android Tarafından Kaydırılır
Android 6.0 sonrası:
• setRepeating() veya benzeri mekanizmalar exact değildir,

-- 28 of 111 --

• Sistem bu alarmları enerji tasarrufu için düzenli biçimde geciktirir,
• Tipik kayma miktarı 10–15 dakikadır.
Bu nedenle günlük ilaç hatırlatmaları için tekrarlayan alarm kullanmak önerilmez.
4. OEM ( Üretici) Pil Optimizasyonları
Bazı üreticiler (Samsung, Xiaomi, Oppo, Vivo) alarm davranışını ek güç koruma
katmanları ile değiştirir. Bu işletim sistemi davranışları:
• exact alarmları inexact hale getirebilir,
• uygulamanın zamanlanmış tetikleyicilerini geciktirebilir,
• bakım pencerelerini daha seyrek hale getirebilir.
Beklenen gecikme süresi çoğu cihazda 10–30 dakika arasıdır.

Zamanlanmış bildirimlerdeki kaymayı azaltmak veya ortadan kaldırmak için
aşağıdaki yöntemler kullanılmalıdır:
1. Exact Alarm Kullanımı
Notifee üzerinde exact tetikleme için aşağıdaki yapılandırma önerilir:

• her tetikleme sonrasında bir sonraki alarmın zamanı manuel olarak yeniden
hesaplanmalı,
• yeni bir exact alarm planlanmalıdır.

Ağ ˙Işlemleri ve Sistem C ağrılarının Temel Mantığı
Mobil işletim sistemlerinde ağ iletişimi, doğrudan kullanıcı arayüzü iş parçacığını
etkileyebilecek yüksek gecikmeli işlemler arasındadır. Bir ağ isteği, uygulama
kodundan çekirdek seviyesindeki system call arabirimine, oradan modem sürücüsüne
ve daha sonra mobil şebeke veya Wi-Fi donanımına iletilir. Bu katmanlı yapı
gereği ağ işlemleri yüksek gecikme ve belirsizlik içerir.
Neden Ağ ˙Işlemleri UI Thread’de Yapılmamalıdır?
Ağ isteğinin tamamlanması, bağlantı kalitesi, modem durumu, DNS çözümleme
süresi, yavaş sunucu yanıtı veya paket kayıpları gibi dış faktörlere bağlıdır. Bu
nedenle ağ bekleme süresi milisaniyeler değil, çoğu zaman yüzlerce milisaniye
hatta saniyeler seviyesinde olabilir.
UI thread üzerinde ağ beklemesi yapılması:
• C erçeve çizimlerini (frame rendering) durdurur,
• Dokunmatik olaylarının işlenmesini engeller,
• ANR (Application Not Responding) hatasına yol açabilir.
Örnek: Hatalı Kullanım
const res = await fetch("https://api.example.com/data");
// Bu kod UI olay döngüsünü bloke eder ve donmaya neden olabilir.
Örnek: Doğru Kullanım
// React Native örneği | asenkron çalışma
useEffect(() => {
(async () => {
const res = await fetch(...);
})();
}, []);
Sonuç
Ağ işlemleri yüksek gecikmeli ve dış faktörlere bağlı olduğu için her zaman arka
plan iş parçacığında veya asenkron mekanizmalarla yürütülmelidir. UI thread
üzerinde kesinlikle yapılmamalıdır.

Ağ ˙Isteklerinin ˙Işlem Boru Hattı
Bir fetch() çağrısı veya URLRequest işlemi aşağıdaki katmanlardan geçer:
1. Uygulama Seviyesi (User Space) React Native, Kotlin, Swift veya
Flutter kodu ağ isteğini oluşturur.
2. Sistem C ağrıları (System Calls) Uygulama, send(), connect(), recv()
gibi POSIX tabanlı çekirdek çağrılarını (syscall) tetikler.
3. Kernel Networking Stack Linux çekirdeği (Android) veya XNU çekirdeği
(iOS) TCP/IP protokollerini, soket yönetimini, paket yönlendirmeyi, güvenlik
politikalarını ve buffer yönetimini yürütür.
4. Sürücüler (Drivers) Kernel, modem veya Wi-Fi sürücüsüne talimat
gönderir.
5. Donanım (Modem / Wi-Fi / Radio) Paket fiziksel olarak gönderilir
veya alınır.
Bu zincirin her bir adımı gecikmeye açıktır ve geliştiricinin doğrudan kon-
trolünün dışındadır.
Ağ Gecikmesine Neden Olan Faktörler
Ağ isteğinin tamamlanma süresi sabit değildir. Aşağıdaki faktörler gecikmenin
temel sebepleridir:
• DNS çözümleme Alan adının IP adresine çevrilmesi 5–150 ms arasında
sürebilir.
• TCP Üçlü El Sıkışması (Handshake) SYN → SYN/ACK → ACK sırası
30–300 ms arası gecikme yaratabilir.
• TLS kurulumu HTTPS bağlantıları için ek 1–2 round-trip gerekir; mobil
ağda gecikme ciddi ölçüde artabilir.
• Paket kaybı ve yeniden iletim Zayıf sinyalde paket kaybı ve gecikme
katlanır.
• Radyo durumları (RRC State Machine) Mobil cihazlarda modem
genellikle düşük güç modundadır (Idle). Bir paket göndermek için modem
önce:
1. Idle → Connected geçişi yapar (50–200 ms),
2. ardından ağ isteğini iletir.
• Operatör gecikmesi 4G/5G altyapısı, baz istasyonu yükü ve mobil çekim
gücü gecikmeyi etkiler.
Bu nedenle ağ isteğinin toplam gecikmesi genellikle 100–200 ms civarında
değil, çoğu zaman 300–800 ms seviyesindedir.

UI thread, çerçeve çizimi (frame rendering), animasyon yürütme, dokunma olay-

• kullanıcının dokunma olaylarına yanıt verilememesi,
• animasyonların durması,
• uygulamanın donmuş gibi görünmesi,

Ek Ağ ve ˙Işletim Sistemi Kavramları (Kısa Teknik
Özet)
Bu bölüm, mobil geliştiricilerin ağ gecikmeleri, modem davranışı ve iş parçacığı
yönetimi konularında bilmesi gereken ileri düzey kavramları kısa fakat teknik
biçimde özetlemektedir.
TCP/IP Üçlü El Sıkışması (Three-Way Handshake)
Bir TCP bağlantısı kurulmadan önce istemci ile sunucu arasında aşağıdaki sıralı
paket alışverişi gerçekleşir:
Client Server
------ ------
| |
| ---------- SYN (seq = x) ----------> |
| |
| <----- SYN + ACK (seq = y, ack = x+1) |
| |
| ---------- ACK (ack = y+1) --------> |
| |
Connection Established
Bu işlem en az bir tam RTT (Round Trip Time) gecikmesi gerektirir. Mobil
ağlarda RTT yüksek olduğundan bağlantı kurulum gecikmesi UI performansını
dolaylı olarak etkileyebilir.
Mobil Radyo Güç Durumları (RRC State Machine)
Mobil modemler enerji tasarrufu amacıyla farklı güç durumları arasında geçiş
yapar. Temel iki durum:
• Idle: Modem düşük güç modundadır; veri aktarımı yoktur. Bu durumdan
çıkıp bağlantı kurmak 50–200 ms gecikme yaratır.
• Connected: Modem tam güçte çalışır; veri aktarımı hızlıdır ancak pil
tüketimi yüksektir.
RRC durum geçişleri, ağ isteğinin başlangıç gecikmesine katkıda bulunur.

TCP, IP, UDP, HTTP ve HTTPS Kavramlarının
Temel Tanımları
Bu bölümde internet iletişimini oluşturan temel protokoller kısa ve teknik bir
biçimde özetlenmektedir. Mobil geliştiriciler için ağ iletişiminin hangi katmanda
nasıl gerçekleştiğini anlamak kritik öneme sahiptir.
IP (Internet Protocol)
IP, internet üzerinde paketlerin hangi hedefe gönderileceğini belirleyen adresleme
protokolüdür. Temel işlevi yönlendirme (routing) ve hedefe ulaşım garantisi ol-
madan paket iletimidir.
• Bağlantısızdır; gönderilen paketler kaybolabilir.
• Paketlerin sırası karışabilir.
• Yalnızca “adresleme” ve “iletim” görevini üstlenir.
TCP (Transmission Control Protocol)
TCP, IP’nin üzerinde çalışan güvenilir bir taşıma protokolüdür. Paketlerin
doğru sırayla, eksiksiz ve güvenli şekilde karşı tarafa ulaşmasını sağlar.
• Bağlantı kurulumu için Three-Way Handshake kullanır.
• Paket kaybı durumunda yeniden iletim yapar.
• Veri akış kontrolü ve sıralama mekanizmaları içerir.
• API çağrılarının ve web isteklerinin temel taşıyıcısıdır.
UDP (User Datagram Protocol)
UDP, TCP’ye göre daha hızlı ancak güvensiz bir taşıma protokolüdür.
• Bağlantı kurulmaz; “en hızlı gönder ve geç” yaklaşımı.
• Paket kaybı veya sıra bozulması önlenmez.
• Oyunlar, canlı yayın, VoIP gibi düşük gecikme gerektiren alanlarda kul-
lanılır.
35

-- 35 of 111 --

HTTP (Hypertext Transfer Protocol)
HTTP, istemci ve sunucu arasındaki iletişimi tanımlayan üst seviye protokoldür.
Web sayfaları, REST API çağrıları ve birçok mobil hizmet HTTP üzerinden
çalışır.
• GET, POST, PUT, DELETE gibi metodlar içerir.
• Header, body, status code yapıları tanımlıdır.
• C oğunlukla TCP üzerinde çalışır.
• Güvenlik sağlamaz; iletilen veri düz metin olabilir.
HTTPS (HTTP + TLS)
HTTPS, HTTP protokolünün TLS (Transport Layer Security) ile şifrelenmiş
halidir.
• Veri şifrelenir (gizlilik).
• Veri bütünlüğü sağlanır (tamper protection).
• Sunucunun kimliği doğrulanır (authentication).
• Modern mobil uygulamalarda zorunlu olarak kullanılır.

Uygulama (fetch, axios, URLRequest)

HTTPS (şifreleme + HTTP)

↓
TCP (güvenilir taşıma)
↓
IP (adresleme ve yönlendirme)

Bu katmanların anlaşılması mobil ağ performansını optimize etmek için temel
gerekliliktir.

˙Iş Parçacıkları ve Eşzamanlılık (Threading ve Con-
currency) Temelleri
Modern mobil işletim sistemleri (Android ve iOS), uygulama iş yükünü eşzamanlı
olarak yürütebilmek için çok iş parçacıklı bir model kullanır. Kullanıcı arayüzü
(UI) iş parçacığı çizim, animasyon ve dokunma olayları gibi zaman duyarlı
görevlerden sorumludur. Bu nedenle ağır işlemlerin UI thread üzerinde çalıştırılması,
uygulamanın donmasına, gecikmelere ve ANR (Application Not Responding)
hatalarına yol açabilir.
˙Iş Parçacığı Türleri
Mobil platformlarda yaygın olarak kullanılan thread türleri şöyledir:
• UI Thread (Main Thread): C izim, layout hesaplamaları, animasyonlar
ve dokunma olaylarının işlendiği ana iş parçacığıdır. Her kare (frame)
yaklaşık 16.67 ms içinde işlenmelidir.
• Background Threads: Ağ istekleri, disk IO, JSON çözümleme veya
yoğun hesaplama gibi zaman alan işlemler için ayrılmış iş parçacıklarıdır.
• Scheduled / Worker Threads: Zamanlanmış görevler, uzun süreli işler
ve sistem tarafından yönetilen arka plan operasyonları için kullanılır.
Android’de bu yapı Looper ve MessageQueue üzerinden yönetilir; iOS’ta ise
Grand Central Dispatch (GCD) ve Dispatch Queues kullanılır.
Neden Arka Plan Thread’leri Gereklidir?
Aşağıdaki işlemler yüksek gecikme (latency) veya yüksek CPU kullanımına sahip

C ok iş parçacıklı sistemlerde veri yarışları ve kilitlenme durumları kritik hata

Thread-Safe Olmayan ˙Işlemlerin Belirlenmesi
Bir işlemin thread-safe olup olmadığını anlamak için şu kriterler göz önünde
bulundurulur:
• Paylaşılan Durum (Shared State): Birden fazla thread tarafından
erişilen değişkenler veya veri yapıları senkronizasyon olmadan güncelleniyorsa
işlem thread-safe değildir.
• Atomic Olmayan ˙Işlemler: Birden fazla adım içeren okuma–yazma
operasyonları bölünemez (atomic) değilse race condition oluşabilir.
• Global veya Statik Kaynak Kullanımı: Statik koleksiyonlar, sayaçlar,
cache yapıları veya singleton nesnelerinde eşzamansız erişim risklidir.
• Yan Etki (Side Effect): Metod, dış dünyada gözle görünür bir değişiklik
yaratıyorsa (örneğin dosya yazma, veri tabanı güncelleme) thread-safe
değildir veya uygun kilitleme gerektirir.
• Dokümantasyon: Android, iOS ve React Native API’ları çoğu zaman
açıkça thread-safe veya not thread-safe olarak işaretlenmiştir.
Thread-safe olmayan işlemler uygun kilitleme (mutex), immutable veri yapıları
veya tek thread kullanımıyla güvenli hale getirilebilir.
Animasyonların Neden UI Thread Üzerinde C izildiği
Mobil işletim sistemlerinde animasyonlar kullanıcı arayüzü ile doğrudan ilişkili
olduğu için UI thread üzerinde yürütülür. Bunun nedenleri:
• C izim Boru Hattı (Rendering Pipeline): Görsel bileşenlerin ölçülmesi,
yerleştirilmesi ve ekrana çizilmesi UI thread üzerinde gerçekleşir. Ani-
masyon bu sürecin bir parçasıdır.
• 16.67 ms Kuralı: 60 FPS akıcılığı sağlamak için her kare 16.67 ms içinde
hesaplanmak zorundadır. UI thread bu süreyi kontrol eder.
• Tutarllık (Consistency): UI bileşenlerinin tüm görsel değişimleri tek
bir merkezi thread üzerinden işlenmezse tutarsız ve yırtılma (tearing) adı
verilen hatalar ortaya çıkar.
• Platform Politikası: Android ve iOS, UI bileşenlerine arka plandan
erişimi yasaklar; çünkü thread yarışları görsel bozulmaya yol açabilir.
Bu nedenle animasyon hesaplamaları UI dışındaki thread’lerde yapılabilir
ancak çizim güncellemeleri mutlaka UI thread’de uygulanır.

Sandboxing ve Uygulama ˙Izolasyonu
Mobil işletim sistemleri (Android ve iOS), güvenlik, gizlilik ve sistem kararlılığını

• Zararlı Yazılım Koruması: Uygulamalar birbirlerine müdahale edemediği

Bellek Yönetimi: GC, ARC, Leak Örüntüleri ve
Memory Ownership
Mobil uygulamalarda bellek yönetimi, performans, kararlılık ve enerji tüketimi
açısından kritik öneme sahiptir. Android ve iOS bu konuda farklı yaklaşımlar
kullanır: Android, ART Garbage Collector ile otomatik bellek yönetimi sunarken;

tive ise her iki platformun üstünde çalışır ve ayrıca JavaScript çalışma zamanının

modeli ayrıntılı olarak ele alınmaktadır.
Android ART Garbage Collector
Android RunTime (ART), bellek yönetimi için generational (kuşaksal) yaklaşımlı
bir garbage collector (GC) kullanır. ART GC’nin temel özellikleri şunlardır:
• Generational Heap: Bellek young, old ve large object space bölgelerine
ayrılır.
• Copying GC: Young space üzerindeki objeler hızlıca kopyalanır; kısa
ömürlü nesneler burada toplanır.
• Mark-and-Sweep: Old space için işaretleme ve temizleme yaklaşımı kul-
lanılır.
• Concurrent Collector: GC çoğunlukla arka planda eşzamanlı çalışır.
GC genel olarak görünmezdir; ancak belirli durumlarda “stop-the-world”
(uygulamanın tamamen durması) fazları yaşanabilir.
Stop-the-World Fazları
GC her zaman tam olarak eşzamanlı çalışamaz. Bazı aşamalarda tüm iş parçacıkları
durdurulur. Bu fazlara stop-the-world denir.
• Root set taraması,
• Objelerin mark safhası,
• Finalize edilecek nesnelerin işaretlenmesi,
• Heap compaction (sıkıştırma).
Bu fazlar nanosaniye–milisaniye arası sürebilir; ancak büyük heaplerde, özellikle
düşük bellekli cihazlarda gözle görülür jank oluşabilir.

iOS Automatic Reference Counting (ARC)
iOS bellek yönetimini referans sayma (reference counting) yöntemiyle gerçekleştirir.
ARC, her nesneyi:
• oluşturulduğunda retain eder,
• kullanılmadığında release eder,
• referans sayısı sıfıra indiğinde bellekten siler.
Bu modelde GC yoktur; bellek tahliyesi deterministiktir. Ancak yanlış kul-
lanımda retain cycle oluşabilir.
iOS Retain Cycle Problemi
Retain cycle, iki nesnenin birbirine güçlü referans (strong reference) tutması
nedeniyle referans sayılarının asla sıfırlanmamasıdır.
Örnek tipik senaryo:
• Bir ViewController, bir closure’ı güçlü biçimde retain eder.
• Closure, ViewController’ı self ile retain eder.
Bu durumda GC olmadığı için memory leak kalıcı hale gelir.
Retain cycle çözümü:
• Weak veya unowned referans kullanımı,
• Objelerin yaşam döngüsünün manuel düzenlenmesi.

React Native ve JavaScript Heap Yönetimi
React Native çalıştığında JavaScript motoru (Hermes veya JSC) bağımsız bir
heap alanı yönetir. JS tarafında GC yapılır ancak bu GC:
• native heap’ten tamamen bağımsızdır,
• bridge üzerinden büyük veri geçişlerinde gecikmeye yol açabilir,
• senkron (stop-the-world) fazları daha belirgin olabilir.
RN’de tipik leak örüntüleri:
• Uzun ömürlü closure içinde büyük state saklamak,
• Event listener’ları temizlememek,
• Büyük JSON objelerini global scope’ta tutmak,
• Navigation parametrelerinde büyük nesne taşımak.
Bu leak’ler JS heap’i büyütür ve GC gecikmelerine neden olur.
Native Modüllerde Memory Ownership
React Native’de native modüller aracılığıyla Java veya Objective-C / Swift
tarafında bellek yönetimi yapılır. Burada sorumluluk tamamen geliştiricidedir.
• Java tarafı (Android): ART GC yönetir ancak JNI ile yapılan native
kod çağrılarında bellek elle yönetilmelidir.
• iOS tarafı (Objective-C / Swift): ARC çoğu durumu yönetir; an-
cak C/C++ objeleri ve Core Foundation türleri için manuel CFRelease()
gerekebilir.
• JSI Modülleri: C++ tarafında bellek yönetimi tamamen manuel veya
akıllı pointer (std::shared ptr) ile yapılmalıdır.
Yanlış ownership modeli:
• native → JS referansı bırakılmazsa,
• JS → native objesi tutulmaya devam ederse,
• circular reference oluşursa,
hem JS heap’te hem native heap’te leak oluşabilir.

˙Işletim Sistemi Zamanlayıcıları ve Güç Yönetimi
Mobil işletim sistemleri, enerji tasarrufu ve pil ömrünü korumak amacıyla arka
plan işlemlerini sıkı şekilde kontrol eder. CPU frekans ölçekleme, arka plan
kısıtlamaları, alarm zamanlaması ve foreground servis politikaları bu denetimin
temel bileşenleridir. Android ve iOS farklı mekanizmalara sahip olsa da ortak
hedef, gereksiz enerji tüketimini azaltmaktır.
CPU Frekans Yöneticisi (CPU Governor)
Mobil SoC’lerde CPU frekansı dinamik olarak ölçeklenir. Android Linux çekirdeği,
farklı governor stratejileri kullanır:
• ondemand: Yük arttığında frekansı hızla yükseltir; klasik yaklaşım.
• interactive: Daha agresif ve kullanıcı etkileşimine duyarlı; modern ciha-
zlarda yaygındır.
• schedutil: CPU scheduler ile entegre; frekans kararları işlem yoğunluğuna
göre yapılır.
• powersave / conservative: Enerji tasarrufu önceliklidir; düşük seviyeli
cihazlarda tercih edilir.
Bu mekanizma nedeniyle:
• yoğun hesaplama → CPU frekans artışı → pil tüketimi,
• uzun süreli arka plan iş → governor kısıtlaması,
• cihaz ekran kapalıyken → düşük frekans moduna geçiş
davranışları görülür.
Android Battery Optimization
Android 6.0 (API 23) ile güç yönetimi daha agresif hale gelmiştir. Başlıca iki
mekanizma:
Doze Mode
Cihaz uzun süre hareketsiz ve ekran kapalı kaldığında:
• ağ istekleri ertelenir,
• zamanlayıcılar toplu çalıştırılır (batching),
• wakelock alma kısıtlanır,
• arka plan servisleri durdurulur.
Doze, scheduled işlerin anında tetiklenmesini engeller. Bu nedenle Android,
alarm’ları ve zamanlanmış işleri toplu ve gecikmeli çalıştırabilir.

• büyük verilerin sıkça işlenmesi,
• sensörlerin sürekli aktif kalması.
iOS enerji sezgisi (heuristics) şu davranışları tetikler:
• arka plan uygulamalarını hızla suspended konumuna almak,
• background fetch pencerelerini nadir çalıştırmak,
• gereksiz CPU kullanımında uygulamayı throttle etmek,

Ağ ˙Iletişimi (Gelişmiş Konular): QUIC, Connec-
tion Pooling, Ağ Geçişi ve Captive Portal
Modern mobil uygulamalar yalnızca HTTP üzerinden veri alışverişi yapmaz;
iletişim performansı işletim sisteminin taşıma katmanı, bağlantı havuzu, ağ
geçiş davranışları ve ağ doğrulama mekanizmalarından doğrudan etkilenir. Bu
bölümde QUIC/HTTP/3 protokolü, OS seviyesinde bağlantı havuzu yönetimi,
Wi-Fi / hücresel ağ geçişi ve captive portal tespiti ayrıntılı olarak ele alınmaktadır.
QUIC ve HTTP/3
HTTP/3, Google tarafından geliştirilen ve daha sonra IETF tarafından stan-
dartlaştırılan QUIC protokolü üzerine kuruludur. QUIC, TCP yerine UDP
üzerinde çalışan modern bir taşıma katmanıdır.
QUIC’in Temel Özellikleri
• UDP tabanlıdır: Bağlantılar UDP üzerinden akış (stream) mantığıyla
yürütülür.
• 0-RTT bağlantı kurulumu: Tekrarlayan bağlantılar için sunucuya istek
hemen gönderilebilir.
• Head-of-Line Blocking yoktur: Tek bir kayıp paket tüm akışı durdur-
maz (TCP’deki en büyük darboğaz çözülmüştür).
• TLS 1.3 şifrelemeyi protokolün içine gömülüdür: TLS ve taşıma
katmanı birleşmiştir.
• Daha iyi çoklu akış desteği: Aynı bağlantı üzerinde paralel istekler
birbirini engellemez.
Mobil Uygulamalar Açısından Avantajları
• Düşük gecikme (RTT azaltımı),
• Paket kaybında hızlı toparlanma,
• Zayıf hücresel bağlantılarda daha stabil çalışma,
• TCP bağlantı kuyruğu etkilerinin azalması.
Android: Modern Chromium tabanlı Cronet kütüphanesi QUIC destekler.
iOS: NSURLSession HTTP/3 desteğini iOS 15+ sürümlerinde sunar.

Android RenderThread Mimarisi
Android 5.0 ile birlikte render pipeline üç ayrı thread’e bölünmüştür:
• UI Thread Layout, measure, input event’ler, animasyon hesaplamaları.
• RenderThread Draw komutlarını GPU’ya hazırlar; CPU tarafında çalışır.
• GPU Thread RenderThread’in hazırladığı komutları işler.
C alışma akışı:
1. UI thread → view hiyerarşisini hesaplar,
2. UI thread → RenderThread’e draw komutlarını gönderir,
3. RenderThread → GPU’ya komutları iletir,
4. GPU → çerçeveyi rasterize eder.
Bu yapı sayesinde UI thread meşgul olsa bile GPU işlemleri gecikmez.
Jank Nedenleri
• UI thread 16 ms içinde layout’u bitiremez → frame düşer,
• RenderThread aşırı draw komutu alır → GPU yetişemez,
• Overdraw → GPU bant genişliğini boğar,
• Büyük bitmap decode → UI thread bloklanır.

iOS’ta UI çizimi Core Animation tarafından yönetilir. Android’den farklı
olarak çoğu işlem GPU optimize edilmiş katmanlarda gerçekleşir.
Pipeline aşamaları:
1. Layout Pass AutoLayout constraint çözümleri, view ölçümleri.
2. Display Pass Her view kendi CALayer ’ına çizim yapar.
3. Compositing Pass GPU tüm layer’ları birleştirerek final görüntüyü üretir.
4. VSync Kare zamanlaması (60 Hz / 120 Hz).
Core Animation’ın avantajları:
• katman tabanlı animasyonlar CPU’yu yormaz,
• opacity, transform, shadow gibi özellikler GPU tarafında çalışır,
• layout ve paint aşaması yalnızca gerektiğinde yapılır.
Bu nedenle iOS animasyonları genellikle daha pürüzsüzdür.

Concurrency ve Parallelism: ˙Işletim Sistemi Temel
Kavramları
Mobil işletim sistemlerinde uygulama davranışını belirleyen en önemli mimaril-
erden biri eşzamanlılık (concurrency) ve paralellik (parallelism) yapılarıdır. Bu
kavramlar CPU çekirdekleri, iş parçacıkları (threads), işletim sistemi zaman-
layıcıları ve context switch gibi düşük seviyeli mekanizmalarla birlikte çalışır.
Bu bölümde OS tabanlı concurrency modeli detaylı şekilde açıklanmaktadır.
Concurrency (Eşzamanlılık)
Concurrency, birden fazla görevin aynı zaman aralığında ilerlemesi demektir.
Burada görevler gerçekten aynı anda çalışmak zorunda değildir; işletim sistemi
görevler arasında hızla geçiş yaparak eşzamanlılık hissi yaratır.
Temel Özellikler
• Tek çekirdekli CPU’da bile mümkündür,
• Görevler sırayla küçük zaman dilimlerinde çalıştırılır,
• Amaç birden fazla işin “ilerlemesini” sağlamaktır,
• Gerçek fiziksel paralellik gerektirmez.
Örnek:
• Kullanıcı kaydırma yaparken ağ isteği devam edebilir,

• C ok çekirdekli CPU gerektirir,
• Görevler aynı anda işlenir,
• Performans artışı sağlar,

Process (Süreç) ve Thread (˙Iş Parçacığı)

Bir uygulamanın yürütüldüğü, kendi adres alanına sahip bağımsız bir çalışma

Özellikleri:

• Context switching çok hızlıdır,
• Veri yarışlarına (race condition) açıktır,
• UI işlemleri genellikle tek bir UI thread’de yapılır.

Context Switch (Bağlam Değişimi)
Context switch, işletim sisteminin CPU’yu bir iş parçacığından (thread) diğerine
aktarmasıdır. Bu işlem, çok görevli (multitasking) işletim sistemlerinin temel
özelliğidir. Mobil cihazlarda (Android/iOS) yoğun context switch genellikle
performans ve enerji tüketimi üzerinde doğrudan etki yaratır.
Context Switch Sırasında Yaşanan Adımlar
Bir thread durdurulup diğerine geçilirken işletim sistemi aşağıdaki işlemleri
gerçekleştirir:
1. CPU register’larının kaydedilmesi Thread’in anlık durumu (program
counter, stack pointer, genel register’lar) kernel’de saklanır.
2. Stack durumunun korunması Mevcut thread’in stack’i kendi süreç
kontrol bloğunda (TCB) işaretlenir.
3. Yeni thread’in register’larının yüklenmesi Scheduler tarafından seçilen
thread’in kaydedilmiş register değerleri CPU’ya geri yüklenir.
4. CPU cache line yenilenmesi Yeni thread’in çalışacağı veri ve talimatlar
genellikle mevcut cache’de bulunmaz, bu nedenle cache kısmen temizlenir.
Cache miss, ek gecikmeye neden olur.
5. Memory barrier’ların uygulanması Paylaşılan hafıza yapılarında veri
tutarlılığı sağlanır.
6. Program counter’ın yeni thread’e yönlendirilmesi CPU artık yeni
thread’in kaldığı yerden devam eder.
Bu döngü milisaniyenin çok küçük bir bölümünde gerçekleşse de yoğun con-
text switch sistem genelini yavaşlatabilir.
Neden Context Switch Yapılır?
• OS scheduler yüksek öncelikli bir thread’i çalıştırmak ister,
• UI thread’e öncelik verilmesi gerekir,
• Bir thread uykuya geçer (I/O, network bekler),
• Bir thread zaman dilimini (time slice) tüketir,
• Bir interrupt (kesme) meydana gelir,
• Arka plan işler tetiklenir (GC, WorkManager, iOS background task).

Context Switch’in Maliyetleri
Context switch donanımsal ve yazılımsal maliyetler içerir:
• CPU maliyeti: Register ve stack durumunun sürekli kaydedilip yüklenmesi.
• Cache maliyeti: Cache pollution oluşur; yeni thread’in verileri cache
dışındadır, bu da CPU’nun RAM’den veri çekmesine neden olur.
• Enerji maliyeti: Mobil CPU’lar düşük güç moduna kolayca geçer; yoğun
thread değişimi CPU’yu sürekli uyanık tutar.
• Latency maliyeti: UI ile ilgili thread planlandığı anda çalışamazsa jank
oluşur.
Aşırı Context Switch’ın Sonuçları
Aşırı context switching şu problemlere yol açar:
• Jank (frame drop): UI thread zamanında çalıştırılamaz, çerçeve süresi
16ms eşiğini aşar.
• Pil tüketimi artışı: CPU derin uyku (deep sleep) moduna geçemez,
sürekli “wakeup” yaşanır.
• Scheduler yoğunluğu: OS zamanlayıcısı daha fazla iş yapmak zorunda
kalır.
• Thread starvation: Düşük öncelikli thread’ler hiç çalıştırılmayabilir.
• Lock-contention artışı: Paylaşılan yapılarda kilit bekleme süreleri uzar.
Android ve iOS Perspektifinde Context Switch
Android (Linux CFS Scheduler)
• C ok sayıda thread oluşturmak yaygındır, bu da context switch sayısını
artırır.
• ART runtime tarafından başlatılan GC thread’leri yoğun switching yarata-
bilir.
• RenderThread ve UI Thread arasında sürekli geçiş olur.
• WorkManager arka planda zamanlanmış işler ile düzenli wakelock tetikler.
iOS (GCD + Preemptive Scheduler)
• GCD (Grand Central Dispatch), gereksiz thread oluşturmayı azaltarak
context switch maliyetini düşürür.
• Dispatch queues sayesinde iş yükü daha kontrollü dağılır.
• Core Animation pipeline CPU ve GPU işlerini paralel optimize eder.

Multithreading ve Multitasking
Mobil işletim sistemleri hem çoklu iş parçacığı (multithreading) hem de çoklu
görev (multitasking) modellerini kullanarak aynı anda birden fazla işin iler-
lemesini sağlar. Bu iki kavram genellikle karıştırılır; ancak işleyişleri ve maliyet-
leri farklıdır.
Multithreading
Multithreading, bir aynı process içerisinde birden fazla iş parçacığının (thread)
paralel veya eşzamanlı çalışmasıdır. Tüm thread’ler aynı adres alanını ve aynı
dosya tanıtıcılarını paylaşır.
Temel Özellikler
• Tüm thread’ler aynı bellek alanını paylaşır,
• Thread oluşturma maliyeti düşüktür,
• Veri yarışları (race condition) ve kilitlenme (deadlock) riski yüksektir,
• ˙Işletim sistemi thread’ler arasında context switch yapar,
• UI thread genellikle diğerlerinden ayrı tutulur.
Mobil Uygulamalardaki Rolü
• UI thread (main thread): kullanıcı etkileşimi ve çizim,
• Background thread: ağ, disk, JSON parse, CPU yoğun işler,
• RenderThread (Android): GPU komutlarını işler,
• JS Thread (React Native): business logic ve state yönetimi,
• iOS GCD queue’ları: concurrency yönetimini otomatik optimize eder.
Multithreading’in Riskleri
• Paylaşılan objeler yanlışlıkla iki thread’de aynı anda güncellenebilir,
• Yanlış kilit tasarımı deadlock oluşturabilir,
• Fazla thread → fazla context switch → jank,
• React Native’de JS thread bloklanırsa UI tamamen donar.

Multitasking
Multitasking, işletim sisteminin birden fazla process’i aynı zaman aralığında
çalıştırmasıdır. Process’ler birbirinden tamamen izole edilmiştir (sandboxing).
Temel Özellikler
• Her process’in kendine ait bellek alanı vardır,
• Process’ler birbirinin RAM’ine erişemez,
• Her process’in kendi iş parçacıkları vardır,
• Process başlatma maliyeti thread’e göre çok daha yüksektir,
• IPC mekanizmaları gerekir (Binder, AIDL, App Groups, XPC).

Android

iOS
• Uygulamalar arka planda yalnızca sınırlı süre çalışır,
• Bellek uyarısı alındığında uygulama askıya alınabilir,
• Arka plan modları yalnızca belirli kategoriler için izinlidir (müzik, VoIP,
konum, sağlık sensörleri).

JavaScript Event Loop Mekanizması
Bu bölüm, JavaScript çalışma zamanında (özellikle tarayıcı ve Node.js ortam-
larında) asenkron yürütmeyi mümkün kılan Event Loop mekanizmasını resmi
ve teknik bir dille açıklamaktadır.
Genel Tanım
JavaScript, tek iş parçacıklı (single-threaded) bir yürütme modeline sahiptir.
Bu nedenle aynı anda yalnızca bir komut dizisi çalıştırabilir. Ancak, asenkron
işlemler, zamanlayıcılar ve giriş/çıkış (I/O) operasyonları tarayıcı veya çalışma
zamanı tarafından arka planda yürütülür. Event Loop, bu arka plan işlemlerinin
sonuçlarını uygun sırayla JavaScript yürütme ortamına geri dahil eden kontrol
mekanizmasıdır.
Temel Bileşenler
• Call Stack (C ağrı Yığını): JavaScript kodunun doğrudan yürütüldüğü
yapıdır. Tek iş parçacıklı olduğundan, aynı anda yalnızca bir çerçeve
işlenir.
• Web API’leri / Runtime API’leri: Zamanlayıcılar (setTimeout, setInterval),
ağ istekleri (fetch) ve diğer asenkron operasyonlar, JavaScript motoru
dışında, tarayıcı veya Node.js tarafından gerçekleştirilir.
• Task Queue (Macro Task Queue): Zamanlayıcı callback’leri, DOM
olayları ve diğer makro görevler, arka plandaki işlemler tamamlandığında
bu kuyruğa eklenir.
• Microtask Queue: Özellikle Promise çözümleme callback’leri (then/catch)
ve Mutation Observer görevleri burada tutulur. Event Loop her adımda
öncelikle bu kuyruğu boşaltır.
C alışma Prensibi
Event Loop, sürekli olarak aşağıdaki döngüyü yürütür:
1. Call Stack boşalana kadar mevcut senkron kod yürütülür.
2. Call Stack boşaldığında, Event Loop önce Microtask Queue’yu kontrol
eder. Eğer microtask varsa, tüm microtask’lar tamamlanana kadar Call
Stack’e aktarılır.
3. Microtask Queue tamamen boşaldıktan sonra Event Loop, Task Queue
içindeki ilk makro görevi Call Stack’e aktarır.
4. Bu döngü sonsuz şekilde devam eder.

Macro Task ve Micro Task Önceliği
Event Loop’un en önemli özelliği, microtask kuyruğunun her zaman makro
görevlerden önce işlenmesidir. Bu nedenle, Promise tabanlı callback’ler, aynı
döngü çevriminde setTimeout gibi zamanlayıcı görevlerinden önce yürütülür.
Örneğin:
console.log("A");
Promise.resolve().then(() => console.log("B"));
setTimeout(() => console.log("C"), 0);
console.log("D");
Yürütme sırası:
A → D → B → C
Özet
Event Loop, JavaScript’in tek iş parçacıklı yürütme modelini bloklamadan sürdürebilmesini
sağlayan temel mekanizmadır. Arka planda tamamlanan asenkron operasyon-
ların callback’leri, belirli kuyruklara eklenir ve Event Loop, Call Stack uygun
olduğunda bu callback’leri yürütme sürecine dahil eder. Özellikle microtask
kuyruğunun öncelikli olması, asenkron davranışın deterministik bir sıralamayla
gerçekleşmesini sağlar.

Node.js’in Asenkron Mimarisinde Arka Plan ˙Işlemlerinin
Yönetimi
Bu bölüm, Node.js’in tek iş parçacıklı (single-threaded) JavaScript yürütme
modeline rağmen asenkron görevleri nasıl arka planda gerçekleştirebildiğini resmi
ve teknik bir dille açıklamaktadır. Node.js’in mimarisi, JavaScript motorunun
tek iş parçacıklı yapısını korurken, I/O ve CPU ile ilgili birçok görevi çok iş
parçacıklı bir altyapı ile yönetir.
Tek ˙Iş Parçacıklı JavaScript Yürütmesi
Node.js, JavaScript kodunu Google V8 motoru üzerinde yürütür ve bu yürütme
tamamen tek iş parçacıklıdır. Yani:
• JavaScript kodu yalnızca bir call stack üzerinde çalışır.
• Her yürütme adımında yalnızca bir fonksiyon çerçevesi işlenebilir.
• Hesaplama yoğun görevler doğrudan bu iş parçacığını bloke edebilir.
Ancak bu sınırlama yalnızca JavaScript yürütme aşaması için geçerlidir.
Node.js çekirdeğinde asenkron işlemleri yöneten farklı katmanlar bulunmak-
tadır.
libuv ve C ok ˙Iş Parçacıklı Altyapı
Node.js, asenkron davranışını sağlamak için libuv adlı bir C kütüphanesini
kullanır. libuv, Node.js’e şu yetenekleri sağlar:
• C ok iş parçacıklı bir Thread Pool (varsayılan 4 iş parçacığı),
• Asenkron dosya işlemleri,
• DNS çözümleri,
• Zamanlayıcı yönetimi,
• Network polling mekanizmaları,
• OS tabanlı asenkron I/O arabirimleri.
Bu mimari sayesinde JavaScript kodu tek iş parçacıklı kalırken, I/O ve bazı
CPU-bound işlemler farklı yürütme kaynaklarına aktarılır.

Node.js’te Uzun Süreli Bir ˙Işlemin Worker Thread
ile Arka Plana Atılması
Aşağıda, CPU yoğun bir işlemin (örneğin büyük bir döngü) ana iş parçacığını
bloklamadan başka bir iş parçacığına (Worker Thread) aktarılmasını gösteren
minimal ve temiz bir Node.js örneği sunulmuştur. Bu yapı, uzun süren hesapla-
maların event loop’u engellemeden yürütülmesini sağlar.
worker.js
// worker.js
const { parentPort } = require("worker_threads");
// CPU-yoğun örnek işlem
function heavyJob() {
let total = 0;
for (let i = 0; i < 300_000_000; i++) {
total += i;
}
return total;
}
// Ana iş parçacığından mesaj alındığında işlemi başlat
parentPort.on("message", () => {
const result = heavyJob();
parentPort.postMessage(result);
});
main.js
// main.js
const { Worker } = require("worker_threads");
console.log("Ana thread çalışıyor...");
// Worker oluştur
const worker = new Worker("./worker.js");
// Worker’a görev gönder
worker.postMessage("start");
// Worker işlemi tamamladığında sonucu döndürür
worker.on("message", (result) => {
console.log("Ağır işlem bitti. Sonuç:", result);
});

Açıklama
Bu örnekte:
• heavyJob() fonksiyonu CPU yoğun bir döngü içermektedir.
• Bu işlem doğrudan ana iş parçacığında çalıştırılsaydı event loop tamamen
bloklanır ve diğer işlemler gecikirdi.
• Worker nesnesi aracılığıyla işlem ayrı bir iş parçacığında gerçekleştirilmekte
ve sonuç postMessage ile geri bildirilmektedir.
Bu yaklaşım, Node.js uygulamalarında uzun hesaplamaların performans kaybına
neden olmadan yürütülmesini mümkün kılar.

Bu bölüm, Worker Threads özelliğinin Node.js çekirdeğinde gerçekten bulunan
bir modül olup olmadığını teknik ve resmi bir dille açıklamaktadır.
Resmı Durum

• Node.js 10.5 sürümü ile tanıtılmıştır.
• Node.js 12 ve üzeri sürümlerde varsayılan olarak etkindir.

• SharedArrayBuffer ve Atomics ile paylaşılan bellek üzerinden paralel
işlemeyi mümkün kılmak,
• Büyük döngüler, kriptografik hesaplamalar, sıkıştırma, makine öğrenmesi
işlemleri gibi CPU-bound görevleri ayrı iş parçacıklarında çalıştırmaktır.
